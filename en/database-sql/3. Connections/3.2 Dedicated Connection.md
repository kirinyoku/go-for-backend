# Dedicated Connection

[`*sql.Conn`](https://pkg.go.dev/database/sql#Conn) represents a single database connection temporarily checked out from the `*sql.DB` pool. When an application connects directly to the database or uses a proxy operating in *session pooling* mode, every query issued through this object reaches the same server session until the application calls `Conn.Close`.

Most data access operations do not require a `*sql.Conn`: `*sql.DB` selects an available connection for each operation. A dedicated connection should be used only when several operations must run through the same underlying driver connection.

A database session holds state associated with a specific connection, including settings, temporary objects and certain kinds of locks. Regular operations through the pool are not guaranteed to reach the same session.

## Acquiring a Dedicated Connection (`db.Conn`)

The [`db.Conn`](https://pkg.go.dev/database/sql#DB.Conn) method returns an available connection from the pool or opens a new one:

```go
conn, err := db.Conn(ctx)
if err != nil {
    return fmt.Errorf("get dedicated connection: %w", err)
}
defer conn.Close()
```

If the pool has reached `MaxOpenConns` and every connection is in use, `db.Conn(ctx)` waits. It stops waiting when a connection becomes available or the supplied context is canceled.

The context passed to `db.Conn(ctx)` governs connection acquisition. Once the call succeeds, canceling that context does not automatically close the `Conn`. Each subsequent operation takes its own context:

```go
if err := conn.PingContext(queryCtx); err != nil {
    return fmt.Errorf("ping dedicated connection: %w", err)
}
```

::: warning
Once `db.Conn` succeeds, the application must call `Conn.Close`, even if a subsequent operation fails. Set up `defer conn.Close()` immediately after checking the error.
:::

## Binding Operations to a Single Session

Session state includes settings and objects that the database associates with the current connection. Examples include the selected temporary schema, time zone, user role or an application name used for diagnostics.

Consecutive queries through `*sql.DB` are not guaranteed to use the same session. You therefore cannot apply a connection-level setting with `db.ExecContext` and assume that the next call through `db` will observe it.

Session affinity can be demonstrated safely with this read-only example using a direct PostgreSQL connection:

```go
var firstBackendPID int
if err := conn.QueryRowContext(ctx,
    "SELECT pg_backend_pid()",
).Scan(&firstBackendPID); err != nil {
    return fmt.Errorf("read first backend pid: %w", err)
}

var secondBackendPID int
if err := conn.QueryRowContext(ctx,
    "SELECT pg_backend_pid()",
).Scan(&secondBackendPID); err != nil {
    return fmt.Errorf("read second backend pid: %w", err)
}
```

With a direct connection, both values identify the same PostgreSQL backend process because both calls run through one `*sql.Conn`. The same calls through `*sql.DB` are not guaranteed to return equal values. A proxy operating in *transaction pooling* mode may assign different backend processes even to a single client connection.

Session affinity is required for temporary objects, session settings and certain kinds of locks. However, changing session state requires a separate cleanup strategy tailored to the driver and database system.

If a session-bound group of operations must also be atomic, start a transaction with [`conn.BeginTx`](https://pkg.go.dev/database/sql#Conn.BeginTx):

```go
tx, err := conn.BeginTx(ctx, nil)
if err != nil {
    return fmt.Errorf("begin transaction on connection: %w", err)
}
defer tx.Rollback()

// All operations through tx run on the connection held by conn.

if err := tx.Commit(); err != nil {
    return fmt.Errorf("commit transaction: %w", err)
}
```

Finish the `tx` before closing the `conn`. If session state is not required, a regular `db.BeginTx` is sufficient because a transaction is already bound to a single connection. The complete transaction pattern is covered in [Managing Transactions](/en/database-sql/queries/transactions).

## Returning the Connection to the Pool (`Conn.Close`)

The [`conn.Close`](https://pkg.go.dev/database/sql#Conn.Close) method returns the connection to the pool. This usually does not close the underlying network connection immediately.

`Close` waits for operations already running on the connection to finish. To stop an in-flight query, cancel its context before closing the `Conn`.

After `Close`, any further operation through this object fails with [`sql.ErrConnDone`](https://pkg.go.dev/database/sql#ErrConnDone). The `*sql.Conn` value cannot be reused. Call `db.Conn` again when another dedicated connection is needed; the pool may return the same physical connection through a new object.
