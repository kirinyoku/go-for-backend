# Parameterizing SQL Queries

The [`ExecContext`](https://pkg.go.dev/database/sql#DB.ExecContext), [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext) and [`QueryRowContext`](https://pkg.go.dev/database/sql#DB.QueryRowContext) methods accept a SQL string followed by a separate list of arguments. This keeps user-supplied values out of the query text itself.

Separating SQL from data prevents supplied values from becoming part of the SQL syntax and removes the need to escape strings manually. However, `database/sql` does not define the placeholder syntax: the selected driver determines whether queries use `?`, `$1`, `@p1` or another form.

## The Risk of Interpolating Values into SQL

An unsafe way to construct a query looks like this:

```go
query := fmt.Sprintf(`
    SELECT id, email
    FROM users
    WHERE email = '%s'
`, email)

rows, err := db.QueryContext(ctx, query)
if err != nil {
    return nil, err
}
defer rows.Close()
```

If `email` comes from a user, it may contain not only an address but also part of the SQL syntax. For example, this value:

```text
x' OR 1=1 -- comment
```

would turn the condition into a query similar to this:

```sql
SELECT id, email
FROM users
WHERE email = 'x' OR 1=1 -- comment'
```

The `OR 1=1` condition is true for every row, while `-- ` with a trailing space starts a single-line comment, including in MySQL. As a result, the query may return data that the user should not be able to access. The exact comment syntax and other details of an attack payload still depend on the SQL dialect; protection must rely on parameters rather than attempts to recognize particular malicious strings.

The same problem occurs with string concatenation:

```go
query := "SELECT id, email FROM users WHERE email = '" + email + "'"
```

Adding quotes manually or replacing individual characters is not a reliable defense. Escaping rules differ between databases, data types and driver modes.

::: info
The problem is not the use of `fmt.Sprintf` itself, but the interpolation of untrusted data into SQL. Later in this article, it is used safely to generate numbered placeholders from a loop index controlled by the application.
:::

## Placeholders and Arguments

A placeholder marks the position where the driver should supply a separate value. The value itself is passed after the SQL string:

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE email = $1",
    email,
)
if err != nil {
    return nil, fmt.Errorf("query user by email: %w", err)
}
defer rows.Close()
```

This call has two independent parts:

- `"SELECT ... WHERE email = $1"` defines the structure of the SQL query;
- `email` is passed as the value for the `$1` placeholder.

`database/sql` passes the query and its arguments to the driver. The driver decides how to bind values to placeholders: through the database protocol, a prepared statement or another supported mechanism. The application does not need to quote or escape the string itself.

Do not place a placeholder inside SQL quotes:

```sql
-- Correct: the placeholder appears where a value is expected.
WHERE email = $1

-- Incorrect: $1 becomes part of the string literal.
WHERE email = '$1'
```

## Placeholder Syntax Across Drivers

The `database/sql` package does not translate one placeholder syntax into another. A query must use the form expected by the selected driver.

Common forms include:

| Syntax | Commonly Used By |
| :--- | :--- |
| `?` | MySQL and SQLite drivers |
| `$1` | PostgreSQL drivers |
| `@p1` | SQL Server drivers |
| `@name`, `:name` | Drivers with named parameter support |

These are common conventions, not universal rules for every database system. Always check the selected driver's documentation for the exact syntax.

Switching drivers may require changes to SQL queries even when the Go code continues to use the same `database/sql` methods. A common API does not automatically make SQL dialects or placeholder syntax portable.

## Argument Order and Types

Query execution methods receive their arguments after the SQL string as `args ...any`. How those arguments are matched depends on the placeholder form:

- each anonymous `?` usually receives the next argument in order;
- `$n` refers to argument number `n`, starting from one, so `$1` can be used more than once and placeholder numbers do not have to appear in order;
- a named placeholder is matched to the name supplied through `sql.Named`, provided the driver supports named parameters.

In the PostgreSQL example below, `$1` receives `email`, `$2` receives `active` and `$3` receives `userID`:

```go
_, err := db.ExecContext(ctx, `
    UPDATE users
    SET email = $1, active = $2
    WHERE id = $3
`, email, active, userID)
if err != nil {
    return fmt.Errorf("update user: %w", err)
}
```

The query must be supplied with arguments corresponding to the placeholders it uses. The selected driver defines the exact rules for reusing placeholders and skipping numbers. It also determines where those rules are validated.

Arguments may include strings, numbers, boolean values, `time.Time`, `nil` and other values supported by `database/sql` or the selected driver. Converting custom types through [`driver.Valuer`](https://pkg.go.dev/database/sql/driver#Valuer) is covered in [Custom Data Types](/en/database-sql/queries/custom-data-types).

::: info
Passing `nil` produces the SQL value `NULL`, but an ordinary `column = NULL` condition does not test whether a value is absent. SQL uses `IS NULL` for this check, and the structure of that condition must be included in the SQL query text.
:::

## Named Arguments

The package provides the [`sql.Named`](https://pkg.go.dev/database/sql#Named) function for creating named arguments:

```go
_, err := db.ExecContext(ctx, `
    UPDATE users
    SET email = @email
    WHERE id = @id
`,
    sql.Named("email", email),
    sql.Named("id", userID),
)
if err != nil {
    return fmt.Errorf("update user: %w", err)
}
```

The name passed to `sql.Named` does not include a prefix: use `"email"`, not `"@email"`. The symbol used in the SQL query depends on the driver's syntax.

Named arguments can make a long query easier to read and reduce the risk of mixing up values of the same type. However, the driver must support the corresponding form of named parameters.

## Building Dynamic Filters Safely

Sometimes the set of conditions depends on the input. For example, a query may filter users by email, active status or both. In this case, only the SQL structure is assembled dynamically, while the values continue to be passed as separate arguments.

```go
type UserFilter struct {
    Email  *string
    Active *bool
}

func buildUserQuery(filter UserFilter) (string, []any) {
    query := "SELECT id, email, active FROM users"
    conditions := make([]string, 0, 2)
    args := make([]any, 0, 2)

    if filter.Email != nil {
        args = append(args, *filter.Email)
        placeholder := fmt.Sprintf("$%d", len(args))
        conditions = append(conditions, "email = "+placeholder)
    }

    if filter.Active != nil {
        args = append(args, *filter.Active)
        placeholder := fmt.Sprintf("$%d", len(args))
        conditions = append(conditions, "active = "+placeholder)
    }

    if len(conditions) > 0 {
        query += " WHERE " + strings.Join(conditions, " AND ")
    }

    query += " ORDER BY id"

    return query, args
}
```

The resulting arguments are expanded with `...`:

```go
query, args := buildUserQuery(filter)

rows, err := db.QueryContext(ctx, query, args...)
if err != nil {
    return nil, fmt.Errorf("query users: %w", err)
}
defer rows.Close()
```

The application generates the `"email = $n"` and `"active = $n"` fragments from each argument's position. User-supplied values never become part of the SQL and are placed only in `args`.

The pointers in `UserFilter` distinguish between two states: the filter is absent or a specific value was provided. This is especially important for `Active`, because `false` is a valid filter value.

Generating a placeholder number from the length of `args` is safe because only an application-generated integer is inserted into the SQL, never user-provided text.

## Variable-Length `IN` Lists

A single placeholder normally represents one value. As a result, the following code usually does not expand a slice into multiple SQL arguments:

```go
ids := []int64{10, 20, 30}

rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE id IN ($1)",
    ids,
)
```

The driver receives one `[]int64` argument rather than three separate values. It may return an unsupported-type error or handle the value differently from what the code expects.

For PostgreSQL, the required number of numbered placeholders can be generated from the length of the slice:

```go
if len(ids) == 0 {
    return []User{}, nil
}

placeholders := make([]string, len(ids))
args := make([]any, len(ids))

for i, id := range ids {
    placeholders[i] = fmt.Sprintf("$%d", i+1)
    args[i] = id
}

query := `
    SELECT id, email
    FROM users
    WHERE id IN (` + strings.Join(placeholders, ", ") + `)
`

rows, err := db.QueryContext(ctx, query, args...)
if err != nil {
    return nil, fmt.Errorf("query users by ids: %w", err)
}
defer rows.Close()
```

An empty slice needs separate handling because `IN ()` is a syntax error in many SQL dialects. If an empty list of IDs should produce an empty result, the function can return it without querying the database.

Some databases and drivers support arrays or special constructs for passing a list as a single argument. This is a database-specific feature that must be checked separately.

## Dynamic Identifiers: Allowlisted Values Only

A placeholder represents a value, not SQL syntax. It cannot be used to substitute:

- a table name;
- a column name;
- `ASC` or `DESC`;
- a comparison operator;
- an arbitrary `WHERE` fragment.

For example, this query does not sort rows by the column whose name is stored in `sortColumn`:

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users ORDER BY $1",
    sortColumn,
)
```

The driver treats `sortColumn` as an ordinary value rather than part of the SQL syntax. The query may fail or effectively sort every row by the same constant.

Use an allowlist for dynamic identifiers: map an external value to a known, safe SQL fragment.

```go
allowedSortColumns := map[string]string{
    "email":      "email",
    "created_at": "created_at",
}

sortColumn, ok := allowedSortColumns[sortKey]
if !ok {
    return nil, ErrInvalidSort
}

direction := "ASC"
if descending {
    direction = "DESC"
}

query := `
    SELECT id, email
    FROM users
    ORDER BY ` + sortColumn + " " + direction

rows, err := db.QueryContext(ctx, query)
if err != nil {
    return nil, fmt.Errorf("query users: %w", err)
}
defer rows.Close()
```

Only strings from `allowedSortColumns` and the two application-defined values of `direction` are inserted into the SQL. `sortKey` is never added to the query directly.
