# Choosing a Query Execution Method

The [`database/sql`](https://pkg.go.dev/database/sql) package provides three primary methods for executing SQL: [`ExecContext`](https://pkg.go.dev/database/sql#DB.ExecContext), [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext) and [`QueryRowContext`](https://pkg.go.dev/database/sql#DB.QueryRowContext). The right choice depends not on the type of SQL statement, but on the shape of the expected result: no rows, multiple rows or at most one row.

The method you choose determines which value the call returns, where errors must be checked and which resources must be released. For example, a regular `INSERT` is usually executed with `ExecContext`, while `INSERT ... RETURNING id` produces a row and generally requires `QueryRowContext`.

::: tip
As a general rule, prefer methods with the `Context` suffix. They allow deadlines and cancellation to propagate to the driver. Whether a running query is interrupted promptly depends on support in the driver and database; without it, the call may not return until the query finishes.
:::

## Choosing a Method

Choose the method based on the result that the Go code needs:

| Expected Result | Method | Return Value |
| :--- | :--- | :--- |
| No result rows are needed | `ExecContext` | [`sql.Result`](https://pkg.go.dev/database/sql#Result) and `error` |
| The result may contain multiple rows | `QueryContext` | [`*sql.Rows`](https://pkg.go.dev/database/sql#Rows) and `error` |
| At most one row is expected | `QueryRowContext` | [`*sql.Row`](https://pkg.go.dev/database/sql#Row) |

The primary consideration is not the first keyword in the SQL statement, but whether the result contains rows and, if so, how many rows the caller expects.

::: info
The examples in this guide use PostgreSQL with [`pgx/v5/stdlib`](https://pkg.go.dev/github.com/jackc/pgx/v5@v5.10.0/stdlib), so the SQL below uses positional placeholders such as `$1`, `$2` and so on. Argument-to-placeholder mapping is covered in [Parameterizing SQL Queries](/en/database-sql/queries/query-parameters).
:::

## `ExecContext`

`ExecContext` executes a statement when no result rows are needed.

```go
_, err := db.ExecContext(ctx,
    "UPDATE users SET active = $1 WHERE id = $2",
    false,
    userID,
)
if err != nil {
    return fmt.Errorf("deactivate user: %w", err)
}
```

If the database cannot execute the statement, `ExecContext` returns an error. Possible causes include invalid SQL, a constraint violation, context cancellation or a connection problem.

::: warning
If the context passed to `ExecContext` reaches its deadline or is canceled, the returned error alone may not reveal whether the database applied the change. The statement could have completed on the server before the application received the result. Do not automatically retry `INSERT`, `UPDATE` or `DELETE` unless the operation was designed to be idempotent, meaning that repeating it produces no additional effect.
:::

On success, the method returns an `sql.Result`. It can be used to obtain the number of affected rows or a generated identifier, provided that the database and driver support the corresponding operation.

### `sql.Result`

| Method | Purpose |
| :--- | :--- |
| [`RowsAffected`](https://pkg.go.dev/database/sql#Result.RowsAffected) | Returns the number of rows affected by the statement |
| [`LastInsertId`](https://pkg.go.dev/database/sql#Result.LastInsertId) | Returns an integer identifier generated by the database |

Both methods return an `error` of their own because support depends on the database system and driver.

#### `RowsAffected`

The affected-row count can be used to determine whether the statement found the target record:

```go
result, err := db.ExecContext(ctx,
    "DELETE FROM users WHERE id = $1",
    userID,
)
if err != nil {
    return fmt.Errorf("delete user: %w", err)
}

affected, err := result.RowsAffected()
if err != nil {
    return fmt.Errorf("get affected rows: %w", err)
}
if affected == 0 {
    return ErrUserNotFound
}

return nil
```

The exact meaning of `RowsAffected` is determined by the database system and driver. For `UPDATE` in particular, database systems differ in whether they count rows that matched the condition but whose stored values did not change.

#### `LastInsertId` and `RETURNING`

Support for `LastInsertId` is likewise driver-specific. With PostgreSQL and pgx—the stack used in this guide—this method does not return the generated identifier. PostgreSQL uses `RETURNING` together with `QueryRowContext` instead:

```go
var id int64

err := db.QueryRowContext(ctx,
    "INSERT INTO users (email) VALUES ($1) RETURNING id",
    email,
).Scan(&id)
if err != nil {
    return 0, fmt.Errorf("insert user: %w", err)
}

return id, nil
```

Even when another driver documents support for `LastInsertId`, the error returned by the method itself must still be checked.

## `QueryRowContext`

Use `QueryRowContext` when a query should return at most one row: for example, a user selected by a unique ID, a record count obtained with `COUNT(*)` or an identifier returned by `INSERT ... RETURNING`.

```go
row := db.QueryRowContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
    userID,
)

var user User
if err := row.Scan(&user.ID, &user.Email); err != nil {
    return User{}, fmt.Errorf("scan user: %w", err)
}

return user, nil
```

This method has no separate `error` return value. It always returns a non-nil `*sql.Row`, while any error encountered during query execution is deferred until [`row.Scan`](https://pkg.go.dev/database/sql#Row.Scan) is called.

Calls are therefore often written in a more compact form:

```go
var count int

err := db.QueryRowContext(ctx,
    "SELECT COUNT(*) FROM users WHERE active = $1",
    true,
).Scan(&count)
if err != nil {
    return 0, fmt.Errorf("count active users: %w", err)
}
```

`QueryRowContext` does not verify that the query actually returned only one row. If the result contains multiple rows, `Scan` reads the first one and discards the rest. When selecting a single record, it is better to filter by a unique column or another condition that guarantees the expected result.

### `sql.ErrNoRows`

If `QueryRowContext` produces no row, `Scan` returns [`sql.ErrNoRows`](https://pkg.go.dev/database/sql#ErrNoRows). Because a missing record is often an expected outcome, this case must be distinguished from system errors involving the connection, query execution or reading the result.

```go
var user User

err := db.QueryRowContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
    userID,
).Scan(&user.ID, &user.Email)
if err != nil {
    if errors.Is(err, sql.ErrNoRows) {
        return User{}, ErrUserNotFound
    }

    return User{}, fmt.Errorf("find user: %w", err)
}

return user, nil
```

In this example, the data access code translates the technical `sql.ErrNoRows` error into the application-level `ErrUserNotFound` error. An HTTP handler higher in the call stack can map it to `404 Not Found`, while another caller can handle it according to its own rules.

Do not turn every `Scan` error into "record not found." The error may indicate context cancellation, a lost connection, incompatible types or a problem with the SQL statement. To identify a missing row, specifically check `errors.Is(err, sql.ErrNoRows)`.

## `QueryContext`

Use `QueryContext` when a query may return multiple rows. The method returns `*sql.Rows`, which must be read sequentially and closed after use.

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE active = $1 ORDER BY id",
    true,
)
if err != nil {
    return nil, fmt.Errorf("query active users: %w", err)
}
defer rows.Close()
```

After a successful call, the result is read with [`rows.Next`](https://pkg.go.dev/database/sql#Rows.Next), [`rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan) and a final check of [`rows.Err()`](https://pkg.go.dev/database/sql#Rows.Err). The complete safe pattern is covered in the next article, [Reading Query Results](/en/database-sql/queries/reading-rows).

Unlike `QueryRowContext`, an empty result from `QueryContext` does not produce [`sql.ErrNoRows`](https://pkg.go.dev/database/sql#ErrNoRows). Instead, the iteration simply yields no rows.
