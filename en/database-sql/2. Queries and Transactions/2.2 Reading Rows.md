# Reading Query Results

The [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext) method returns [`*sql.Rows`](https://pkg.go.dev/database/sql#Rows), a value used to read result rows sequentially. It does not turn the query result into a ready-made slice of Go structs. The application advances from row to row, scans the columns and collects the values it needs.

Working with `*sql.Rows` requires several steps. First, check the error returned by `QueryContext`. Then arrange for the result to be closed, read the rows in a loop and check for an iteration error after the loop. Skipping any of these steps can cause an error to be lost or a connection to remain occupied in the pool.

## Complete Read Cycle

A typical function that returns a list of users looks like this:

```go
func listActiveUsers(ctx context.Context, db *sql.DB) ([]User, error) {
    rows, err := db.QueryContext(ctx, `
        SELECT id, email, active
        FROM users
        WHERE active = $1
        ORDER BY id
    `, true)
    if err != nil {
        return nil, fmt.Errorf("query active users: %w", err)
    }
    defer rows.Close()

    var users []User

    for rows.Next() {
        var user User

        if err := rows.Scan(&user.ID, &user.Email, &user.Active); err != nil {
            return nil, fmt.Errorf("scan user: %w", err)
        }

        users = append(users, user)
    }

    if err := rows.Err(); err != nil {
        return nil, fmt.Errorf("iterate users: %w", err)
    }

    return users, nil
}
```

The order of these steps matters:

1. `QueryContext` executes the query and returns a `*sql.Rows`.
2. `defer rows.Close()` ensures that the result is closed whenever the function returns.
3. `rows.Next()` prepares the next row.
4. `rows.Scan(...)` copies the columns from the current row into the fields of a Go struct.
5. `rows.Err()` checks whether iteration ended because of an error.

## Obtaining a Result Set (`*sql.Rows`)

Before reading any rows, check the error returned by `QueryContext`:

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE active = $1",
    true,
)
if err != nil {
    return nil, fmt.Errorf("query users: %w", err)
}
defer rows.Close()
```

An error at this stage may mean that the package could not acquire a connection, the context was canceled, the driver rejected the query or the database reported a problem while executing it.

A successful `QueryContext` call means that the application has received a value from which it can read the result. It does not guarantee that every row will be read successfully: a problem may occur later during iteration.

## Closing the Result (`rows.Close`)

The [`rows.Close`](https://pkg.go.dev/database/sql#Rows.Close) method stops reading the result and releases its associated resources. It is usually scheduled with `defer` immediately after a successful `QueryContext` call:

```go
rows, err := db.QueryContext(ctx, query, args...)
if err != nil {
    return err
}
defer rows.Close()
```

This order ensures that `Rows` is closed if the function returns early, for example because of a `Scan` error, a business logic error or an explicit `return` inside the loop.

::: warning
Only schedule the call to `Close` after checking `err`. If `QueryContext` returns an error, `rows` is `nil`, and attempting to call `Close` on it will cause a `panic`.
:::

When `rows.Next()` reaches the end of the final result set, `database/sql` closes `Rows` automatically. Even so, `defer rows.Close()` remains a useful safeguard when the code reads only part of the result or returns early.

Calling `Close` more than once is safe because the method is idempotent. The automatic close after the final `Next` therefore does not conflict with the deferred `rows.Close()` call.

## Advancing Through Rows (`rows.Next`)

Immediately after `QueryContext`, the cursor is positioned before the first result row. The [`rows.Next`](https://pkg.go.dev/database/sql#Rows.Next) method prepares the next row and returns `true` when it can be read with `Scan`.

```go
for rows.Next() {
    // The current row is available here.
}
```

Every call to `Scan`, including the first, must be made only after `Next` has returned `true`. On the following iteration, `Next` advances the cursor to the next row.

From the application's perspective, the result is processed one row at a time rather than returned immediately as a `[]User`. The driver may fetch data from the database in batches; the exact buffering strategy depends on its implementation.

`Next` returns `false` in two cases:

- no rows remain;
- an error occurred while preparing the next row.

The return value alone does not distinguish between these cases. That is why [`rows.Err()`](https://pkg.go.dev/database/sql#Rows.Err) must be called after the loop.

If the query returns no rows, the first call to `Next` returns `false` and `rows.Err()` returns `nil`. This is a normal successful result from `QueryContext`, not `sql.ErrNoRows`. If the application contract needs to distinguish an empty result set from a non-empty one, check the length of the resulting slice or set a flag inside the loop.

## Reading Values (`rows.Scan`)

The [`rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan) method copies the columns from the current row into the supplied Go variables:

```go
for rows.Next() {
    var user User

    if err := rows.Scan(&user.ID, &user.Email, &user.Active); err != nil {
        return nil, fmt.Errorf("scan user: %w", err)
    }

    users = append(users, user)
}
```

The arguments passed to `Scan` must match the columns in the `SELECT` list in both number and order:

```sql
SELECT id, email, active FROM users
```

```go
rows.Scan(&user.ID, &user.Email, &user.Active)
```

Struct field names are not matched to column names automatically. Only the order of the destination arguments passed to `Scan` matters.

`Scan` receives pointers because it needs to modify the variables. The following call is incorrect:

```go
// Values are passed instead of variable addresses.
err := rows.Scan(user.ID, user.Email, user.Active)
```

If the number of arguments does not match the number of columns, or a value cannot be converted to the selected Go type, `Scan` returns an error. Handle these errors directly inside the loop; `rows.Err()` does not report them.

::: tip
List the required columns explicitly instead of using `SELECT *`. Changes to the table's column order or the addition of new columns will then not break the correspondence between the query result and the arguments passed to `Scan`.
:::

Working with `NULL`, `time.Time`, numeric types, `[]byte` and the `sql.Null*` types is covered in [Scanning SQL Values into Go Types](/en/database-sql/queries/scanning-types). For now, it is enough to remember that the Go type must be compatible with the value in the selected column.

## Checking Iteration Errors (`rows.Err`)

When `rows.Next()` returns `false`, the loop ends. To determine whether reading completed normally or stopped because of an error, call [`rows.Err`](https://pkg.go.dev/database/sql#Rows.Err):

```go
for rows.Next() {
    var user User

    if err := rows.Scan(&user.ID, &user.Email); err != nil {
        return nil, fmt.Errorf("scan user: %w", err)
    }

    users = append(users, user)
}

if err := rows.Err(); err != nil {
    return nil, fmt.Errorf("iterate users: %w", err)
}
```

An iteration error may occur after several rows have already been read successfully. For example, the context may be canceled, the connection may be lost or the driver may fail to fetch the next part of the result.

Without the `rows.Err()` check, the function would return an incomplete list as though the query had completed successfully. This is especially dangerous because part of the data appears valid while the error remains unnoticed.

On normal completion or for an empty result, `rows.Err()` returns `nil`.

## Why the Result Holds a Connection

To execute `QueryContext`, the package acquires a connection from the pool. While `*sql.Rows` remains open, that connection is occupied with reading the result and usually cannot serve another query through `*sql.DB`.

If the application forgets to close `Rows`, the number of available connections decreases. Once the pool is exhausted, new queries wait for a connection to be released and may eventually time out.

::: warning
Avoid running additional queries through `db` while iterating over an open `Rows` value unless necessary. With a small pool, the new query may wait for a connection held by the current result. It is often better to fetch the required data with a single SQL query, or to finish reading and close the result before starting another query.
:::

For the same reason, do not accumulate `defer rows.Close()` calls inside a long loop that executes many separate queries. All deferred calls run only when the surrounding function returns. Move one iteration into a separate function with its own `defer`, or close each result before continuing to the next query.
