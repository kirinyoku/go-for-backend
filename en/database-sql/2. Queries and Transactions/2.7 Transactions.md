# Managing Transactions

[`*sql.Tx`](https://pkg.go.dev/database/sql#Tx) represents an in-progress transaction. It allows multiple SQL operations to run on the same connection and complete as a unit: successful changes are committed with [`tx.Commit`](https://pkg.go.dev/database/sql#Tx.Commit), while failures are handled by discarding the changes with [`tx.Rollback`](https://pkg.go.dev/database/sql#Tx.Rollback).

A transaction is necessary when partially completing an operation would be unacceptable. For example, a funds transfer must not debit one account without crediting the other. Both statements must either succeed or leave the data unchanged.

## Transaction Lifecycle

The [`database/sql`](https://pkg.go.dev/database/sql) package manages the transaction lifecycle, while the driver and database system execute the transaction itself. The application starts it with [`db.BeginTx`](https://pkg.go.dev/database/sql#DB.BeginTx), receives a `*sql.Tx` and uses that object until the operation is complete.

| Stage | Action |
| :--- | :--- |
| Begin | `db.BeginTx` acquires a connection from the pool and creates a `*sql.Tx`. |
| Execution | `tx.ExecContext`, `tx.QueryContext` and other methods run inside the transaction. |
| Success | `tx.Commit` commits the changes. |
| Failure or cancellation | `tx.Rollback` discards the changes. |

An open transaction holds one connection from the pool. After `Commit` or `Rollback`, the connection is returned to the pool, and any further operation through `tx` fails with [`sql.ErrTxDone`](https://pkg.go.dev/database/sql#ErrTxDone).

::: warning
Every successfully started transaction must end with either `Commit` or `Rollback`. Otherwise, it continues to hold a connection and may prevent other queries from acquiring one from the pool.
:::

## Starting a Transaction (`BeginTx`)

The [`db.BeginTx`](https://pkg.go.dev/database/sql#DB.BeginTx) method accepts a context and optional transaction settings:

```go
tx, err := db.BeginTx(ctx, nil)
if err != nil {
    return fmt.Errorf("begin transaction: %w", err)
}
defer tx.Rollback()
```

Passing `nil` uses the default transaction settings of the driver or database system. An error can occur at this stage—for example, if a connection cannot be acquired, the context has already been canceled or the driver does not support the requested settings.

## Transaction Options (`sql.TxOptions`)

The [`sql.TxOptions`](https://pkg.go.dev/database/sql#TxOptions) struct lets the application request an isolation level and read-only mode:

| Field | Purpose |
| :--- | :--- |
| `Isolation` | An isolation level of type [`sql.IsolationLevel`](https://pkg.go.dev/database/sql#IsolationLevel). The zero value uses the default level of the driver or database system. |
| `ReadOnly` | Requests a read-only transaction. |

For example, a transaction for a consistent read can be started like this:

```go
opts := &sql.TxOptions{
    Isolation: sql.LevelSerializable,
    ReadOnly:  true,
}

tx, err := db.BeginTx(ctx, opts)
if err != nil {
    return fmt.Errorf("begin read transaction: %w", err)
}
defer tx.Rollback()
```

`database/sql` defines common isolation-level constants, but their support and exact behavior depend on the driver and database system. If the requested level is unsupported, `BeginTx` returns an error. `ReadOnly` support is also driver-dependent and does not replace the permissions of the database user.

::: tip
If an operation does not require a specific isolation level or read-only mode, pass `nil`. Default settings are usually clearer than an option selected without considering the behavior of the particular database system.
:::

## Executing Queries

A `*sql.Tx` provides the familiar query execution methods:

| Operation | Transaction Method |
| :--- | :--- |
| Statement without result rows | [`tx.ExecContext`](https://pkg.go.dev/database/sql#Tx.ExecContext) |
| Multiple rows | [`tx.QueryContext`](https://pkg.go.dev/database/sql#Tx.QueryContext) |
| One row | [`tx.QueryRowContext`](https://pkg.go.dev/database/sql#Tx.QueryRowContext) |
| Prepared statement | [`tx.PrepareContext`](https://pkg.go.dev/database/sql#Tx.PrepareContext) or [`tx.StmtContext`](https://pkg.go.dev/database/sql#Tx.StmtContext) |

Every SQL operation that belongs to the transaction must be performed through `tx`. Calling a method on the original `db` does not automatically make it part of the transaction and may use a different connection.

## Context and Transaction Lifetime

The context passed to `BeginTx` and the contexts passed to subsequent queries serve different purposes. `database/sql` does not automatically use the `BeginTx` context for `tx.ExecContext`, `tx.QueryContext` or other methods. Each method uses the context explicitly passed to that particular query.

The same context does not have to be used throughout. In practice, applications usually follow one of two approaches:

- pass the same `ctx` to `BeginTx` and all subsequent queries when the entire operation has a single deadline;
- pass a child context with a shorter timeout to a particular query when it needs a stricter limit than the transaction as a whole.

At the same time, the context passed to `BeginTx` remains in effect until `Commit` or `Rollback` and sets the upper bound for the lifetime of the entire transaction. If it is canceled earlier, `database/sql` rolls the transaction back, and a subsequent `Commit` returns an error.

## Deferred `Rollback`

Immediately after a successful `BeginTx`, it is common to call `defer tx.Rollback()`:

```go
tx, err := db.BeginTx(ctx, nil)
if err != nil {
    return fmt.Errorf("begin transfer: %w", err)
}
defer tx.Rollback()
```

The deferred call runs on every subsequent return from the function, so error paths do not need to repeat `Rollback` manually.

If `Commit` succeeds, the transaction is already complete, and the later deferred `Rollback` does not change anything.

## Committing a Transaction

A call to `tx.Commit` should be the final operation on the successful path. The transaction can be considered successfully complete only when `Commit` returns `nil`.

A `Commit` error must be returned to the caller. It may occur because the context was canceled, the connection was lost or the database rejected the commit. Results from queries within that transaction must not be treated as committed.

A constraint may be checked immediately, causing `ExecContext` or `Scan` to return an error, or the check may be deferred until `Commit`. Error mapping for a particular operation must therefore also be applied at the commit boundary. With the selected PostgreSQL and pgx stack, this can be done using the structured SQLSTATE and constraint name:

```go
var ErrEmailTaken = errors.New("email is already used")

func commitUserRegistration(tx *sql.Tx) error {
    err := tx.Commit()
    if err == nil {
        return nil
    }

    commitErr := fmt.Errorf("commit user registration: %w", err)

    var pgErr *pgconn.PgError
    if errors.As(err, &pgErr) &&
        pgErr.Code == "23505" &&
        pgErr.ConstraintName == "users_email_key" {
        return errors.Join(ErrEmailTaken, commitErr)
    }

    return commitErr
}
```

Here, [`pgconn`](https://pkg.go.dev/github.com/jackc/pgx/v5@v5.10.0/pgconn) is part of the selected driver stack, and `23505` represents a unique-constraint violation (`unique_violation`). A generic transaction helper cannot know which constraint corresponds to `ErrEmailTaken`; that mapping belongs to the specific operation.

::: warning
Do not automatically retry an entire operation after an unknown `Commit` error. If the network fails, the application may not know whether the database managed to commit the changes. For a retry to be safe, repeating the operation must produce no additional effect—a property known as idempotency. Another option is to check the current state of the data first.
:::

After calling `Commit`, regardless of its result, do not attempt to continue using the same `tx`. Start a new transaction for the next operation.

## Complete Success Path

The transaction owner, both completion paths and the `Commit` check should all be visible within one function. In this example, both the user update and the audit entry are performed through the same `tx`:

```go
func updateUserAndAudit(
    ctx context.Context,
    db *sql.DB,
    userID int64,
    displayName string,
) error {
    tx, err := db.BeginTx(ctx, nil)
    if err != nil {
        return fmt.Errorf("begin user update: %w", err)
    }
    defer tx.Rollback()

    result, err := tx.ExecContext(ctx, `
        UPDATE users
        SET display_name = $1
        WHERE id = $2
    `, displayName, userID)
    if err != nil {
        return fmt.Errorf("update user: %w", err)
    }

    affected, err := result.RowsAffected()
    if err != nil {
        return fmt.Errorf("get updated rows: %w", err)
    }
    if affected != 1 {
        return fmt.Errorf("update user: expected one row, got %d", affected)
    }

    if _, err := tx.ExecContext(ctx, `
        INSERT INTO audit_log (user_id, event)
        VALUES ($1, 'user_updated')
    `, userID); err != nil {
        return fmt.Errorf("write audit event: %w", err)
    }

    if err := tx.Commit(); err != nil {
        return fmt.Errorf("commit user update: %w", err)
    }

    return nil
}
```

Any early `return` triggers the deferred `Rollback`, while the successful path ends with a checked `Commit`. Calling `db.ExecContext` instead of `tx.ExecContext` would break this atomicity.

## Retrying Serialization Conflicts

PostgreSQL returns SQLSTATE `40001` for a serialization failure (`serialization_failure`). If the application decides to retry such an operation, the [PostgreSQL documentation](https://www.postgresql.org/docs/current/mvcc-serialization-failure-handling.html) requires retrying **the entire transaction with a new `BeginTx`**, including all reads and the logic that selects subsequent SQL statements. A deadlock error (`deadlock_detected`, code `40P01`) may also be a candidate for this kind of retry.

One possible coordinator with a limited number of attempts looks like this:

```go
func isRetryablePostgresTxError(err error) bool {
    var pgErr *pgconn.PgError
    if !errors.As(err, &pgErr) {
        return false
    }

    return pgErr.Code == "40001" || pgErr.Code == "40P01"
}

func runSerializableAttempt(
    ctx context.Context,
    db *sql.DB,
    fn func(context.Context, *sql.Tx) error,
) error {
    tx, err := db.BeginTx(ctx, &sql.TxOptions{
        Isolation: sql.LevelSerializable,
    })
    if err != nil {
        return fmt.Errorf("begin serializable transaction: %w", err)
    }
    defer tx.Rollback()

    if err := fn(ctx, tx); err != nil {
        return fmt.Errorf("run serializable transaction: %w", err)
    }
    if err := tx.Commit(); err != nil {
        return fmt.Errorf("commit serializable transaction: %w", err)
    }

    return nil
}

func runSerializable(
    ctx context.Context,
    db *sql.DB,
    fn func(context.Context, *sql.Tx) error,
) error {
    const maxAttempts = 3

    var lastErr error
    for attempt := 1; attempt <= maxAttempts; attempt++ {
        lastErr = runSerializableAttempt(ctx, db, fn)
        if lastErr == nil {
            return nil
        }
        if !isRetryablePostgresTxError(lastErr) ||
            attempt == maxAttempts {
            return lastErr
        }

        backoff := 20*time.Millisecond << (attempt - 1)
        jitter := time.Duration(rand.IntN(20)) * time.Millisecond

        select {
        case <-ctx.Done():
            return fmt.Errorf("wait before transaction retry: %w", ctx.Err())
        case <-time.After(backoff + jitter):
        }
    }

    return lastErr
}
```

In this example, `rand` refers to [`math/rand/v2`](https://pkg.go.dev/math/rand/v2), which is available starting with Go 1.22. The attempt limit protects the database from an infinite retry loop, exponential backoff increases the delay between attempts and jitter reduces the likelihood of another simultaneous conflict. Waiting stops when the context is canceled.

::: warning
The `fn` callback may run more than once. It may perform operations whose effects are part of the same transaction, but it must not send HTTP requests, emails or messages to an external broker. The coordinator does not retry unknown network errors, timeouts or `Commit` errors without a structured `40001` or `40P01`.
:::

## Handling `Rollback` Errors

In the basic `defer tx.Rollback()` pattern, the returned error is intentionally ignored. After a successful `Commit`, the deferred `Rollback` normally returns `sql.ErrTxDone` because the transaction has already finished. This is expected behavior, not an error in the completed operation.

If a rollback error is important for diagnostics, it can be filtered and logged:

```go
defer func() {
    err := tx.Rollback()
    if err != nil && !errors.Is(err, sql.ErrTxDone) {
        slog.Error("rollback transaction", "error", err)
    }
}()
```

An error while rolling back an unfinished transaction may indicate a problem with the connection or driver. The `*sql.Tx` can no longer be used regardless. If the main query error occurred before the rollback, preserve it as the reason the operation failed and either log the rollback error or join it with the primary error where that additional detail is genuinely needed.
