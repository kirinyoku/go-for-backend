# Custom Data Types

The [`database/sql`](https://pkg.go.dev/database/sql) package can scan standard database values into `string`, `int64`, `time.Time`, `[]byte` and other basic Go types. Applications, however, often need more expressive types: an order status with a limited set of valid values, JSON metadata, a UUID or another application-specific representation.

A type that needs custom conversion in both directions implements two standard interfaces: [`sql.Scanner`](https://pkg.go.dev/database/sql#Scanner) for reading values from the database and [`driver.Valuer`](https://pkg.go.dev/database/sql/driver#Valuer) for passing values to the database.

## Why Use Custom Types

Without a custom type, the same conversion logic is often repeated in every data access method. For example, a textual status has to be scanned, validated and converted manually:

```go
var rawStatus string

if err := row.Scan(&order.ID, &rawStatus); err != nil {
    return OrderWithStatus{}, fmt.Errorf("scan order: %w", err)
}

status := OrderStatus(rawStatus)
if !status.Valid() {
    return OrderWithStatus{}, fmt.Errorf("invalid order status %q", rawStatus)
}

order.Status = status
```

When this code appears in many queries, it is easy to omit the validation or implement it inconsistently. A custom type centralizes conversion and validation:

```go
if err := row.Scan(&order.ID, &order.Status); err != nil {
    return OrderWithStatus{}, fmt.Errorf("scan order: %w", err)
}
```

The same type can be passed as a query argument:

```go
_, err := db.ExecContext(ctx,
    "UPDATE orders SET status = $1 WHERE id = $2",
    order.Status,
    order.ID,
)
if err != nil {
    return fmt.Errorf("update order status: %w", err)
}
```

`database/sql` then calls the appropriate interface method automatically in each direction.

## Reading from the Database: `sql.Scanner`

The [`sql.Scanner`](https://pkg.go.dev/database/sql#Scanner) interface contains a single method:

```go
type Scanner interface {
    Scan(src any) error
}
```

When a destination implementing this interface is passed to [`Row.Scan`](https://pkg.go.dev/database/sql#Row.Scan) or [`Rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan), the package calls its `Scan` method with the value supplied by the driver.

The `src` argument has one of the supported types:

| `src` Type | Typical Use |
| :--- | :--- |
| `int64` | Integer values |
| `float64` | Floating-point values |
| `bool` | Boolean values |
| `string` | Text |
| `[]byte` | Textual or binary data |
| `time.Time` | Dates and times, if the driver supports the conversion |
| `nil` | SQL `NULL` |

A `Scan` implementation should:

- account for relevant driver representationsâ€”for example, text may arrive as either `string` or `[]byte`â€”and handle them appropriately;
- explicitly define whether `NULL` is allowed;
- fully validate and convert the input before updating the destination;
- return an error if conversion is impossible or would lose information.

`Scan` is normally implemented with a pointer receiver because its purpose is to modify the object and store the value that was read.

```go
func (s *OrderStatus) Scan(src any) error {
    var raw string

    switch value := src.(type) {
    case nil:
        return fmt.Errorf("scan order status: NULL is not allowed")
    case string:
        raw = value
    case []byte:
        raw = string(value)
    default:
        return fmt.Errorf(
            "scan order status: unsupported source type %T",
            src,
        )
    }

    status := OrderStatus(raw)
    if !status.Valid() {
        return fmt.Errorf("scan order status: invalid value %q", raw)
    }

    *s = status
    return nil
}
```

::: warning
A `[]byte` slice passed by the driver to `Scan` may reference memory that remains valid only until the next call to `Scan`. If the custom type needs to retain the original bytes, copy them inside the `case []byte:` branch with `bytes.Clone(value)` or `append([]byte(nil), value...)`.
:::

## Writing to the Database: `driver.Valuer`

The [`driver.Valuer`](https://pkg.go.dev/database/sql/driver#Valuer) interface handles conversion in the other direction:

```go
type Valuer interface {
    Value() (driver.Value, error)
}
```

When a custom type is passed to `ExecContext`, `QueryContext` or `QueryRowContext` as an argument, `database/sql` calls its `Value` method and passes the result to the driver. The method must return a [`driver.Value`](https://pkg.go.dev/database/sql/driver#Value) and an `error`.

`Value` must not panic. If the current value cannot be written to the database, the method returns an error:

```go
func (s OrderStatus) Value() (driver.Value, error) {
    if !s.Valid() {
        return nil, fmt.Errorf("invalid order status %q", s)
    }

    return string(s), nil
}
```

In this example, `Value` has a value receiver because the method does not modify the object. This is a convenient choice, but not an interface requirement: a pointer receiver is also valid if the SQL call receives a pointer.

Errors returned by `Value` are wrapped by `database/sql` in a way that preserves them for `errors.Is`. If `Value` itself wraps a lower-level or sentinel error with `%w`, callers can still identify that error after `ExecContext` or another SQL operation.

## When the Interfaces Are Not Needed

Custom `Scanner` and `Valuer` implementations are useful when conversion is an inherent, stable property of the type and is repeated across different queries. Not every newly named type needs these interfaces, however.

For example, a simple ID can often be scanned without any additional code:

```go
type UserID int64

var id UserID

if err := row.Scan(&id); err != nil {
    return 0, fmt.Errorf("scan user id: %w", err)
}
```

`database/sql` can perform safe conversions into custom types with a compatible underlying type. A `Scanner` implementation is needed for `UserID` only when additional validation or a non-standard representation is required.

Good candidates for custom interface implementations include:

- values that require repeated validation;
- JSON and other serialized structures;
- types that the driver returns in a representation different from the domain model.

::: tip
The `Scan` and `Value` methods should remain small, predictable and independent of external state. Their job is to convert one value, not to hide database queries or a business workflow inside a type.
:::
