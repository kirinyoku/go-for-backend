# Prepared Statements

A prepared statement lets an application create a reusable object for a fixed SQL template and then execute it one or more times with different arguments.

In [`database/sql`](https://pkg.go.dev/database/sql), a prepared statement is represented by [`*sql.Stmt`](https://pkg.go.dev/database/sql#Stmt). It is created with [`PrepareContext`](https://pkg.go.dev/database/sql#DB.PrepareContext), executed with [`ExecContext`](https://pkg.go.dev/database/sql#Stmt.ExecContext), [`QueryContext`](https://pkg.go.dev/database/sql#Stmt.QueryContext) or [`QueryRowContext`](https://pkg.go.dev/database/sql#Stmt.QueryRowContext) and closed with [`Close`](https://pkg.go.dev/database/sql#Stmt.Close).

## How a Prepared Statement Works

A regular parameterized query passes the SQL and its arguments in a single call:

```go
err := db.QueryRowContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
    userID,
).Scan(&user.ID, &user.Email)
if err != nil {
    return fmt.Errorf("find user: %w", err)
}
```

With explicit preparation, the SQL is supplied in advance and each execution receives only the arguments:

```go
stmt, err := db.PrepareContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
)
if err != nil {
    return fmt.Errorf("prepare user query: %w", err)
}
defer stmt.Close()

err = stmt.QueryRowContext(ctx, userID).Scan(&user.ID, &user.Email)
if err != nil {
    return fmt.Errorf("find user: %w", err)
}
```

At the `database/sql` level, calling `PrepareContext` guarantees the creation of a `*sql.Stmt` associated with specific SQL text. The statement can be executed repeatedly with different arguments as long as it remains open and the object that owns it is still usable.

However, `database/sql` does not guarantee that the SQL will be sent to the server exactly once, that the database will retain a single execution plan or that every direct `db.QueryContext` call performs preparation from scratch. The implementation depends on the driver, protocol and database system:

- the driver may automatically prepare and cache regular queries without an explicit `PrepareContext`;
- a statement created through `*sql.DB` may be prepared separately on several physical connections in the pool;
- the database may choose between a generic execution plan and one tailored to specific arguments.

With the [`pgx`](https://pkg.go.dev/github.com/jackc/pgx/v5@v5.10.0) driver used in these examples, regular calls already use an automatic prepared-statement cache by default. An explicit `PrepareContext` should therefore be treated as an API for explicit ownership and reuse of a `*sql.Stmt`, not as a guaranteed performance optimization.

## Preparing with `PrepareContext`

The [`db.PrepareContext`](https://pkg.go.dev/database/sql#DB.PrepareContext) method creates a `*sql.Stmt` for later queries or executions:

```go
stmt, err := db.PrepareContext(ctx, `
    SELECT id, email, active
    FROM users
    WHERE id = $1
`)
if err != nil {
    return nil, fmt.Errorf("prepare find user: %w", err)
}
defer stmt.Close()
```

The SQL still contains placeholders, but their values are not supplied yet. They are passed when the statement is executed.

The context passed to `PrepareContext` applies only to the preparation stage. It is not stored in the `*sql.Stmt` and does not control later executions. Each statement execution should therefore receive the context of the current operation, such as [`r.Context()`](https://pkg.go.dev/net/http#Request.Context) for an HTTP request or a separate context with a timeout.

## Executing a `*sql.Stmt`

The methods on `*sql.Stmt` correspond to the main methods on `*sql.DB`, but they do not receive the SQL string again.

### Single Row: `stmt.QueryRowContext`

For a query expected to return at most one row, pass the arguments directly to `QueryRowContext` and check the error when calling `Scan`:

```go
findUserStmt, err := db.PrepareContext(ctx, `
    SELECT id, email
    FROM users
    WHERE id = $1
`)
if err != nil {
    return User{}, fmt.Errorf("prepare find user: %w", err)
}
defer findUserStmt.Close()

var user User

err = findUserStmt.QueryRowContext(ctx, userID).Scan(
    &user.ID,
    &user.Email,
)
if err != nil {
    if errors.Is(err, sql.ErrNoRows) {
        return User{}, ErrUserNotFound
    }

    return User{}, fmt.Errorf("find user: %w", err)
}

return user, nil
```

As with `db.QueryRowContext`, execution errors are deferred until `Scan` is called.

### `stmt.ExecContext`

Use `ExecContext` for a statement that does not return rows:

```go
updateUserStmt, err := db.PrepareContext(ctx, `
    UPDATE users
    SET active = $1
    WHERE id = $2
`)
if err != nil {
    return fmt.Errorf("prepare update user: %w", err)
}
defer updateUserStmt.Close()

_, err = updateUserStmt.ExecContext(ctx, active, userID)
if err != nil {
    return fmt.Errorf("update user: %w", err)
}
```

The method also returns an [`sql.Result`](https://pkg.go.dev/database/sql#Result), which behaves in the same way as the result of `db.ExecContext`. The [`RowsAffected`](https://pkg.go.dev/database/sql#Result.RowsAffected) and [`LastInsertId`](https://pkg.go.dev/database/sql#Result.LastInsertId) methods are covered in [Choosing a Query Execution Method](/en/database-sql/queries/exec-query-queryrow).

### `stmt.QueryContext`

If a query returns multiple rows, `*sql.Stmt` returns a regular `*sql.Rows`:

```go
listActiveStmt, err := db.PrepareContext(ctx, `
    SELECT id, email
    FROM users
    WHERE active = $1
    ORDER BY id
`)
if err != nil {
    return nil, fmt.Errorf("prepare active users: %w", err)
}
defer listActiveStmt.Close()

rows, err := listActiveStmt.QueryContext(ctx, true)
if err != nil {
    return nil, fmt.Errorf("query active users: %w", err)
}
defer rows.Close()
```

The usual rules for [`rows.Next`](https://pkg.go.dev/database/sql#Rows.Next), [`rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan) and [`rows.Err`](https://pkg.go.dev/database/sql#Rows.Err) still apply. The complete pattern is covered in [Reading Query Results](/en/database-sql/queries/reading-rows).

## Reuse and Concurrency

A `*sql.Stmt` is safe for concurrent use by multiple goroutines. The same prepared statement can be shared across multiple application operations as long as its lifetime matches that of its owner and it is not tied to a transaction that has ended or a connection that has closed.

Prepare the statement outside the loop:

```go
activateUserStmt, err := db.PrepareContext(ctx,
    "UPDATE users SET active = $1 WHERE id = $2",
)
if err != nil {
    return fmt.Errorf("prepare update user: %w", err)
}
defer activateUserStmt.Close()

for _, userID := range userIDs {
    if _, err := activateUserStmt.ExecContext(ctx, true, userID); err != nil {
        return fmt.Errorf("activate user %d: %w", userID, err)
    }
}
```

Calling `PrepareContext` during every iteration repeats the preparation step and allocates additional resources without reusing the prepared statement.

::: info
Separate `UPDATE` calls do not become a single atomic operation merely because they use the same prepared statement. If all changes must either be applied together or rolled back together, use a transaction. This is covered in the next article, [Managing Transactions](/en/database-sql/queries/transactions).
:::

## `*sql.Stmt` Lifetime and Binding

The lifetime and binding of a `*sql.Stmt` depend on the object through which it was prepared:

| Created With | Binding | Becomes Unusable |
| :--- | :--- | :--- |
| `db.PrepareContext` | Managed by the `*sql.DB` pool | After `stmt.Close()` or `db.Close()` |
| `conn.PrepareContext` | One dedicated connection | After `stmt.Close()` or `conn.Close()` |
| `tx.PrepareContext` | One connection for the lifetime of one transaction | After `stmt.Close()`, `tx.Commit()` or `tx.Rollback()` |

A statement prepared through `*sql.DB` does not reserve a single connection permanently. It remains usable for the lifetime of the `DB` unless closed sooner. When an execution uses a new connection from the pool, `database/sql` automatically prepares the SQL on that connection.

This leads to an important detail: one `*sql.Stmt` in Go may correspond to several prepared statements on different connections. The larger the pool and the more long-lived `*sql.Stmt` values the application maintains, the more resources the driver and database may need.

A statement prepared through `*sql.Conn` or `*sql.Tx`, by contrast, is permanently bound to a single physical connection. It cannot be used after the `Conn` is closed or the transaction ends.

## Closing a `*sql.Stmt`

A `*sql.Stmt` must be closed when it is no longer needed:

```go
stmt, err := db.PrepareContext(ctx, query)
if err != nil {
    return err
}
defer stmt.Close()
```

`Close` releases the resources associated with the `*sql.Stmt` and prevents new operations through it.

For a local prepared statement, call `defer stmt.Close()` immediately after successful preparation. A long-lived `*sql.Stmt` stored in an application struct should be closed by that struct's owner during shutdown, usually alongside the `*sql.DB`.

## When Explicit Preparation Is Useful

Explicit preparation is appropriate when the same SQL template is executed repeatedly and the statement has a clearly defined lifetime: for example, within a short loop, by a long-lived object across many application operations or several times within a single transaction.

For a short loop, prepare the statement before the loop and close it afterward. A long-lived `*sql.Stmt` on a `*sql.DB` should be closed by the component owner during shutdown. A statement used within a transaction should be created with `tx.PrepareContext` and must not be used after the transaction ends.

Explicit preparation does not guarantee better performance. The driver may already cache queries automatically, while a single `*sql.Stmt` on a `*sql.DB` can consume resources on several connections in the pool. If there is no deliberate reuse, a direct call to `db.ExecContext`, `db.QueryContext` or `db.QueryRowContext` is usually simpler. Any performance benefit should be confirmed by measuring the application under realistic load.
