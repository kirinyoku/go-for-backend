# Connecting to a Database

Once a driver has been selected, an application can create a shared [`*sql.DB`](https://pkg.go.dev/database/sql#DB) connection pool and verify database connectivity. The [`database/sql`](https://pkg.go.dev/database/sql) package manages the standard types and the pool itself, while the selected driver handles driver-specific connection configuration and establishes physical connections.

Basic initialization has three steps: register the driver, create a [`*sql.DB`](https://pkg.go.dev/database/sql#DB) and then verify database connectivity with [`db.PingContext`](https://pkg.go.dev/database/sql#DB.PingContext). The examples below first show the usual approach with [`sql.Open`](https://pkg.go.dev/database/sql#Open), followed by an optional alternative based on [`sql.OpenDB`](https://pkg.go.dev/database/sql#OpenDB) and a typed [`driver.Connector`](https://pkg.go.dev/database/sql/driver#Connector). Do not confuse creating a `*sql.DB` value with establishing a network connection: these happen at different stages.

## Registering the Driver

A driver is added to the program like any other Go package, but it is often imported only for its side effects, using a *blank import*:

```go
import (
    "database/sql"

    _ "path/to/driver" // replace with the selected driver's import path
)
```

The `_` identifier means that the package is imported only for its side effects. Application code does not refer directly to any of the driver's exported identifiers. Instead, the package registers itself with `database/sql` during initialization.

After registration, `database/sql` can look up the appropriate driver when the application calls `sql.Open("driver-name", dsn)`.

The exact import path and the driver name passed to `sql.Open` are documented by the selected driver. Neither is standardized by `database/sql`.

## Connection String (DSN)

Another key part of the connection setup is the DSN (*data source name*). It is a string containing database access parameters such as the address, database name, username, password, TLS mode, timeout and other settings.

The DSN format is driver-specific. `database/sql` passes the string to the driver without interpreting it as a universal format.

These are examples of formats used by different drivers:

```text
postgres://user:password@localhost:5432/app?sslmode=disable

postgres://user:password@db.example.com:5432/app?sslmode=verify-full&sslrootcert=/etc/app/ca.crt

user:password@tcp(localhost:3306)/app?parseTime=true

file:app.db?cache=shared
```

::: warning
`sslmode=disable` does not encrypt the connection or verify the server's identity. Use it only in a controlled local environment. For a remote PostgreSQL server, configure certificate and hostname verification as described in the driver documentation; in the DSN format shown above, this uses `verify-full`.
:::

In a real application, the DSN normally comes from configuration such as environment variables, a configuration file or a secrets manager. Do not hard-code the database password or address in the source code.

## Creating a Pool with `sql.Open`

The [`sql.Open`](https://pkg.go.dev/database/sql#Open) function creates a new [`*sql.DB`](https://pkg.go.dev/database/sql#DB):

```go
db, err := sql.Open("driver-name", dsn)
if err != nil {
    return nil, fmt.Errorf("open db: %w", err)
}
```

The first argument is the name of a registered driver. The second is a DSN in the format expected by that driver.

Despite its name, `sql.Open` does not necessarily establish a network connection immediately. It may only validate the driver name and perform basic validation of the arguments. A successful call therefore does not mean that the database is reachable, the credentials are valid, the network is working or the requested database exists.

Do not treat a successful `sql.Open` call as a connectivity check. If the database is unavailable, the error often surfaces only when the first query runs or when `PingContext` is called explicitly.

## Alternative: `sql.OpenDB`

For a basic connection, `sql.Open` is usually sufficient. Use `sql.OpenDB` when the application has a [`driver.Connector`](https://pkg.go.dev/database/sql/driver#Connector) built from typed configuration or needs additional settings that cannot be expressed through a DSN.

For example, the third-party [`pgx/v5`](https://pkg.go.dev/github.com/jackc/pgx/v5@v5.10.0) and [`pgx/v5/stdlib`](https://pkg.go.dev/github.com/jackc/pgx/v5@v5.10.0/stdlib) packages can parse a typed configuration, create a `driver.Connector` and pass it to the standard library:

```go
config, err := pgx.ParseConfig(os.Getenv("DATABASE_URL"))
if err != nil {
    return nil, fmt.Errorf("parse database config: %w", err)
}

connector := stdlib.GetConnector(*config)
db := sql.OpenDB(connector)
```

Like `sql.Open`, `sql.OpenDB` does not confirm that the server is reachable, so `PingContext` is still required after the pool has been configured. A driver may also provide a helper that returns a `*sql.DB` directly. Follow the approach documented by that driver and keep ownership of the pool explicit.

## When Connections Are Opened

A `*sql.DB` is not a single connection. It is a handle that manages a connection pool. Physical connections are opened as needed: during `PingContext`, `ExecContext`, `QueryContext`, `QueryRowContext`, when a transaction begins or when a dedicated connection is acquired.

This model allows the application to create a `*sql.DB` once at startup and pass it through the rest of the program as a dependency. The value itself is safe for concurrent use by multiple goroutines and manages both in-use and idle connections.

::: tip
Create one `*sql.DB` for each distinct connection configuration and reuse it. A separate pool makes sense when the DSN, credentials or workload isolation requirements differ; do not create a new pool for every handler or data access method.
:::

## Verifying Database Connectivity

[`db.PingContext`](https://pkg.go.dev/database/sql#DB.PingContext) checks that a database connection is still alive, establishing one if necessary.

It is usually called during application startup so that a problem is discovered immediately rather than during the first user request.

It is useful to include this check in the function that creates the pool. If `PingContext` fails, the function closes the `*sql.DB` before ownership has been handed to the caller:

```go
func openDB(
    ctx context.Context,
    driverName string,
    dsn string,
) (*sql.DB, error) {
    db, err := sql.Open(driverName, dsn)
    if err != nil {
        return nil, fmt.Errorf("open db: %w", err)
    }

    pingCtx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()

    if err := db.PingContext(pingCtx); err != nil {
        if closeErr := db.Close(); closeErr != nil {
            return nil, errors.Join(
                fmt.Errorf("ping db: %w", err),
                fmt.Errorf("close db after failed ping: %w", closeErr),
            )
        }

        return nil, fmt.Errorf("ping db: %w", err)
    }

    return db, nil
}
```

::: info
A successful `PingContext` confirms availability only at the time of the check. Errors must still be handled for every database operation.
:::

## Pool Lifetime and `db.Close`

A `*sql.DB` is designed to be long-lived. It is usually closed only once: when the application shuts down, a test finishes or a temporary environment is torn down.

The [`db.Close`](https://pkg.go.dev/database/sql#DB.Close) method prevents new queries from starting and waits for operations already in progress to finish. For this reason, a `*sql.DB` must not be closed inside a handler or after every data access method.

If a `*sql.DB` is closed too early, subsequent database operations will begin to fail. If it is not closed when a short-lived program or test finishes, resources may remain allocated longer than expected.
