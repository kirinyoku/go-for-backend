# Groups and Namespaces

A single log event may contain fields with similar meanings that belong to distinct domain entities. For example, an HTTP request and an authenticated user might both possess an `id` field:

```go
logger.Info("request authorized",
    "id", requestID,
    "id", userID,
)
```

This event contains duplicate `id` keys, leaving downstream consumers to guess which ID belongs to which entity. Renaming keys to `request_id` and `user_id` resolves simple collisions, but when entities carry multiple attributes, organizing related fields into named groups provides better clarity and structure.

## Inline Event Groups via `slog.Group`

The [`slog.Group`](https://pkg.go.dev/log/slog#Group) function organizes related attributes under a common parent key:

```go
logger.Info("request authorized",
    slog.Group("request",
        slog.String("id", requestID),
        slog.String("method", "GET"),
    ),
    slog.Group("user",
        slog.Int64("id", userID),
        slog.String("role", "admin"),
    ),
)
```

Within each group, concise key names like `id` can be used safely because their qualified names include the parent namespace (`request` or `user`). An inline group applies only to that specific log event and behaves as a single attribute.

`slog.Group` processes its variadic arguments using the same rules as logger methods: it accepts alternating `"key", value` pairs or explicit `slog.Attr` values. Using explicit attributes like `slog.String` helps visually separate the contents of nested groups.

## Text and JSON Output Representations

An attribute group defines the logical structure of an event, while its visual layout depends on the configured log handler. (Timestamps are omitted in the examples below for brevity.)

`TextHandler` formats group keys using dot-separated notation:

```text
level=INFO msg="request authorized" request.id=req-7 request.method=GET user.id=42 user.role=admin
```

`JSONHandler` renders groups as nested JSON objects:

```json
{"level":"INFO","msg":"request authorized","request":{"id":"req-7","method":"GET"},"user":{"id":42,"role":"admin"}}
```

In both representations, `request.id` and `user.id` remain distinct fields. Application code does not choose between dot-separated text strings and nested JSON objects—it defines the semantic group structure, while the handler controls the output representation.

Avoid grouping every single isolated field. For standard flat event schemas (`request_id`, `method`, `route`, `status`), unnecessary grouping complicates log querying. Groups are most valuable when bundling several related fields or resolving key collisions.

## Persistent Namespaces via `WithGroup`

When a component regularly logs attributes within a dedicated namespace, you can construct a derived logger with [`Logger.WithGroup`](https://pkg.go.dev/log/slog#Logger.WithGroup):

```go
upstreamLogger := logger.
    With(slog.String("component", "http-client")).
    WithGroup("http")

upstreamLogger.Info("upstream request completed",
    slog.String("method", "GET"),
    slog.String("route", "/profiles/{id}"),
    slog.Int("status", 200),
)
```

Because `component` was added before `WithGroup("http")`, it remains at the root level of the log record. All attributes appended after opening the group fall into the `http` namespace:

```text
level=INFO msg="upstream request completed" component=http-client http.method=GET http.route=/profiles/{id} http.status=200
```

```json
{"level":"INFO","msg":"upstream request completed","component":"http-client","http":{"method":"GET","route":"/profiles/{id}","status":200}}
```

Like `With`, `WithGroup` returns a new `*slog.Logger` without mutating the parent instance. All subsequent attributes added to this derived logger will be scoped inside the open group.

## Evaluation Order of `With` and `WithGroup`

The sequence of `With` and `WithGroup` calls determines where bound attributes are placed:

```go
logger.
    With(slog.String("service", "users-api")).
    WithGroup("http")
// "service" remains at the root level.

logger.
    WithGroup("http").
    With(slog.String("service", "profiles-api"))
// "service" is nested inside the "http" group.
```

`WithGroup` affects only attributes appended *after* its invocation. This applies both to subsequent `With` calls and to call-site event attributes. To prevent confusion in long method chains, construct derived loggers step by step with descriptive variable names.

Groups can also be nested:

```go
databaseLogger := logger.
    WithGroup("database").
    WithGroup("query")

databaseLogger.Info("query completed",
    slog.Duration("duration", duration),
)
```

`TextHandler` formats this as `database.query.duration`, while `JSONHandler` constructs a nested `database` object containing a `query` object. Deep nesting quickly complicates query schemas, so one or two namespace levels are usually optimal for application logs.

## Choosing Between `Group` and `WithGroup`

Both APIs construct equivalent logical event structures, but they differ in scope:

| API | Scope | Typical Use Case |
| :--- | :--- | :--- |
| `slog.Group` | Single event | Grouping a set of related fields for a specific operation |
| `Logger.WithGroup` | All subsequent events on derived logger | Establishing a persistent component namespace |

If a group is needed for only one event, `slog.Group` keeps the grouping localized to that call site. If every log entry emitted by an HTTP client component should feature `http.method`, `http.route`, and `http.status`, creating a derived logger with `WithGroup("http")` eliminates boilerplate duplication.

::: info Go 1.25+
The [`slog.GroupAttrs`](https://pkg.go.dev/log/slog#GroupAttrs) function, introduced in [Go 1.25](https://go.dev/doc/go1.25#log/slog), constructs a group specifically from a slice of pre-built `slog.Attr` values:

```go
attrs := []slog.Attr{
    slog.String("method", "GET"),
    slog.Int("status", 200),
}

logger.Info("request completed",
    slog.GroupAttrs("http", attrs...),
)
```

For inline key-value pairs passed directly at call sites, standard `slog.Group` (available since Go 1.21) remains the standard choice.
:::

## Maintaining a Consistent Group Schema

Group names become part of your event schema. Avoid logging the same logical field under varying paths such as `method`, `http.method`, and `request.method`: to log aggregators, these are three distinct fields.

Additionally, avoid embedding dots directly inside attribute key strings:

```go
logger.Info("request completed", "http.method", "GET")
logger.Info("request completed", slog.Group("http", "method", "GET"))
```

In `TextHandler`, both lines render as `http.method=GET`, but their JSON representations differ significantly:

```json
{"http.method":"GET"}
{"http":{"method":"GET"}}
```

Using un-dotted key names inside explicit groups preserves 1-to-1 parity between text and JSON representations. Maintaining a stable field hierarchy is just as critical as maintaining consistent field names and types.
