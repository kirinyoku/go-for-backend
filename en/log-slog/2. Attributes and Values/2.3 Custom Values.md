# Custom Values

Functions such as [`slog.String`](https://pkg.go.dev/log/slog#String), [`slog.Int`](https://pkg.go.dev/log/slog#Int), and [`slog.Bool`](https://pkg.go.dev/log/slog#Bool) cover Go's primitive types. For structs, slices, errors, or arbitrary objects, the [`log/slog`](https://pkg.go.dev/log/slog) package provides [`slog.Any`](https://pkg.go.dev/log/slog#Any). When default type formatting is insufficient, custom Go types can implement [`slog.LogValuer`](https://pkg.go.dev/log/slog#LogValuer) to customize their exact log representation.

This approach preserves domain object structure, hides sensitive fields from logs, and defers expensive string transformations until an event is actually processed by a log handler.

## Logging Arbitrary Values with `slog.Any`

The `slog.Any` function accepts a string key and an arbitrary value, returning an `slog.Attr`:

```go
type JobResult struct {
    JobID    string
    Attempts int
    Success  bool
}

result := JobResult{
    JobID:    "job-42",
    Attempts: 2,
    Success:  true,
}

logger.Info("job completed",
    slog.Any("result", result),
)
```

How the value is rendered depends on the active log handler. `JSONHandler` serializes arbitrary structs using standard `encoding/json` rules, encoding exported fields into a nested JSON object (timestamps omitted for brevity):

```json
{"level":"INFO","msg":"job completed","result":{"JobID":"job-42","Attempts":2,"Success":true}}
```

While convenient for one-off structs, struct field names become part of your log schema. If your log schema requires different key casing or structure than internal Go struct tags, specify the representation explicitly using attribute groups or `LogValuer`.

The [`slog.AnyValue`](https://pkg.go.dev/log/slog#AnyValue) function performs the same value conversion but returns an un-keyed [`slog.Value`](https://pkg.go.dev/log/slog#Value). This is useful when constructing values outside an attribute parameter list, such as inside a `LogValue` method:

```go
value := slog.AnyValue(result)
attr := slog.Attr{
    Key:   "result",
    Value: value,
}
```

For primitive types, prefer typed constructors like `slog.String` and `slog.Int` because they declare field types explicitly. Reserve `slog.Any` for complex types or instances lacking dedicated constructors.

## Logging Errors

The `slog` package does not provide a dedicated `slog.Error` attribute constructor. Errors are passed via `slog.Any` like any other object:

```go
logger.Error("job failed",
    slog.String("job_id", jobID),
    slog.Any("err", err),
)
```

Built-in handlers format top-level errors by calling their `.Error()` method. Calling `logger.Error` sets the log event's severity level, but it does not capture stack traces or unpack error chains into separate fields automatically.

Pre-converting errors via `err.Error()` is usually unnecessary. The original `error` value is preserved inside the attribute, allowing custom handlers or error types implementing `LogValuer` to customize how errors are rendered.

## The `LogValuer` Interface

The [`slog.LogValuer`](https://pkg.go.dev/log/slog#LogValuer) interface consists of a single method:

```go
type LogValuer interface {
    LogValue() slog.Value
}
```

When an attribute value implements `LogValuer`, handlers invoke `LogValue()` instead of applying default reflection or JSON formatting. This lets domain types define a stable log representation independent of internal Go field names or JSON tags:

```go
type JobResult struct {
    JobID    string
    Attempts int
    Success  bool
}

func (r JobResult) LogValue() slog.Value {
    return slog.GroupValue(
        slog.String("job_id", r.JobID),
        slog.Int("attempts", r.Attempts),
        slog.Bool("success", r.Success),
    )
}
```

Standard `slog.Any` calls automatically utilize this custom representation:

```go
logger.Info("job completed",
    slog.Any("result", result),
)
```

`JSONHandler` emits a nested group using the field names returned by `LogValue`:

```json
{"level":"INFO","msg":"job completed","result":{"job_id":"job-42","attempts":2,"success":true}}
```

Callers logging the event do not manually break out struct fields, yet the emitted schema remains explicit and maintainable alongside the type definition.

## Structuring Group Values via `GroupValue`

The [`slog.GroupValue`](https://pkg.go.dev/log/slog#GroupValue) function constructs an `slog.Value` holding multiple attributes. Unlike `slog.Group`, it does not take a group key: the parent attribute key at the call site becomes the enclosing group name.

```go
func (r JobResult) LogValue() slog.Value {
    return slog.GroupValue(
        slog.String("job_id", r.JobID),
        slog.Int("attempts", r.Attempts),
        slog.Bool("success", r.Success),
    )
}

logger.Info("job completed",
    slog.Any("result", result),
)
```

Here, `"result"` acts as the enclosing group key, while `job_id`, `attempts`, and `success` become its nested attributes.

If `LogValue` should resolve to a single scalar value instead of a group, return a scalar `slog.Value` constructor:

```go
type Region string

func (r Region) LogValue() slog.Value {
    return slog.StringValue(string(r))
}
```

Choose between a scalar value and `GroupValue` based on domain semantics: identifiers or enum-like types map naturally to scalar strings, while multi-field structs map to `GroupValue`.

## Masking Sensitive Data and Controlling Fields

`LogValuer` is particularly effective for types that contain both diagnostic metadata and sensitive credentials. Rather than logging an entire struct, `LogValue` can expose only safe fields:

```go
type UpstreamCredential struct {
    Service   string
    Token     string
    ExpiresAt time.Time
}

func (c UpstreamCredential) LogValue() slog.Value {
    return slog.GroupValue(
        slog.String("service", c.Service),
        slog.Time("expires_at", c.ExpiresAt),
    )
}
```

The application logs the object normally:

```go
logger.Info("upstream credential loaded",
    slog.Any("credential", credential),
)
```

The output contains diagnostic metadata while completely omitting `Token`:

```json
{"level":"INFO","msg":"upstream credential loaded","credential":{"service":"payments","expires_at":"2026-07-22T12:00:00Z"}}
```

::: warning
`LogValuer` controls formatting only when the struct is passed as a complete object. If code explicitly passes `credential.Token` as a separate attribute, `LogValuer` cannot mask it. Preventing sensitive data leaks still requires developer care at log call sites.
:::

## Deferred Value Resolution

Calling `slog.Any` does not invoke `LogValue()` immediately. Instead, value resolution is deferred until the handler processes the record *after* level filtering. If a log level is disabled, `LogValue()` is never executed. This enables lazy evaluation for expensive operations:

```go
type DeferredSummary struct {
    Build func() string
}

func (s DeferredSummary) LogValue() slog.Value {
    return slog.StringValue(s.Build())
}

logger.Debug("cache state",
    slog.Any("summary", DeferredSummary{
        Build: buildCacheSummary,
    }),
)
```

If `Debug` level is disabled, built-in handlers discard the event early, preventing `buildCacheSummary` from running. Note that Go function argument evaluation still constructs the `DeferredSummary` struct at call time; only the work inside `LogValue()` is deferred.

Attributes attached via `Logger.With` are processed when the derived logger is instantiated. Handlers may resolve `With` attributes immediately, so avoid using `LogValuer` in `With` calls if you expect values to re-evaluate on every log entry.

## Implementation Guidelines and Limitations

`LogValue()` methods should be deterministic, fast, and free of side effects. Loggers are used concurrently across goroutines, and handlers may invoke `LogValue()` multiple times. The method must not mutate application state or depend on invocation counters.

`LogValue()` must not return a value that directly re-triggers itself:

```go
func (r JobResult) LogValue() slog.Value {
    // Error: returns JobResult, causing infinite recursion
    return slog.AnyValue(r)
}
```

The [`slog.Value.Resolve`](https://pkg.go.dev/log/slog#Value.Resolve) method repeatedly resolves nested `LogValuer` implementations up to a safety depth limit to prevent infinite loops. `LogValue()` must return a terminal `slog.Value` (string, int, time, group, etc.) rather than the receiver type.

Built-in `TextHandler` and `JSONHandler` call `Resolve()` automatically. Custom handler authors must explicitly call `Resolve()` before reading attribute values:

```go
value := attr.Value.Resolve()
```

If a panic occurs inside `LogValue()`, `Resolve()` catches it and substitutes a diagnostic error value. However, do not rely on this fallback for routine error handling. Because `LogValue()` cannot return a Go `error`, execute fallible or blocking calculations before logging and pass resolved values to `slog`.
