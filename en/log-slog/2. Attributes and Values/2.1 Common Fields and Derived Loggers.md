# Common Fields and Derived Loggers

Certain attributes recur across many log events. For example, service names and component identifiers often need to be recorded during startup, HTTP request processing, and database operations:

```go
logger.Info("service started",
    "service", "users-api",
    "component", "http",
)

logger.Info("request completed",
    "service", "users-api",
    "component", "http",
    "status", 200,
)
```

Manually repeating these fields at every call site is error-prone: developers may forget a field, use inconsistent key names, or log wrong values. The [`Logger.With`](https://pkg.go.dev/log/slog#Logger.With) method allows you to bind common attributes to a derived logger once.

## Common Attributes via `With`

The `With` method returns a new `*slog.Logger` instance that automatically prepends the specified attributes to every log event it emits:

```go
baseLogger := slog.New(slog.NewTextHandler(os.Stdout, nil))

serviceLogger := baseLogger.With(
    slog.String("service", "users-api"),
)

httpLogger := serviceLogger.With(
    slog.String("component", "http"),
)

httpLogger.Info("service started")
httpLogger.Info("request completed", slog.Int("status", 200))
```

The resulting output automatically includes the bound fields:

```text
level=INFO msg="service started" service=users-api component=http
level=INFO msg="request completed" service=users-api component=http status=200
```

`With` accepts arguments using the same variadic conventions as logger call methods: you can pass alternating `"key", "value"` pairs or explicit `slog.Attr` values like `slog.String(...)`.

## Immutability of Logger Instances

Calling `With` does not mutate the existing `*slog.Logger` instance. Instead, it creates a separate derived instance. As a result, parent and child loggers remain completely independent and can be used concurrently without interference:

```go
baseLogger.Info("base logger ready")
serviceLogger.Info("service started")
httpLogger.Info("listener started", "address", ":8080")
```

```text
level=INFO msg="base logger ready"
level=INFO msg="service started" service=users-api
level=INFO msg="listener started" service=users-api component=http address=:8080
```

The `service` attribute is not retroactively added to `baseLogger`, nor does `component` mutate `serviceLogger`. In this hierarchy, all three loggers share the same output stream and format handler, but each maintains its own immutably bound attributes.

This behavior enables building clean logger hierarchies:

```text
baseLogger
└── serviceLogger          service=users-api
    ├── httpLogger         component=http
    └── databaseLogger     component=database
```

Each child logger inherits all attributes from its parent and attaches its own, without affecting sibling branches.

## Attribute Lifecycles

Because attributes added via `With` persist across every log event emitted by that derived logger, the lifecycle of an attribute must match the lifecycle of the logger instance itself.

| Attribute | Appropriate Scope | Recommended Pattern |
| :--- | :--- | :--- |
| `service` | Entire application lifetime | Service-level logger created via `With` |
| `component` | Long-lived service component | Component-level logger created via `With` |
| `request_id` | Single HTTP request | Request-scoped logger created via `With` |
| `status` | Single operation outcome | Event-specific attribute at call site |
| `duration` | Single operation outcome | Event-specific attribute at call site |

Dynamic parameters like HTTP status, execution latency, or error instances vary per operation and do not belong in long-lived loggers. Adding a dynamic parameter like `status` to a long-lived logger would incorrectly attach that status to every subsequent log entry.

## Component-Level Loggers

In `main`, a base service logger can be subdivided into component-specific loggers and injected as dependencies into application components:

```go
baseLogger := slog.New(slog.NewJSONHandler(os.Stdout, nil))
serviceLogger := baseLogger.With(
    slog.String("service", "users-api"),
)

httpLogger := serviceLogger.With(
    slog.String("component", "http"),
)
databaseLogger := serviceLogger.With(
    slog.String("component", "database"),
)

handler := NewUserHandler(httpLogger)
repository := NewUserRepository(databaseLogger)
```

`UserHandler` and `UserRepository` accept a pre-configured `*slog.Logger`. Component implementations do not need to know the global log format or manually repeat service and component metadata. Individual methods log only operation-specific details:

```go
func (r *UserRepository) FindUser(userID int64) {
    // Database query omitted for brevity.

    r.logger.Info("user loaded",
        slog.Int64("user_id", userID),
    )
}
```

The resulting log record automatically includes `service=users-api` and `component=database`, even though `FindUser` specifies only `user_id`.

## Request-Scoped Loggers

When handling an HTTP request or asynchronous task that emits multiple log events, you can attach a unique `request_id` to a request-scoped derived logger:

```go
func handleRequest(logger *slog.Logger, requestID string) {
    requestLogger := logger.With(
        slog.String("request_id", requestID),
    )

    requestLogger.Info("request accepted")

    // Request processing logic omitted.

    requestLogger.Info("request completed",
        slog.Int("status", 200),
    )
}
```

This derived logger exists only for the duration of the HTTP request. It must not be stored in long-lived global structs; otherwise, one client's `request_id` could leak into log events of subsequent requests. For isolated, single-event log calls, pass `request_id` directly in the call method instead of creating a derived logger.

Because `With` is non-mutating and thread-safe, concurrent goroutines processing requests can safely derive their own request loggers from a shared `httpLogger`. Each request logger maintains its own distinct `request_id` without corrupting the shared parent logger.

## Attribute Collisions and Key Overwriting

Passing an attribute key in a call method that matches a key previously bound via `With` does *not* overwrite or replace the bound attribute:

```go
serviceLogger := baseLogger.With("service", "users-api")

serviceLogger.Info("worker started",
    "service", "background-worker",
)
```

The default `TextHandler` emits both values:

```text
level=INFO msg="worker started" service=users-api service=background-worker
```

Downstream log aggregators handle duplicate keys inconsistently: some take the first value, some take the last, and others store both as an array. Do not rely on attribute overwriting. If an event belongs to a different service context or subsystem, derive a logger from an appropriate parent logger or use a distinct key name (e.g., `target_service`).

Derived loggers are best suited for common contextual attributes that remain constant throughout the logger's lifecycle. High-level attributes like `service` and `component` belong on long-lived component loggers, `request_id` belongs on request-scoped loggers, and per-operation results belong directly in event log calls.
