# From Text to Structured Event

The [`log/slog`](https://pkg.go.dev/log/slog) package, introduced in Go 1.21, is designed for structured logging. Instead of building a pre-formatted text string, an application passes a discrete event to the logger: a short message, a severity level, and key-value attributes. These records can be ingested by centralized log aggregators and queried by request ID, HTTP route, status code, or other fields. When the output format and log collector preserve these attributes, searching and filtering require no custom message string parsing.

## Plain Text Records via `log.Printf`

For basic, line-oriented text output, standard [`log.Printf`](https://pkg.go.dev/log#Printf) is straightforward:

```go
method := http.MethodGet
route := "/users/{id}"
status := http.StatusOK
duration := 37 * time.Millisecond

log.Printf(
    "request completed: method=%s route=%s status=%d duration=%s",
    method,
    route,
    status,
    duration,
)
```

With default settings, the resulting output looks similar to:

```text
2026/07/22 10:15:42 request completed: method=GET route=/users/{id} status=200 duration=37ms
```

This formatted string is human-readable and clearly indicates which request completed and how long it took. For lightweight utilities or simple diagnostic output, this approach may be sufficient.

The drawback arises when logs need to be processed programmatically as structured data. `log.Printf` formats arguments and writes the output as a single string. Once emitted, field boundaries and native Go types are lost: `status` becomes plain text, and `duration` is permanently formatted into a string representation of `time.Duration`.

## Structured Logging via `slog.Info`

The [`slog.Info`](https://pkg.go.dev/log/slog#Info) function accepts an event message followed by key-value attribute pairs:

```go
slog.Info("request completed",
    "method", method,
    "route", route,
    "status", status,
    "duration", duration,
)
```

The arguments following the message string are evaluated sequentially as key-value pairs, where each string key names the value that follows. This allows attaching multiple independent attributes to a single event.

With default text configuration, the visual output changes slightly:

```text
2026/07/22 10:15:42 INFO request completed method=GET route=/users/{id} status=200 duration=37ms
```

Following the timestamp, a structured record consists of three main components:

| Component | Example | Purpose |
| :--- | :--- | :--- |
| Severity Level | `INFO` | Indicates the operational importance of the event |
| Message | `request completed` | Briefly describes what occurred |
| Attributes | `method`, `route`, `status`, `duration` | Provide specific context and dynamic details for the event instance |

The fundamental distinction occurs *before* the output string is generated: the application supplies `status` as an integer, `duration` as a `time.Duration`, and `method` and `route` as distinct string values. `slog` receives these arguments independently, preserving their distinct identities for the underlying log handler.

## Separating Messages from Attributes

The log message answers the core question: *what happened?* Attributes describe the specific instance. If you concatenate values into a single string upfront using [`fmt.Sprintf`](https://pkg.go.dev/fmt#Sprintf), structured fields become buried within the message text:

```go
message := fmt.Sprintf(
    "%s %s completed with status %d in %s",
    method,
    route,
    status,
    duration,
)
slog.Info(message)
```

In this call, while timestamp and severity level remain structured metadata, all HTTP request parameters are merged into a single unstructured string. The structured alternative keeps them separate:

```go
slog.Info("request completed",
    "method", http.MethodGet,
    "route", "/users/{id}",
    "status", http.StatusOK,
    "duration", 37*time.Millisecond,
)
```

The message `"request completed"` remains identical across all completed request events. Parameters like HTTP method, route, status code, and duration change per request and are therefore passed as named attributes.

::: tip
If a value changes frequently across calls but does not alter the fundamental meaning of the event, it belongs in an attribute. However, avoid converting every single word into a separate field. Keep the message concise and human-readable, while reserving attributes for data that needs to be queried, filtered, or correlated downstream.
:::

::: info
`slog.Info` does not perform string formatting like `fmt.Printf`. Keep static template text in the message and pass dynamic variables as key-value attributes.
:::

## Event Structure vs. Output Format

You can manually construct `key=value` strings with standard `log.Printf`:

```go
log.Printf(
    "msg=request_completed method=%s status=%d",
    method,
    status,
)
```

While this output looks structured, `log.Printf` still treats it as a single string. The separation into `msg`, `method`, and `status` exists purely as a convention for an external parser to decode.

In contrast, when using `slog.Info("request completed", "status", status)`, the message, key, and integer value remain distinct entities prior to formatting. Thus, structured logging is defined not by JSON syntax or `key=value` formatting, but by how the application delivers event data to the logger API.

Next, the event is processed by a log handler ([`slog.Handler`](https://pkg.go.dev/log/slog#Handler))—the component in `slog` responsible for formatting and routing entries. For instance, [`slog.JSONHandler`](https://pkg.go.dev/log/slog#JSONHandler) formats the preceding log entry as JSON:

```json
{"time":"2026-07-22T10:15:42.123456789+02:00","level":"INFO","msg":"request completed","status":200}
```

Here, `status` is preserved as a JSON number rather than embedded in a message string. JSON is simply one output format—the underlying event structure was established when `slog.Info` was called.

## Consuming Structured Log Records

When your log handler and downstream collector preserve named attributes, you can query and process logs programmatically without text parsing:

- Filter log entries matching `status >= 500`
- Group and aggregate error rates by `route`
- Measure latency distribution across identical endpoint calls

`log/slog` itself does not include log storage or a query engine. Its responsibility ends once the event message, level, and attributes are delivered to the handler. Data storage, indexing, and visualization are managed by your log infrastructure.

Event structure is established at the call site: the message describes the event type, while attributes preserve the parameters of the specific call. Before selecting an output format, identify which parts of your log entry are static and which data should remain structured fields.
