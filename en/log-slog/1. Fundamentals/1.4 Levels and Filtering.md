# Levels and Filtering

After creating a logger, you might encounter an unexpected result:

```go
logger.Debug("configuration loaded")
logger.Info("service started")
```

With default handler settings, only `"service started"` appears in the output. The `Debug` entry is not missing due to a bug: by default, built-in log handlers ignore events below the `Info` level.

A severity level specifies the operational importance of a log event, while a handler's minimum level threshold determines whether that event is written. This separation lets you disable detailed diagnostic logging in production without removing log calls from your application codebase.

## Standard Log Levels

The [`slog.Logger`](https://pkg.go.dev/log/slog#Logger) type provides four primary methods for logging at standard severity levels:

```go
logger.Debug("cache lookup started")
logger.Info("request completed")
logger.Warn("cache unavailable")
logger.Error("database query failed")
```

Choose a log level based on the operational outcome of the action rather than just the message wording:

| Level | Suitable Scenario |
| :--- | :--- |
| `Debug` | Fine-grained diagnostic details useful during development or troubleshooting, but usually unnecessary during normal operation. |
| `Info` | Expected, notable application events (such as service startup or request completion). |
| `Warn` | Non-fatal operational issues where execution continues, potentially using a fallback path. |
| `Error` | Failures where an operation could not be completed successfully. |

For instance, a cache miss is a routine `Debug` event if the service falls back to the database as expected. If the cache server drops offline entirely but fallback database queries succeed, `Warn` is appropriate. If the database query subsequently fails and the request cannot proceed, the event escalates to `Error`.

Severity levels reflect outcomes from the perspective of the application. An HTTP `4xx` response is not necessarily a log `Error`: validation errors or missing resources are routine, expected client outcomes. The key is establishing a consistent logging policy across your application services.

## Handler Level Thresholds

Built-in severity levels form an ordered scale of integer values:

```text
Debug (-4) < Info (0) < Warn (4) < Error (8)
```

A log handler emits a record only if the event's level is greater than or equal to the handler's configured minimum threshold. Consequently, raising the minimum threshold reduces output volume:

| Threshold | Recorded Log Levels |
| :--- | :--- |
| `Debug` | `Debug`, `Info`, `Warn`, `Error` |
| `Info` | `Info`, `Warn`, `Error` |
| `Warn` | `Warn`, `Error` |
| `Error` | `Error` |

You set the threshold using the `Level` field of [`slog.HandlerOptions`](https://pkg.go.dev/log/slog#HandlerOptions):

```go
handler := slog.NewTextHandler(os.Stdout, &slog.HandlerOptions{
    Level: slog.LevelWarn,
})
logger := slog.New(handler)

logger.Debug("cache lookup started")
logger.Info("request completed")
logger.Warn("cache unavailable")
logger.Error("database query failed")
```

The first two log events (`Debug` and `Info`) are discarded. Only `Warn` and `Error` entries are emitted:

```text
time=... level=WARN msg="cache unavailable"
time=... level=ERROR msg="database query failed"
```

If `HandlerOptions.Level` is `nil`, both `TextHandler` and `JSONHandler` default to `slog.LevelInfo`. This explains why passing `nil` as handler options in previous examples filtered out `Debug` messages.

To enable verbose diagnostic logging, set the threshold explicitly to [`slog.LevelDebug`](https://pkg.go.dev/log/slog#LevelDebug):

```go
handler := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
    Level: slog.LevelDebug,
})
logger := slog.New(handler)
```

Set the log threshold at application startup. This allows the same component code to produce detailed debug logs in local development while remaining concise in production.

## Errors and Control Flow

The [`Logger.Error`](https://pkg.go.dev/log/slog#Logger.Error) method records a log event at the `Error` severity level. It does not return a Go `error` value, modify existing error instances, or interrupt program execution.

When an error requires terminating or aborting an operation, explicit control flow remains necessary:

```go
if err != nil {
    logger.Error("configuration loading failed", "err", err)
    return err
}
```

Logging an event and returning an error serve different purposes: logging records diagnostic details for operators, while `return` propagates the error up the call stack. Invoking `logger.Error` without returning would allow execution to continue.

`log/slog` contains no counterparts to legacy `Fatal` or `Panic` methods. Decisions to exit (`os.Exit`), panic, or return are made by application control flow explicitly. This separation keeps process control transparent in code rather than hidden inside a logger call.

## Dynamic Thresholds via `LevelVar`

Using a constant value like `slog.LevelWarn` sets a fixed threshold. If you need to change the logging threshold dynamically at runtime without restarting the application, use [`slog.LevelVar`](https://pkg.go.dev/log/slog#LevelVar):

```go
var minimumLevel slog.LevelVar // Zero value corresponds to Info level.

handler := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
    Level: &minimumLevel,
})
logger := slog.New(handler)

logger.Debug("configuration loaded") // Discarded.

minimumLevel.Set(slog.LevelDebug)
logger.Debug("configuration loaded") // Written.
```

The handler queries the `LevelVar` dynamically on every log evaluation, so calling `Set()` updates existing loggers instantly. `LevelVar` is safe for concurrent access. The trigger to adjust log levels—whether from OS signals, admin HTTP endpoints, or remote configuration managers—is handled by your application logic.

For most microservices, static startup configuration is sufficient. Dynamic thresholds are useful when you need to temporarily enable debug logging in live environments for troubleshooting.

## Custom Log Levels

The [`slog.Level`](https://pkg.go.dev/log/slog#Level) type is an underlying integer type, with built-in levels spaced four units apart: `Debug` (-4), `Info` (0), `Warn` (4), and `Error` (8). If necessary, custom levels can be defined between these values and emitted using [`Logger.Log`](https://pkg.go.dev/log/slog#Logger.Log):

```go
const LevelNotice = slog.LevelInfo + 2

logger.Log(ctx, LevelNotice, "configuration reloaded")
```

Standard handlers format custom levels as `INFO+2`. For most applications, the four standard levels are more than enough; custom levels are only practical when supported by a clear team policy and log aggregator mapping.

Log levels separate routine diagnostics from actionable issues, while handler thresholds control log volume without code changes. Ultimately, log levels describe event severity, leaving program control flow to explicit Go language constructs.
