# Logger Configuration

Package-level functions such as [`slog.Info`](https://pkg.go.dev/log/slog#Info) allow you to start logging immediately without initial setup. However, in production applications, you usually construct a `*slog.Logger` explicitly during application startup to centralize decisions about log output formatting and destination streams.

Setting up the logger at startup keeps logging configuration localized. The rest of the codebase receives a pre-configured `*slog.Logger` instance and focuses purely on describing events, decoupled from how and where those events are written.

## Creating a Custom Logger

Constructing a logger requires a log handler. For example, [`slog.NewJSONHandler`](https://pkg.go.dev/log/slog#NewJSONHandler) creates a handler that formats log events as JSON:

```go
handler := slog.NewJSONHandler(os.Stdout, nil)
logger := slog.New(handler)

logger.Info("service started", "address", ":8080")
```

This generates output formatted as:

```json
{"time":"2026-07-22T10:15:42.123+02:00","level":"INFO","msg":"service started","address":":8080"}
```

In this architecture, [`slog.Logger`](https://pkg.go.dev/log/slog#Logger) acts as the entry point that accepts log events from the application. The handler, defined by the [`slog.Handler`](https://pkg.go.dev/log/slog#Handler) interface, determines which records to process and how to format them for the target output stream.

The [`slog.New`](https://pkg.go.dev/log/slog#New) function connects these parts and returns a new `*slog.Logger`. Passing a `nil` handler to `slog.New` causes a panic. Passing `nil` as the options argument (the second parameter in `NewJSONHandler`) has a different meaning: it instructs the handler to use default configuration settings.

## Text and JSON Representations

The standard library includes two built-in handlers: [`slog.NewTextHandler`](https://pkg.go.dev/log/slog#NewTextHandler) and [`slog.NewJSONHandler`](https://pkg.go.dev/log/slog#NewJSONHandler). Comparing them on the same log event highlights their differences:

```go
textLogger := slog.New(slog.NewTextHandler(os.Stdout, nil))
jsonLogger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

textLogger.Info("request completed", "method", "GET", "status", 200)
jsonLogger.Info("request completed", "method", "GET", "status", 200)
```

[`slog.TextHandler`](https://pkg.go.dev/log/slog#TextHandler) formats log attributes as `key=value` pairs:

```text
time=2026-07-22T10:15:42.123+02:00 level=INFO msg="request completed" method=GET status=200
```

`slog.JSONHandler` formats each log event as a standalone JSON object:

```json
{"time":"2026-07-22T10:15:42.123+02:00","level":"INFO","msg":"request completed","method":"GET","status":200}
```

Both handlers receive the exact same structured event from the application. The distinction lies entirely in the output representation: `TextHandler` is easy to read in a local terminal during development, while `JSONHandler` produces newline-delimited JSON that log collection pipelines can easily ingest and index without complex parsing.

Switching between handlers requires zero changes to application code. Your domain logic continues calling `.Info()`, `.Error()`, etc. with messages and attributes, completely agnostic of downstream output formats.

## Output Streams

Constructors for both `TextHandler` and `JSONHandler` accept an [`io.Writer`](https://pkg.go.dev/io#Writer). In server applications, this is commonly [`os.Stdout`](https://pkg.go.dev/os#Stdout) or [`os.Stderr`](https://pkg.go.dev/os#Stderr), but `slog` places no restrictions on the target stream:

```go
func newLogger(out io.Writer) *slog.Logger {
    handler := slog.NewJSONHandler(out, nil)
    return slog.New(handler)
}
```

This factory function can be called with `os.Stdout` in production and with a `bytes.Buffer` in unit tests. The log handler does not take ownership of the `io.Writer`. `slog.Logger` provides no `Close` or `Flush` methods, so resource management (such as closing log files) remains the responsibility of the code that created the underlying stream.

::: tip
In containerized environments (such as Kubernetes or Docker), standard practice is to write log events directly to standard output (`os.Stdout`) and delegate log collection, filtering, and file rotation to the container runtime or external log collector.
:::

## Injecting Loggers into Components

A common pattern is to construct a base `*slog.Logger` in `main` and pass it to application services as an explicit dependency:

```go
type UserService struct {
    logger *slog.Logger
}

func NewUserService(logger *slog.Logger) *UserService {
    return &UserService{logger: logger}
}

func (s *UserService) CreateUser(userID int64) {
    // User creation logic omitted.
    s.logger.Info("user created", "user_id", userID)
}

func main() {
    handler := slog.NewJSONHandler(os.Stdout, nil)
    logger := slog.New(handler)

    users := NewUserService(logger)
    users.CreateUser(42)
}
```

Here, only `main` knows that logs are formatted as JSON and written to `os.Stdout`. `UserService` depends strictly on `*slog.Logger` without coupling itself to specific output streams or formats. This keeps logging configuration centralized and maintainable.

A single `*slog.Logger` instance is completely safe for concurrent use across multiple goroutines. There is no need to instantiate a new base logger per request or per goroutine. Later, you can derive specialized loggers with pre-attached context attributes from this base logger without modifying its original configuration.

## The Default Logger

The [`slog.Default`](https://pkg.go.dev/log/slog#Default) function returns the process-wide default logger. Top-level package functions invoke this default instance under the hood:

```go
slog.Info("service started")

logger := slog.Default()
logger.Info("service ready")
```

Using the default logger is convenient for simple scripts, quick experiments, or small utilities. For larger applications, explicitly passing `*slog.Logger` via constructor injection makes component dependencies explicit and avoids hidden global state.

## Interoperability with the `log` Package

Legacy Go code and third-party packages often rely on the standard [`log`](https://pkg.go.dev/log) package. `log/slog` provides bridge utilities to route legacy `log` output into your `slog` handler chain. The exact setup depends on whether the legacy code accepts an explicit `*log.Logger` or calls package-level functions.

### Bridges for `*log.Logger`

When a third-party library expects a [`*log.Logger`](https://pkg.go.dev/log#Logger) parameter, you can bridge it to `slog` using [`slog.NewLogLogger`](https://pkg.go.dev/log/slog#NewLogLogger):

```go
legacyLogger := slog.NewLogLogger(
    logger.Handler(),
    slog.LevelWarn,
)

legacyLogger.Printf("connection attempt failed: %v", err)
```

Calls to this bridge logger create `slog` records at `Warn` level and forward them directly to the underlying `slog.Handler`. Because standard `log.Printf` produces plain text messages, arguments are not automatically parsed into separate `slog` attributes.

The bridge logger suppresses standard `log` date and time prefixes so that timestamps and severity levels are handled cleanly and consistently by the `slog` handler.

### Redirecting Package-Level `log.Print` and `log.Printf`

If legacy code calls package-level functions like `log.Printf`, you can redirect all standard `log` output to your `slog` handler using [`slog.SetDefault`](https://pkg.go.dev/log/slog#SetDefault):

```go
slog.SetDefault(logger)
slog.SetLogLoggerLevel(slog.LevelWarn)

log.Printf("legacy subsystem is unavailable: %v", err)
```

The [`slog.SetLogLoggerLevel`](https://pkg.go.dev/log/slog#SetLogLoggerLevel) function, added in Go 1.22, specifies the severity level assigned to redirected standard `log` calls (defaulting to `Info` if unset).

Both `slog.SetDefault` and `slog.SetLogLoggerLevel` modify process-wide settings and should be executed once during application startup. Note that redirecting default logging does not affect custom instances created via `log.New`; those must be bridged using `slog.NewLogLogger`.
