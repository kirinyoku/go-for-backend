# Attributes and Event Schema

An attribute links a field name to a specific value. In a log event recording the completion of an HTTP request, attributes might include the HTTP method, route template, status code, and request duration:

```go
logger.Info("request completed",
    "method", "GET",
    "route", "/users/{id}",
    "status", 200,
    "duration", 37*time.Millisecond,
)
```

Each log entry captures data for an individual request, but the schema of field names remains uniform across all events of this type. This consistency enables searching, filtering, and aggregating events by known keys. Attributes effectively establish an event schema, for which the application is responsible for keeping stable.

## Key-Value Pairs

The [`Logger.Info`](https://pkg.go.dev/log/slog#Logger.Info) method (along with `Debug`, `Warn`, and `Error`) accepts variadic arguments of type `...any`. When called with alternating key-value pairs, arguments are read sequentially: each string key defines the field name for the value that follows.

```go
logger.Info("user loaded",
    "user_id", userID,
    "from_cache", fromCache,
)
```

Because a key represents the semantic meaning of a field, its name should be fixed in code. Dynamic runtime data (such as a specific user ID) should be passed as the value of a constant key like `"user_id"`, rather than used as the key itself:

```go
// Incorrect: field name changes with runtime data
logger.Info("user loaded", strconv.FormatInt(userID, 10), true)

// Correct: field name remains constant, only the value varies
logger.Info("user loaded", "user_id", userID)
```

Using dynamic keys makes log querying unpredictable and bloats log indexing systems with thousands of unique field names. Variable runtime values must always be assigned to constant keys.

## Explicit Typed Attributes

Instead of alternating key-value pairs, you can pass explicit [`slog.Attr`](https://pkg.go.dev/log/slog#Attr) values. Each `slog.Attr` pairs a string key with a strongly typed [`slog.Value`](https://pkg.go.dev/log/slog#Value):

```go
logger.Info("request completed",
    slog.String("method", "GET"),
    slog.String("route", "/users/{id}"),
    slog.Int("status", 200),
    slog.Duration("duration", 37*time.Millisecond),
)
```

Explicit attribute syntax is slightly more verbose, but it binds key and value in a single expression and enforces type safety at the call site. The alternating pair `"status", 200` and the typed attribute `slog.Int("status", 200)` construct equivalent log fields. You can use whichever style best fits your team's code conventions.

The `slog` package provides helper functions for common types:

| Constructor | Value Type | Common Use Case |
| :--- | :--- | :--- |
| [`slog.String`](https://pkg.go.dev/log/slog#String) | `string` | HTTP method, route, or identifier |
| [`slog.Int`](https://pkg.go.dev/log/slog#Int) | `int` | HTTP status code, port, count |
| [`slog.Duration`](https://pkg.go.dev/log/slog#Duration) | `time.Duration` | Latency, execution time |
| [`slog.Any`](https://pkg.go.dev/log/slog#Any) | Arbitrary Go type | Errors or custom structs |

Choose constructor functions based on the actual data type rather than how the field might look when rendered as text. For instance, passing HTTP status via `slog.Int("status", 200)` preserves it as a numeric value for log handlers and aggregators that support numeric indexing, rather than converting it to a string `"200"` prematurely.

`slog.Any` is used when no dedicated type helper exists:

```go
logger.Error("database query failed",
    slog.String("operation", "find_user"),
    slog.Any("err", err),
)
```

Avoid passing large struct instances directly through `slog.Any`. Entire structs can inadvertently expose sensitive data (such as passwords or personal tokens) or bloat log records with irrelevant fields. Select and log only the specific fields required for operational diagnostics.

## Maintaining a Stable Event Schema

For recurring application events, define consistent field names and types upfront. For example, a completed HTTP request might follow this schema:

| Key | Type | Example |
| :--- | :--- | :--- |
| `method` | `string` | `GET` |
| `route` | `string` | `/users/{id}` |
| `status` | `int` | `200` |
| `duration` | `time.Duration` | `37ms` |

To maintain schema consistency:

- **Use standardized key names**: do not alternate between `status`, `status_code`, and `code`.
- **Preserve consistent types**: `status` should always be an integer, never a string in some events and an integer in others.
- **Log route templates**: record `/users/{id}` rather than parameterized paths like `/users/42` to allow route aggregation.
- **Never use dynamic user data as keys**.

The `log/slog` package does not enforce a rigid schema validator at runtime. Maintaining schema discipline is an application-level responsibility: if the same event type continuously alters key names or field types, downstream log indexing degrades into parsing ad-hoc strings.

## Duplicate Keys

`log/slog` allows adding duplicate keys within a single call:

```go
logger.Info("request completed",
    "status", 200,
    "status", 500,
)
```

The default `TextHandler` renders both entries:

```text
level=INFO msg="request completed" status=200 status=500
```

From an operational perspective, duplicate keys create ambiguity: downstream query tools cannot reliably determine which status value represents the final outcome. Because log handlers and aggregators handle duplicate keys differently, always keep key names unique within an event. When logging values from multiple stages of an operation, distinguish them with distinct key names like `status` and `upstream_status`.

## Argument Mismatches and `!BADKEY`

The alternating key-value syntax requires every key to have a corresponding value. If an odd number of variadic arguments is passed, `slog` does not panic; instead, it captures the dangling argument under a special `!BADKEY` attribute:

```go
logger.Info("request completed",
    "status", 200,
    "duration",
)
```

This renders text output similar to:

```text
level=INFO msg="request completed" status=200 !BADKEY=duration
```

The `!BADKEY` attribute also appears when passing arguments where `slog` expects a string key or an `slog.Attr`:

```go
userID := 42
logger.Info("user %d", userID)
```

Because `slog.Info` does not interpret `fmt.Printf` format specifiers like `%d`, the message text remains `"user %d"`, and `userID` is appended as `!BADKEY=42`. The correct call separates static event messages from dynamic attributes:

```go
logger.Info("user loaded", "user_id", userID)
```

Starting with Go 1.22, [`go vet`](https://pkg.go.dev/cmd/vet) includes static analysis checks for `log/slog` calls with alternating key-value pairs. It catches missing values and mismatched argument types at compile time:

```sh
go vet ./...
```
