# Custom Handler

The [`slog.Handler`](https://pkg.go.dev/log/slog#Handler) interface is the primary extension point in [`log/slog`](https://pkg.go.dev/log/slog). A custom handler is warranted when you need a new log format, a specialized storage backend, or an integration with an API that has no off-the-shelf implementation.

If your requirements reduce to JSON or text output, modifying built-in fields, or filtering by level, a standard handler with [`HandlerOptions`](/en/log-slog/advanced/handler-configuration) is the right tool. A full `Handler` implementation is responsible for enforcing the entire attribute, group, derived logger, and concurrency contracts on its own.

Throughout this article the running example will be `MemoryHandler`, which stores events as `map[string]any` slices. It is not a production-grade log sink, but it illustrates the full `Handler` contract without the noise of a network client or a custom serialization format.

## The Four Handler Methods

The `slog.Handler` interface has exactly four methods:

```go
type Handler interface {
    Enabled(context.Context, slog.Level) bool
    Handle(context.Context, slog.Record) error
    WithAttrs([]slog.Attr) slog.Handler
    WithGroup(string) slog.Handler
}
```

Each method covers a distinct area of responsibility:

| Method | Responsibility |
| :--- | :--- |
| `Enabled` | Decide whether an event at the given level should be created and processed. |
| `Handle` | Process a fully-formed `slog.Record`. |
| `WithAttrs` | Produce a derived handler with the provided persistent attributes pre-bound. |
| `WithGroup` | Produce a derived handler with an open group namespace. |

`WithAttrs` and `WithGroup` must not mutate the receiver. Base loggers and their derived counterparts may be used concurrently, so immutable configuration snapshots are the cleanest approach. Mutable sink state should live separately.

## `MemoryHandler` State

The handler holds a minimum level, persistent attributes, an open-group path, and a shared event list:

```go
type MemoryHandler struct {
    state  *memoryState
    level  slog.Leveler
    attrs  map[string]any
    groups []string
}

type memoryState struct {
    mu     sync.Mutex
    events []map[string]any
}

func NewMemoryHandler(level slog.Leveler) *MemoryHandler {
    if level == nil {
        level = slog.LevelInfo
    }

    return &MemoryHandler{
        state: &memoryState{},
        level: level,
        attrs: make(map[string]any),
    }
}
```

All derived handler instances share a single `*memoryState` pointer because they represent one logical event sink. The `attrs` and `groups` fields are never modified after construction; methods that produce derived handlers allocate fresh values instead.

The `slog.Leveler` parameter accepts both a static `slog.Level` and a dynamic `*slog.LevelVar`. A `nil` value defaults to `LevelInfo`, consistent with how the standard handlers behave.

## Early Level Filtering

`Enabled` compares the event level against the current threshold:

```go
func (h *MemoryHandler) Enabled(
    _ context.Context,
    level slog.Level,
) bool {
    return level >= h.level.Level()
}
```

The logger calls `Enabled` before allocating a full `slog.Record`. If `Enabled` returns `false`, `Handle` is never called. As a consequence, `Enabled` must be fast and must not perform I/O.

Context can inform level filtering when per-request overrides are needed, but for a static threshold, ignoring it entirely is the correct choice.

## Processing a Record

`Handle` receives a fully formed [`slog.Record`](https://pkg.go.dev/log/slog#Record). `MemoryHandler` builds an event map, copies built-in fields, and iterates over call-site attributes using [`Record.Attrs`](https://pkg.go.dev/log/slog#Record.Attrs):

```go
func (h *MemoryHandler) Handle(
    _ context.Context,
    record slog.Record,
) error {
    event := cloneMap(h.attrs)

    record.Attrs(func(attr slog.Attr) bool {
        addAttrAtGroups(event, h.groups, attr)
        return true
    })

    if !record.Time.IsZero() {
        event[slog.TimeKey] = record.Time
    }
    event[slog.LevelKey] = record.Level
    event[slog.MessageKey] = record.Message

    h.state.mu.Lock()
    h.state.events = append(h.state.events, event)
    h.state.mu.Unlock()

    return nil
}
```

A zero `record.Time` is intentionally omitted — this is a requirement of the `Handler` contract. Level and message are stored under the standard `slog` key constants so that tooling, including `testing/slogtest`, can interpret them uniformly.

The `record.PC` field carries call-site source information. `MemoryHandler` does not expose source location; a handler that does must skip a zero `PC` and resolve non-zero values via the `runtime` package or `Record.Source`.

The context passed to `Handle` is intended for reading request-scoped values, not for lifecycle control. A cancelled context must not prevent the event from being recorded — log calls about timeouts or cancellations frequently happen *after* the context has already been cancelled.

Because events are represented as maps, a duplicate key silently overwrites its predecessor. The built-in fields occupy the reserved keys `time`, `level`, and `msg`. When key ordering or duplicate preservation matters, a sequence of attributes is a better internal representation than a map.

## Values and Groups

An attribute's value might be a plain value, a `slog.LogValuer` implementation, or a nested group. Before reading a value, a handler should call [`Value.Resolve`](https://pkg.go.dev/log/slog#Value.Resolve):

```go
func addAttr(
    target map[string]any,
    attr slog.Attr,
) {
    attr.Value = attr.Value.Resolve()
    if attr.Equal(slog.Attr{}) {
        return
    }

    if attr.Value.Kind() == slog.KindGroup {
        fields := make(map[string]any)
        for _, child := range attr.Value.Group() {
            addAttr(fields, child)
        }

        if len(fields) == 0 {
            return
        }
        if attr.Key == "" {
            mergeMap(target, fields)
            return
        }

        target[attr.Key] = fields
        return
    }

    target[attr.Key] = valueAsAny(attr.Value)
}
```

`Resolve` repeatedly invokes `LogValue` until it obtains a non-`LogValuer` value and guards against infinite recursion. Calling `LogValue` directly once does not provide these guarantees.

A zero-value `slog.Attr{}` is silently skipped. A group with an empty key is inlined into the current scope. A group with no fields produces no output, avoiding empty objects in the result. For nested groups, the function recurses, applying the same rules at each level.

All known `slog.Value` kinds convert cleanly via `Value.Any`. Returning the string representation for unrecognized future kinds is safer than panicking:

```go
func valueAsAny(value slog.Value) any {
    switch value.Kind() {
    case slog.KindAny,
        slog.KindBool,
        slog.KindDuration,
        slog.KindFloat64,
        slog.KindInt64,
        slog.KindString,
        slog.KindTime,
        slog.KindUint64:
        return value.Any()
    default:
        return value.String()
    }
}
```

The remaining helpers create the open-group path and copy maps before mutating them:

```go
func addAttrAtGroups(
    event map[string]any,
    groups []string,
    attr slog.Attr,
) {
    fields := make(map[string]any)
    addAttr(fields, attr)
    if len(fields) == 0 {
        return
    }

    target := groupTarget(event, groups)
    mergeMap(target, fields)
}

func groupTarget(
    event map[string]any,
    groups []string,
) map[string]any {
    target := event
    for _, name := range groups {
        child, ok := target[name].(map[string]any)
        if !ok {
            child = make(map[string]any)
            target[name] = child
        }
        target = child
    }
    return target
}

func mergeMap(
    target map[string]any,
    source map[string]any,
) {
    for key, value := range source {
        target[key] = value
    }
}

func cloneMap(source map[string]any) map[string]any {
    clone := make(map[string]any, len(source))
    for key, value := range source {
        if group, ok := value.(map[string]any); ok {
            value = cloneMap(group)
        }
        clone[key] = value
    }
    return clone
}
```

Group nesting is only created when there is at least one non-empty attribute to place inside it. This prevents `logger.WithGroup("http").Info("request started")` from leaving an empty `http` object in the event.

`cloneMap` performs a deep copy of nested maps produced by the handler itself but does not attempt to deep-copy arbitrary values passed via `slog.Any`. If an attribute contains a pointer, map, or slice, application code must not mutate that data while the handler might still reference it.

## Persistent Attributes and Groups

`WithAttrs` constructs a new handler instance from a copy of the accumulated fields and merges the incoming attributes into the current group scope:

```go
func (h *MemoryHandler) WithAttrs(
    attrs []slog.Attr,
) slog.Handler {
    next := &MemoryHandler{
        state:  h.state,
        level:  h.level,
        attrs:  cloneMap(h.attrs),
        groups: h.groups,
    }

    for _, attr := range attrs {
        addAttrAtGroups(next.attrs, next.groups, attr)
    }

    return next
}
```

The `slog.Handler` specification allows the implementation to retain the `attrs` slice, but this handler eagerly converts attributes into its internal map representation. The original `MemoryHandler` instance remains immutable.

`WithGroup` copies the current group path and appends the new name:

```go
func (h *MemoryHandler) WithGroup(
    name string,
) slog.Handler {
    if name == "" {
        return h
    }

    groups := make(
        []string,
        len(h.groups),
        len(h.groups)+1,
    )
    copy(groups, h.groups)
    groups = append(groups, name)

    return &MemoryHandler{
        state:  h.state,
        level:  h.level,
        attrs:  h.attrs,
        groups: groups,
    }
}
```

Explicit slice copying preserves ordering along the chain. For example, after `logger.With("service", "users").WithGroup("http").With("method", "GET")`, the `service` field stays at root level while `method` is scoped under the `http` group.

An empty group name has no effect on attribute qualification, so the method returns the receiver unchanged.

## Reading Stored Events

To expose collected events, the handler returns a snapshot under the same lock that protects writes:

```go
func (h *MemoryHandler) Events() []map[string]any {
    h.state.mu.Lock()
    defer h.state.mu.Unlock()

    events := make([]map[string]any, len(h.state.events))
    for i, event := range h.state.events {
        events[i] = cloneMap(event)
    }
    return events
}
```

Returning the internal slice directly would allow callers to race against concurrent `Handle` calls. Returning a clone decouples the snapshot from ongoing logger activity.

The shared `*memoryState` and its mutex matter for derived handlers specifically. If each `WithAttrs` result created its own lock, sibling loggers could still mutate the same event slice concurrently without synchronization.

## When to Use `Record.Clone`

`slog.Record` is passed to `Handle` by value, but its internal attribute storage may be shared with the original record. A simple assignment is insufficient when a handler intends to mutate the record or retain it beyond the `Handle` call:

```go
copy := record // internal attribute storage may still be shared
```

Use [`Record.Clone`](https://pkg.go.dev/log/slog#Record.Clone) before modifying or passing a record to an asynchronous consumer:

```go
record = record.Clone()
record.AddAttrs(
    slog.String("transport", "queue"),
)

h.records <- record
```

`MemoryHandler` does not retain the `slog.Record` itself — it transfers attributes synchronously into its own map — so calling `Clone` inside its `Handle` is unnecessary. Note that `Clone` does not perform deep copies of arbitrary objects passed via `slog.Any`.

## Concurrency

The `slog.Handler` contract permits any method on a handler to be called concurrently with any other method on the same instance. `MemoryHandler` satisfies this in two ways: instance configuration is immutable after construction, and shared event storage is protected by a mutex.

Map construction happens outside the lock because it depends only on local data and immutable handler state. The critical section covers only the append of the finished event into the shared slice. A handler writing to an `io.Writer` should follow the same pattern: assemble the log line into a local buffer, then write it under a lock in a single call.

Always verify concurrency safety with the race detector:

```bash
go test -race ./...
```

A passing test without `-race` does not detect unsynchronized access to shared memory.

## Verifying with `testing/slogtest`

The [`testing/slogtest`](https://pkg.go.dev/testing/slogtest) package validates the general `Handler` contract: built-in fields, `WithAttrs`, `WithGroup`, empty and nested groups, zero values, and `LogValuer` resolution.

`Events` already returns the representation that [`slogtest.TestHandler`](https://pkg.go.dev/testing/slogtest#TestHandler) expects, so the test is concise:

```go
func TestMemoryHandlerContract(t *testing.T) {
    handler := NewMemoryHandler(slog.LevelInfo)

    if err := slogtest.TestHandler(
        handler,
        handler.Events,
    ); err != nil {
        t.Fatal(err)
    }
}
```

`TestHandler` wires the handler into a `slog.Logger`, emits a battery of structured log calls, and checks that the results conform to the interface specification. For handlers that write JSON, the result function typically decodes each output line into `map[string]any`.

This check is oblivious to application-specific concerns. Custom level thresholds, sink error paths, source location output, and similar behaviors require separate tests. `slogtest` also does not replace running with `-race`.

`MemoryHandler.Handle` always returns `nil` because writing to an in-memory slice has no error channel. A production sink must return its underlying I/O or serialization error, but standard `slog.Logger` methods silently discard it. Observing sink failures and implementing fallback paths should be handled within the logging infrastructure — not by logging through the same broken handler.

---

A fully custom `slog.Handler` is the right choice when an application genuinely needs a new log sink or output format. For adding context fields, routing events, or reusing the existing JSON encoding, [wrapping an existing handler](/en/log-slog/advanced/wrappers-and-composition) is usually sufficient.
