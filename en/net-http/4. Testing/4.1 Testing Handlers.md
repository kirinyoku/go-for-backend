# Testing Handlers

Testing HTTP handlers is a form of unit testing: the logic of a specific function is tested in isolation, without actual network connections or starting a server. The primary tool for this in the Go standard library is the [`net/http/httptest`](https://pkg.go.dev/net/http/httptest) package.

::: info 
When you need to test the behavior of the entire HTTP stack — client execution, TLS connections, or redirect handling — use [`httptest.NewServer`](https://pkg.go.dev/net/http/httptest#NewServer). This approach is covered in [Testing with Servers](./testing-with-servers). 
:::

## Recording Responses with ResponseRecorder

The [`httptest.ResponseRecorder`](https://pkg.go.dev/net/http/httptest#ResponseRecorder) struct implements the [`http.ResponseWriter`](https://pkg.go.dev/net/http#ResponseWriter) interface. Instead of sending data over a network, it writes the response into an internal buffer that can be inspected later.

| Method / Field                                                             | Purpose                                                                                             |
| :------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------- |
| [`Code`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Code)       | The recorded HTTP status code. If the handler wrote nothing, this remains `0`.                      |
| [`Header()`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Header) | The response header map.                                                                            |
| [`Body`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Body)       | A buffer ([`*bytes.Buffer`](https://pkg.go.dev/bytes#Buffer)) containing the response body.         |
| [`Result()`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Result) | Returns an [`*http.Response`](https://pkg.go.dev/net/http#Response) to inspect cookies and trailers.|

::: info
The `Result()` method returns a response with an implicit `200 OK` status if the handler did not call `WriteHeader` or write a body. The `rr.Code` field will remain `0` in this case, so to check the "real" resulting HTTP status code, use `rr.Result().StatusCode`.
:::

## Test Structure: Arrange, Act, Assert

A standard handler test consists of three steps:

- **Arrange**: Set up the test by creating a request and a `ResponseRecorder`.
- **Act**: Execute the action by calling the handler directly.
- **Assert**: Verify the outcome by comparing the actual response with the expected one.

```go
// handlers_test.go
package main

import (
	"net/http"
	"net/http/httptest"
	"testing"
)

func TestHealthCheckHandler(t *testing.T) {
	// Arrange: create a mock request in memory and a response recorder.
	req := httptest.NewRequest(http.MethodGet, "/health", nil)
	rr := httptest.NewRecorder()

	// Act: call the handler directly — no network connection is needed.
	HealthCheckHandler(rr, req)

	// Assert: verify the status code and response body.
	if rr.Code != http.StatusOK {
		t.Errorf("status: got %d, want %d", rr.Code, http.StatusOK)
	}
	if rr.Body.String() != "OK" {
		t.Errorf("body: got %q, want %q", rr.Body.String(), "OK")
	}
}
```

### t.Errorf vs t.Fatalf

Two main error-reporting functions are used in handler tests:

- [`t.Errorf`](https://pkg.go.dev/testing#T.Errorf): Marks the test as failed but **continues** execution. It is used when assertions are independent of one another.
- [`t.Fatalf`](https://pkg.go.dev/testing#T.Fatalf): Marks the test as failed and **stops** execution immediately. It is used when a failed assertion makes further checks meaningless or would cause a panic.

**Typical example:** If [`Decode`](https://pkg.go.dev/encoding/json#Decoder.Decode) returns an error, accessing the fields of the decoded struct is pointless — here, `t.Fatalf` is required.

## Testing JSON Responses

Comparing JSON responses as plain strings is not recommended because the order of fields in JSON is not guaranteed. The idiomatic approach is to deserialize the response into a struct and compare the field values. To verify the `Content-Type`, use [`mime.ParseMediaType`](https://pkg.go.dev/mime#ParseMediaType) to correctly handle values like `application/json; charset=utf-8`.

```go
func TestUserHandler(t *testing.T) {
	// Arrange
	req := httptest.NewRequest(http.MethodGet, "/user", nil)
	rr := httptest.NewRecorder()

	// Act
	UserHandler(rr, req)

	// Assert: Check the status first using Fatalf — if there is an error,
	// decoding further data is meaningless.
	if rr.Code != http.StatusOK {
		t.Fatalf("status: got %d, want %d", rr.Code, http.StatusOK)
	}

	gotContentType, _, err := mime.ParseMediaType(rr.Header().Get("Content-Type"))
	if err != nil {
		t.Fatalf("content-type: parse: %v", err)
	}
	if gotContentType != "application/json" {
		t.Errorf("content-type: got %q, want %q", gotContentType, "application/json")
	}

	var got User
	if err := json.NewDecoder(rr.Body).Decode(&got); err != nil {
		t.Fatalf("failed to decode response: %v", err)
	}

	want := User{ID: 1, Username: "gopher"}
	if got != want {
		t.Errorf("response: got %+v, want %+v", got, want)
	}
}
```

::: info 
For comparing complex structs (with slices, nested fields, or `nil` values), the [`github.com/google/go-cmp/cmp`](https://pkg.go.dev/github.com/google/go-cmp/cmp) package is very convenient. Instead of `t.Errorf("got %+v, want %+v", got, want)`, you can write `t.Errorf("mismatch (-want +got):\n%s", cmp.Diff(want, got))` — the output will only show the differences, rather than the entirety of both values.
:::

## Test Helpers

When writing many tests, it's common to extract repetitive assertions into separate helper functions. Two tools are used to ensure they work correctly:

[`t.Helper()`](https://pkg.go.dev/testing#T.Helper) tells the testing framework that the current function is a helper. When an error occurs, the stack trace will point to the calling test rather than the line inside the helper.

[`testing.TB`](https://pkg.go.dev/testing#TB) is an interface that combines [`*testing.T`](https://pkg.go.dev/testing#T) and [`*testing.B`](https://pkg.go.dev/testing#B). Helpers that accept `testing.TB` work seamlessly in both tests and benchmarks.

```go
// assertStatus checks the HTTP response status code.
// It accepts testing.TB instead of *testing.T so the helper is also suitable for benchmarks.
func assertStatus(tb testing.TB, rr *httptest.ResponseRecorder, want int) {
	tb.Helper() // On error, the trace will point to the calling test, not this line.
	if rr.Code != want {
		tb.Errorf("status: got %d, want %d", rr.Code, want)
	}
}

// assertBody checks the response body.
func assertBody(tb testing.TB, rr *httptest.ResponseRecorder, want string) {
	tb.Helper()
	if got := rr.Body.String(); got != want {
		tb.Errorf("body: got %q, want %q", got, want)
	}
}
```

Using helpers reduces the test body to a minimum:

```go
func TestHealthCheckHandler(t *testing.T) {
	req := httptest.NewRequest(http.MethodGet, "/health", nil)
	rr := httptest.NewRecorder()

	HealthCheckHandler(rr, req)

	assertStatus(t, rr, http.StatusOK)
	assertBody(t, rr, "OK")
}
```

## Table-Driven Tests

When a single handler requires testing multiple scenarios, you shouldn't duplicate test functions. An idiomatic pattern in Go is table-driven tests: all cases are defined in a single slice of structs, and the logic is executed in a loop using [`t.Run`](https://pkg.go.dev/testing#T.Run).

```go
func TestGreetHandler(t *testing.T) {
	tests := []struct {
		name       string
		url        string
		wantStatus int
		wantBody   string
	}{
		{
			name:       "known user",
			url:        "/greet?name=gopher",
			wantStatus: http.StatusOK,
			wantBody:   "Hello, gopher!",
		},
		{
			name:       "missing name",
			url:        "/greet",
			wantStatus: http.StatusBadRequest,
			wantBody:   "name is required\n",
		},
	}

	for _, tc := range tests {
		// t.Run creates an isolated subtest. A specific case can be run with:
		// go test -run TestGreetHandler/known_user
		t.Run(tc.name, func(t *testing.T) {
			req := httptest.NewRequest(http.MethodGet, tc.url, nil)
			rr := httptest.NewRecorder()

			GreetHandler(rr, req)

			assertStatus(t, rr, tc.wantStatus)
			assertBody(t, rr, tc.wantBody)
		})
	}
}
```

::: tip 
Subtests can be run concurrently by adding [`t.Parallel()`](https://pkg.go.dev/testing#T.Parallel) at the beginning of the `t.Run` body. This is safe as long as each subtest uses its own `ResponseRecorder` and request, and the handler itself (or its dependencies) doesn't mutate shared state without synchronization.
:::

## Testing Middleware

Middleware is tested in isolation: a minimal stub function is passed as the next handler, which merely records that control was transferred to it. Table-driven tests are particularly useful here to compactly cover all logic branches.

```go
func TestAuthMiddleware(t *testing.T) {
	// nextHandler is a stub confirming that the middleware passed control forward.
	// It is defined once and used across all subtests as it holds no state.
	nextHandler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
	})

	tests := []struct {
		name       string
		authHeader string
		wantStatus int
	}{
		{name: "missing token", authHeader: "", wantStatus: http.StatusUnauthorized},
		{name: "present token", authHeader: "Bearer secret", wantStatus: http.StatusOK},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			t.Parallel()

			req := httptest.NewRequest(http.MethodGet, "/", nil)
			if tc.authHeader != "" {
				req.Header.Set("Authorization", tc.authHeader)
			}
			rr := httptest.NewRecorder()

			AuthMiddleware(nextHandler).ServeHTTP(rr, req)

			assertStatus(t, rr, tc.wantStatus)
		})
	}
}
```

## Testing Handlers with Context

Using string literals as context keys leads to collisions: different packages might accidentally use the same strings. The standard Go idiom is to declare a private custom type for keys at the package level.

```go
// contextKey is a private type for context keys in this package.
// Values of type contextKey will not collide with string values,
// even if the underlying string content is identical.
type contextKey string

const userIDKey contextKey = "user_id"
```

A test for a handler that interacts with context:

```go
func TestContextHandler(t *testing.T) {
	handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// The specific type is asserted via type assertion — a missing key
		// or incorrect type yields ok == false, rather than causing a panic.
		id, ok := r.Context().Value(userIDKey).(int)
		if !ok {
			http.Error(w, "user_id not found in context", http.StatusInternalServerError)
			return
		}
		fmt.Fprintf(w, "%d", id)
	})

	req := httptest.NewRequest(http.MethodGet, "/", nil)
	// Inject the value into the request context using the typed key.
	ctx := context.WithValue(req.Context(), userIDKey, 42)
	req = req.WithContext(ctx)

	rr := httptest.NewRecorder()
	handler.ServeHTTP(rr, req)

	assertStatus(t, rr, http.StatusOK)
	assertBody(t, rr, "42")
}
```

## Inspecting Cookies

The `Header()` method on `ResponseRecorder` returns an `http.Header` map, where `Set-Cookie` is accessible as a standard header value. However, to get parsed cookies, it's much more convenient to use the `Result()` method, which returns a full `*http.Response`.

```go
func TestCookieHandler(t *testing.T) {
	handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		http.SetCookie(w, &http.Cookie{Name: "session", Value: "abc123"})
		w.WriteHeader(http.StatusOK)
	})

	req := httptest.NewRequest(http.MethodGet, "/", nil)
	rr := httptest.NewRecorder()
	handler.ServeHTTP(rr, req)

	assertStatus(t, rr, http.StatusOK)

	// Result() returns an *http.Response with parsed cookies.
	// The body is wrapped in an io.NopCloser — Close is a no-op,
	// but calling it is still idiomatic.
	resp := rr.Result()
	defer resp.Body.Close()

	cookies := resp.Cookies()
	if len(cookies) == 0 {
		t.Fatal("cookies: got none, want at least one")
	}
	if got, want := cookies[0].Name, "session"; got != want {
		t.Errorf("cookie name: got %q, want %q", got, want)
	}
}
```
