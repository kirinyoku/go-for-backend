# Mocking External APIs

When application code communicates with an external HTTP API, tests should not depend on the real external server. Instead, a local test server is started, acting as the external API and returning predetermined responses.

::: info
In this article, "mocking an external API" does not refer to substituting a stub function, but rather starting a controlled HTTP server in the test. This approach verifies the actual HTTP request construction: the method, path, query parameters, headers, request body, status handling, and response decoding.
:::

## Why Real APIs Are Unsuitable for Tests

Calling a real external API makes tests flaky and expensive to maintain:

- The network might be unavailable or slow.
- The external service could change its data, rate limits, or response format.
- Tests become dependent on API keys and secrets.
- Negative scenarios are difficult to reproduce deterministically.
- Running tests could consume quotas or create real, irreversible operations.

A client code test shouldn't verify whether the external service is working correctly, but rather that the application interacts with the expected HTTP contract correctly.

::: warning
Real API keys must not be used in unit tests. If a test needs to verify an authorization header, a dummy value like `test-api-key` or `secret-key` is sufficient.
:::

## The Interception Point: Base URL and HTTP Client

Code is difficult to test if the external API's URL is hardcoded directly inside the method:

```go
func CurrentWeather(city string) error {
	resp, err := http.Get("https://api.weather.example/v1/weather/current?city=" + city)
	if err != nil {
		return err
	}
	defer resp.Body.Close()

	return nil
}
```

This code has several immediate problems:

- The base URL cannot be replaced with a test server address.
- It uses [`http.DefaultClient`](https://pkg.go.dev/net/http#DefaultClient), which has no default timeout.
- It's impossible to verify the request without making a real network call.
- It is much harder to simulate network errors or context cancellations.

The idiomatic solution is to store the base URL and the HTTP client in a dependency struct. In production code, the real URL is passed in, while in tests, you pass [`ts.URL`](https://pkg.go.dev/net/http/httptest#Server.URL) from `httptest.NewServer`.

```go
// weather.go
package weather

import (
	"context"
	"encoding/json"
	"fmt"
	"net/http"
	"net/url"
	"path"
)

type Client struct {
	BaseURL    string
	APIKey     string
	HTTPClient *http.Client
}

type CurrentWeather struct {
	City  string  `json:"city"`
	TempC float64 `json:"temp_c"`
}

func (c *Client) CurrentWeather(ctx context.Context, city string) (CurrentWeather, error) {
	endpoint, err := url.Parse(c.BaseURL)
	if err != nil {
		return CurrentWeather{}, fmt.Errorf("parse base URL: %w", err)
	}

	// path.Join preserves a possible base URL prefix, like /v1.
	// If the API distinguishes between /path and /path/, note that path.Join
	// cleans the path and removes trailing slashes.
	endpoint.Path = path.Join(endpoint.Path, "/weather/current")

	query := endpoint.Query()
	query.Set("city", city)
	endpoint.RawQuery = query.Encode()

	req, err := http.NewRequestWithContext(ctx, http.MethodGet, endpoint.String(), nil)
	if err != nil {
		return CurrentWeather{}, fmt.Errorf("create weather request: %w", err)
	}

	req.Header.Set("Accept", "application/json")
	if c.APIKey != "" {
		req.Header.Set("X-API-Key", c.APIKey)
	}

	httpClient := c.HTTPClient
	if httpClient == nil {
		httpClient = http.DefaultClient
	}

	resp, err := httpClient.Do(req)
	if err != nil {
		return CurrentWeather{}, fmt.Errorf("send weather request: %w", err)
	}
	defer resp.Body.Close()

	if resp.StatusCode != http.StatusOK {
		return CurrentWeather{}, fmt.Errorf("weather API status: %s", resp.Status)
	}

	var weather CurrentWeather
	if err := json.NewDecoder(resp.Body).Decode(&weather); err != nil {
		return CurrentWeather{}, fmt.Errorf("decode weather response: %w", err)
	}

	return weather, nil
}
```

This implementation provides two critical points of control:

- `BaseURL` allows redirecting requests to a test server.
- `HTTPClient` allows setting timeouts, TLS configurations, or swapping out the transport in tests.

## Basic Mocking with httptest.NewServer

`httptest.NewServer` starts a local HTTP server on a random available port. Its address is exposed via the `ts.URL` field. This is the address you provide to the client being tested instead of the real external API.

```go
// weather_test.go
package weather

import (
	"context"
	"fmt"
	"net/http"
	"net/http/httptest"
	"testing"
)

func newWeatherAPIServer(t testing.TB, handler http.Handler) *httptest.Server {
	t.Helper()

	ts := httptest.NewServer(handler)
	t.Cleanup(ts.Close)

	return ts
}

func TestClient_CurrentWeather(t *testing.T) {
	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if r.Method != http.MethodGet {
			http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
			return
		}
		if r.URL.Path != "/weather/current" {
			http.NotFound(w, r)
			return
		}
		if got := r.URL.Query().Get("city"); got != "London" {
			http.Error(w, "city is required", http.StatusBadRequest)
			return
		}
		if got := r.Header.Get("X-API-Key"); got != "secret-key" {
			http.Error(w, "invalid API key", http.StatusUnauthorized)
			return
		}

		w.Header().Set("Content-Type", "application/json")
		fmt.Fprintln(w, `{"city":"London","temp_c":15.5}`)
	}))

	client := &Client{
		BaseURL:    ts.URL,
		APIKey:     "secret-key",
		HTTPClient: ts.Client(),
	}

	got, err := client.CurrentWeather(context.Background(), "London")
	if err != nil {
		t.Fatalf("current weather: %v", err)
	}

	want := CurrentWeather{City: "London", TempC: 15.5}
	if got != want {
		t.Errorf("weather: got %+v, want %+v", got, want)
	}
}
```

The mock server serves two purposes here:

- It asserts that the client sent the correct HTTP request.
- It returns a controlled JSON response to verify the result processing.

This type of test is closer to real behavior than replacing a single function, because the request traverses the standard HTTP client, URL serialization, headers, and response decoding.

::: warning
`httptest.Server` handles requests in separate goroutines. You should not call [`t.Fatal`](https://pkg.go.dev/testing#T.Fatal) or [`t.Fatalf`](https://pkg.go.dev/testing#T.Fatalf) from within the handler: `FailNow` must be called from the test's own goroutine, not from goroutines spawned during the test. For contract errors, it is better to return an HTTP error using [`http.Error`](https://pkg.go.dev/net/http#Error) and assert the resulting client error in the main test body.
:::

## Verifying Outgoing Requests

An external API mock shouldn't just verify the end result; it must also inspect the outgoing request itself. Typically, the following elements are important:

| Request Part | What to Verify |
| :--- | :--- |
| HTTP Method | `GET`, `POST`, `PUT`, `DELETE` |
| Path | `/weather/current`, `/v1/users`, `/bot/sendMessage` |
| Query Parameters | `city=London`, `limit=10`, `page=2` |
| Headers | `Accept`, `Content-Type`, `Authorization`, API keys |
| Request Body | JSON, form-data, external API payloads |
| Request Count | Retries, token refreshes, pagination |

For `GET` requests, verifying the method, path, query parameters, and headers is usually sufficient. For `POST` requests, the request body should also be decoded. The `Content-Type` header is best parsed via [`mime.ParseMediaType`](https://pkg.go.dev/mime#ParseMediaType) to correctly handle parameters like `application/json; charset=utf-8`.

```go
func TestSendAlert_SendsJSONBody(t *testing.T) {
	type createAlertRequest struct {
		City    string `json:"city"`
		Message string `json:"message"`
	}

	sendAlert := func(ctx context.Context, client *http.Client, endpoint string, alert createAlertRequest) error {
		var body bytes.Buffer
		if err := json.NewEncoder(&body).Encode(alert); err != nil {
			return fmt.Errorf("encode alert request: %w", err)
		}

		req, err := http.NewRequestWithContext(ctx, http.MethodPost, endpoint+"/alerts", &body)
		if err != nil {
			return fmt.Errorf("create alert request: %w", err)
		}
		req.Header.Set("Content-Type", "application/json")

		resp, err := client.Do(req)
		if err != nil {
			return fmt.Errorf("send alert request: %w", err)
		}
		defer resp.Body.Close()

		if resp.StatusCode != http.StatusCreated {
			return fmt.Errorf("alert API status: %s", resp.Status)
		}

		return nil
	}

	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if r.Method != http.MethodPost {
			http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
			return
		}
		if r.URL.Path != "/alerts" {
			http.NotFound(w, r)
			return
		}
		contentType, _, err := mime.ParseMediaType(r.Header.Get("Content-Type"))
		if err != nil {
			http.Error(w, "invalid content type", http.StatusUnsupportedMediaType)
			return
		}
		if contentType != "application/json" {
			http.Error(w, "content type must be application/json", http.StatusUnsupportedMediaType)
			return
		}

		var got createAlertRequest
		if err := json.NewDecoder(r.Body).Decode(&got); err != nil {
			http.Error(w, "invalid JSON body", http.StatusBadRequest)
			return
		}

		want := createAlertRequest{City: "London", Message: "Rain expected"}
		if got != want {
			http.Error(w, "unexpected request body", http.StatusBadRequest)
			return
		}

		w.WriteHeader(http.StatusCreated)
	}))

	alert := createAlertRequest{City: "London", Message: "Rain expected"}
	if err := sendAlert(context.Background(), ts.Client(), ts.URL, alert); err != nil {
		t.Fatalf("send alert: %v", err)
	}
}
```

This example illustrates the body verification pattern: the mock handler decodes the incoming JSON and returns an error status if the request does not meet the expected contract. After the client function executes, the main test asserts that no error was returned.

::: info
When asserting outgoing JSON request bodies, it is much more robust to decode the JSON into a struct rather than comparing raw strings. The order of fields in JSON shouldn't cause the test to fail.
:::

## Table-Driven Tests for External API Responses

A single client method usually needs to handle multiple response scenarios correctly:

- A successful `200 OK` response.
- Error statuses like `400 Bad Request`, `401 Unauthorized`, `500 Internal Server Error`.
- An empty response body.
- Malformed JSON.
- Unexpected data formats.

The same table-driven testing pattern used for handler testing applies perfectly here.

```go
func TestClient_CurrentWeather_UpstreamErrors(t *testing.T) {
	tests := []struct {
		name       string
		statusCode int
		body       string
		wantErr    string
	}{
		{
			name:       "server error",
			statusCode: http.StatusInternalServerError,
			body:       `{"error":"temporary failure"}`,
			wantErr:    "weather API status",
		},
		{
			name:       "invalid JSON",
			statusCode: http.StatusOK,
			body:       `{"city":"London","temp_c":`,
			wantErr:    "decode weather response",
		},
		{
			name:       "empty body",
			statusCode: http.StatusOK,
			body:       ``,
			wantErr:    "decode weather response",
		},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
				w.Header().Set("Content-Type", "application/json")
				w.WriteHeader(tc.statusCode)
				fmt.Fprint(w, tc.body)
			}))

			client := &Client{
				BaseURL:    ts.URL,
				HTTPClient: ts.Client(),
			}

			_, err := client.CurrentWeather(context.Background(), "London")
			if err == nil {
				t.Fatal("error: got nil, want non-nil")
			}
			if !strings.Contains(err.Error(), tc.wantErr) {
				t.Errorf("error: got %q, want substring %q", err.Error(), tc.wantErr)
			}
		})
	}
}
```

Here, each subtest spins up its own mock server. This keeps scenarios independent: altering the response for one case will not affect the others.

## Simulating Timeouts and Context Cancellation

External APIs may respond slowly or fail to respond entirely. Testing these scenarios is particularly important for code utilizing `context.Context`.

```go
func TestClient_CurrentWeather_ContextTimeout(t *testing.T) {
	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// A delay longer than the context timeout ensures the client cancels the request.
		time.Sleep(100 * time.Millisecond)
		w.WriteHeader(http.StatusOK)
	}))

	client := &Client{
		BaseURL:    ts.URL,
		HTTPClient: ts.Client(),
	}

	ctx, cancel := context.WithTimeout(context.Background(), 10*time.Millisecond)
	defer cancel()

	_, err := client.CurrentWeather(ctx, "London")
	if err == nil {
		t.Fatal("error: got nil, want timeout error")
	}
	if !errors.Is(err, context.DeadlineExceeded) {
		t.Errorf("error: got %v, want context deadline exceeded", err)
	}
}
```

This test guarantees that the method correctly passes the context to [`http.NewRequestWithContext`](https://pkg.go.dev/net/http#NewRequestWithContext). If [`http.Get`](https://pkg.go.dev/net/http#Get) were used instead, the test context wouldn't be tied to the request, as `http.Get` does not accept a `ctx`.

::: tip
When testing timeouts, the server delay should significantly exceed the context timeout. Values that are too close make the test sensitive to machine load and can cause flaky failures.
:::

## Stateful Mocks for Sequential Responses

Some scenarios rely on the order of requests. For example, an external API might initially return a `500 Internal Server Error`, but succeed with `200 OK` upon a retry. To test such logic, the mock server can maintain state between requests.

```go
func TestFetchWithRetry(t *testing.T) {
	var attempts atomic.Int32

	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if attempts.Add(1) == 1 {
			http.Error(w, "temporary failure", http.StatusInternalServerError)
			return
		}

		w.WriteHeader(http.StatusOK)
	}))

	err := fetchWithRetry(context.Background(), ts.Client(), ts.URL, 2)
	if err != nil {
		t.Fatalf("fetch with retry: %v", err)
	}

	if got, want := attempts.Load(), int32(2); got != want {
		t.Errorf("attempts: got %d, want %d", got, want)
	}
}

func fetchWithRetry(ctx context.Context, client *http.Client, endpoint string, maxAttempts int) error {
	if maxAttempts < 1 {
		return errors.New("max attempts must be positive")
	}

	var lastErr error

	for attempt := 0; attempt < maxAttempts; attempt++ {
		req, err := http.NewRequestWithContext(ctx, http.MethodGet, endpoint, nil)
		if err != nil {
			return fmt.Errorf("create request: %w", err)
		}

		resp, err := client.Do(req)
		if err != nil {
			lastErr = err
			continue
		}

		_, copyErr := io.Copy(io.Discard, resp.Body)
		closeErr := resp.Body.Close()
		if copyErr != nil {
			return fmt.Errorf("read response body: %w", copyErr)
		}
		if closeErr != nil {
			return fmt.Errorf("close response body: %w", closeErr)
		}

		if resp.StatusCode == http.StatusOK {
			return nil
		}

		lastErr = fmt.Errorf("status: %s", resp.Status)
	}

	return fmt.Errorf("all attempts failed: %w", lastErr)
}
```

An [`atomic.Int32`](https://pkg.go.dev/sync/atomic#Int32) is used for the counter because the HTTP server can process requests concurrently in different goroutines. While a standard variable might suffice if the test is strictly sequential, an atomic counter makes the implementation robust and thread-safe.

## Intercepting the Transport Without a Server

`httptest.NewServer` is the primary tool for mocking external HTTP APIs. However, you might occasionally need to simulate errors that occur *before* an HTTP response is received, such as connection failures, DNS errors, or transport layer issues. In these cases, you can inject a custom [`http.RoundTripper`](https://pkg.go.dev/net/http#RoundTripper), which [`http.Client`](https://pkg.go.dev/net/http#Client) uses internally to execute a single HTTP request.

```go
type roundTripFunc func(req *http.Request) (*http.Response, error)

func (f roundTripFunc) RoundTrip(req *http.Request) (*http.Response, error) {
	return f(req)
}

func TestClient_CurrentWeather_NetworkError(t *testing.T) {
	httpClient := &http.Client{
		Transport: roundTripFunc(func(req *http.Request) (*http.Response, error) {
			return nil, errors.New("connection refused")
		}),
	}

	client := &Client{
		BaseURL:    "https://weather.example.test",
		HTTPClient: httpClient,
	}

	_, err := client.CurrentWeather(context.Background(), "London")
	if err == nil {
		t.Fatal("error: got nil, want non-nil")
	}
	if !strings.Contains(err.Error(), "send weather request") {
		t.Errorf("error: got %q, want transport error", err.Error())
	}
}
```

This test opens no network connections at all. The client forms the request as usual, but instead of routing it through a real transport, it invokes the `roundTripFunc`.

If the mock transport returns a successful [`*http.Response`](https://pkg.go.dev/net/http#Response), it is recommended to explicitly set the [`Body`](https://pkg.go.dev/net/http#Response.Body) field. While `http.Client` may insert an empty body if the body is `nil` and the response length is zero, an explicit `io.NopCloser` provides a more accurate simulation of a real response and is strictly required if the body is not empty.

```go
httpClient := &http.Client{
	Transport: roundTripFunc(func(req *http.Request) (*http.Response, error) {
		return &http.Response{
			StatusCode: http.StatusOK,
			Status:     "200 OK",
			Header:     make(http.Header),
			Body:       io.NopCloser(strings.NewReader(`{"city":"London","temp_c":15.5}`)),
			Request:    req,
		}, nil
	}),
}
```

::: info
Replacing `RoundTripper` is great for testing low-level client errors. However, to comprehensively verify an external API's HTTP contract — paths, headers, query parameters, and request payloads — `httptest.NewServer` remains the superior choice.
:::

## The Doer Interface

Another strategy is to accept a smaller interface outlining the `Do` method, rather than depending directly on `*http.Client`. This design is beneficial when you need to completely decouple your code from a specific HTTP client implementation.

```go
type Doer interface {
	Do(req *http.Request) (*http.Response, error)
}

type Client struct {
	BaseURL    string
	APIKey     string
	HTTPClient Doer
}
```

The interface should be declared on the consumer side, not by the external dependency. This respects the principle of minimal contracts: the client code only needs the `Do` method, so the interface shouldn't dictate anything more.

In your test, you can supply a simple stub struct:

```go
type mockDoer struct {
	doFunc func(req *http.Request) (*http.Response, error)
}

func (m mockDoer) Do(req *http.Request) (*http.Response, error) {
	return m.doFunc(req)
}
```

Yet, for HTTP clients, this approach isn't always superior to `httptest.NewServer`. A `Doer` interface asserts the interaction at the `Do` method call boundary, but it bypasses the real HTTP request network pipeline. Thus, for testing external APIs, `httptest.NewServer` is generally preferred, reserving the interface or mock transport for targeted, specific use cases.

## How to Choose an Approach

| Approach | When to Use |
| :--- | :--- |
| `httptest.NewServer` | The primary choice for testing clients of external HTTP APIs. |
| `httptest.NewTLSServer` | Verifying HTTPS, TLS connections, and clients trusting test certificates. |
| Replacing `RoundTripper` | Simulating low-level network errors without launching a server. |
| `Doer` Interface | Fully substituting the request execution mechanism via a minimal consumer API. |
| Real External API | Suitable only for smoke tests or isolated integration tests, not routine unit tests. |

**The Golden Rule:** A test must control all inputs and responses of an external dependency. If a test's outcome relies on the state of a live server, the network, or an API key, it cannot be considered deterministic.
