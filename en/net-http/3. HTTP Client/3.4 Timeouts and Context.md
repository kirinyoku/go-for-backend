# Timeouts and Context

Controlling request duration in Go is built from several independent mechanisms. They look similar from the outside because the request ends with an error, but they operate at different levels:

- [`context.Context`](https://pkg.go.dev/context#Context) sets the deadline for a specific operation and ties the HTTP request to the lifecycle of the calling code.
- [`http.Client.Timeout`](https://pkg.go.dev/net/http#Client.Timeout) sets an overall limit for one HTTP request, including reading the response body.
- [`http.Transport`](https://pkg.go.dev/net/http#Transport) timeouts limit individual network phases: connecting, the TLS handshake and waiting for response headers.

These mechanisms can be used together. The request fails when the first limit fires.

## Request Timeout with Context

A context is useful when the time limit belongs to a specific call, not to the entire HTTP client. For example, an outgoing request may be part of handling an incoming HTTP request, a background job, an RPC call or a chain of calls to several services.

```go
// fetchStatus performs a GET request and returns the response status code.
// The function accepts a parent context so the caller can cancel the operation.
func fetchStatus(ctx context.Context, client *http.Client, url string) (int, error) {
    ctx, cancel := context.WithTimeout(ctx, 2*time.Second)
    defer cancel()

    req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
    if err != nil {
        return 0, err
    }

    resp, err := client.Do(req)
    if err != nil {
        return 0, err
    }
    defer resp.Body.Close()

    _, _ = io.Copy(io.Discard, resp.Body)

    return resp.StatusCode, nil
}
```

If the parent context is cancelled before the local `2*time.Second` timeout expires, the HTTP request is cancelled too. This is especially important on the server side: an outgoing request should not keep running indefinitely after the incoming client has disconnected.

## Overall Client Timeout

The `http.Client.Timeout` field sets an overall limit for one [`client.Do`](https://pkg.go.dev/net/http#Client.Do) call and the subsequent response body read. This time includes:

1. Opening the TCP connection.
2. The TLS handshake.
3. Sending the request.
4. Waiting for and reading the response headers.
5. Reading the entire response body.

```go
client := &http.Client{
    Timeout: 10 * time.Second,
}
```

If the operation has not completed after 10 seconds, the connection is closed. The error may appear not only from `client.Do`, but also later while reading `resp.Body`, because `Client.Timeout` covers the response body.

## http.Transport Timeouts

[`http.Transport`](/en/net-http/client/transport-layer-configuration) controls the low-level execution of a request: connection creation, TLS, keep-alive and response waiting. Its timeouts are useful when you need to limit a specific phase instead of the whole request.

| Setting | What it limits |
| :------ | :------------- |
| [`DialContext`](https://pkg.go.dev/net/http#Transport.DialContext) | Time allowed to establish the network connection. DNS lookup may be part of this phase depending on the resolver and environment. |
| [`TLSHandshakeTimeout`](https://pkg.go.dev/net/http#Transport.TLSHandshakeTimeout) | Time allowed for the TLS handshake after connecting. |
| [`ResponseHeaderTimeout`](https://pkg.go.dev/net/http#Transport.ResponseHeaderTimeout) | Time spent waiting for response headers after the request has been sent. |
| [`ExpectContinueTimeout`](https://pkg.go.dev/net/http#Transport.ExpectContinueTimeout) | Time spent waiting for a `100 Continue` response when the request uses `Expect: 100-continue`. |
| [`IdleConnTimeout`](https://pkg.go.dev/net/http#Transport.IdleConnTimeout) | How long a keep-alive connection can stay idle in the pool. This is not a timeout for one request. |

```go
transport := &http.Transport{
    DialContext: (&net.Dialer{
        Timeout:   3 * time.Second,
        KeepAlive: 30 * time.Second,
    }).DialContext,
    TLSHandshakeTimeout:   3 * time.Second,
    ResponseHeaderTimeout: 5 * time.Second,
    ExpectContinueTimeout: 1 * time.Second,
    IdleConnTimeout:       90 * time.Second,
}

client := &http.Client{
    Transport: transport,
    Timeout:   15 * time.Second,
}
```

::: info
In this example, `DialContext` is configured through [`net.Dialer`](https://pkg.go.dev/net#Dialer): its `Timeout` limits connection setup, and its `DialContext` method is passed to `http.Transport` as the function that opens connections.
:::

This client limits individual request phases and also has an overall `15*time.Second` limit. If `ResponseHeaderTimeout` expires before the overall `Client.Timeout`, the error is returned while waiting for headers. If the server sends headers quickly but then streams the body slowly, the body read is still governed by `Client.Timeout` or the context deadline.

## Request Cancellation

Context lets you cancel a request before any timeout expires. This happens, for example, when the user closes the connection to your server, the parent task ends or the caller explicitly calls `cancel`.

```go
func process(ctx context.Context, client *http.Client, url string) error {
    req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
    if err != nil {
        return err
    }

    resp, err := client.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()

    _, err = io.Copy(io.Discard, resp.Body)
    return err
}
```

When `ctx` is cancelled, `client.Do` or a read from `resp.Body` returns an error. To diagnose the cause, inspect both the error and the context state.

## Identifying the Cause

Errors from [`net/http`](https://pkg.go.dev/net/http) are often wrapped, for example in [`*url.Error`](https://pkg.go.dev/net/url#Error). Check them with [`errors.Is`](https://pkg.go.dev/errors#Is), [`os.IsTimeout`](https://pkg.go.dev/os#IsTimeout) or the [`net.Error`](https://pkg.go.dev/net#Error) interface instead of comparing strings.

```go
resp, err := client.Do(req)
if err != nil {
    switch {
    case errors.Is(err, context.Canceled):
        return fmt.Errorf("request canceled: %w", err)

    case errors.Is(req.Context().Err(), context.DeadlineExceeded):
        return fmt.Errorf("request context deadline exceeded: %w", err)

    case os.IsTimeout(err):
        return fmt.Errorf("client or transport timeout: %w", err)
    }

    var netErr net.Error
    if errors.As(err, &netErr) && netErr.Timeout() {
        return fmt.Errorf("network timeout: %w", err)
    }

    return fmt.Errorf("network error: %w", err)
}
defer resp.Body.Close()
```

For response body reads, perform the check separately: the error can happen after `client.Do` has already succeeded.

```go
body, err := io.ReadAll(resp.Body)
if err != nil {
    return fmt.Errorf("read response body: %w", err)
}
_ = body
```

::: tip
`http.Client.Timeout` and context deadlines can appear as timeout errors at different stages. For logging, it is useful to record not only that a timeout happened, but also the configured values: operation deadline, `Client.Timeout`, `Dialer.Timeout`, `TLSHandshakeTimeout` and `ResponseHeaderTimeout`.
:::
