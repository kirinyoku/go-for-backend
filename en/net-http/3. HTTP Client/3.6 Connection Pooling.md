# Connection Pooling

To ensure high performance, Go uses the **Keep-Alive** mechanism, which allows reusing the same TCP connection for multiple sequential HTTP requests to the same host. This mechanism is managed via the connection pool in [`http.Transport`](https://pkg.go.dev/net/http#Transport).

## How Keep-Alive Works

When a request is completed and the response body is fully read and closed, the connection is not dropped. Instead, it is placed into an **idle connection pool**. For the next request to the same host, the client first checks the pool. If there is an available idle connection, it is reused. This eliminates the overhead of the [TCP Three-way Handshake](https://en.wikipedia.org/wiki/Handshake_(computing)#TCP_three-way_handshake) and [TLS Handshake](https://en.wikipedia.org/wiki/Handshake_(computing)#TLS_handshake) (which is the most expensive part).

## Pool Configuration

The default values in [`http.DefaultTransport`](https://pkg.go.dev/net/http#DefaultTransport) are suitable for most scenarios, but they can become a bottleneck under high load to specific hosts.

| Parameter                                                                    | Default        | Description                                                                     |
| :--------------------------------------------------------------------------- | :------------- | :------------------------------------------------------------------------------ |
| [`MaxIdleConns`](https://pkg.go.dev/net/http#Transport.MaxIdleConns)         | 100            | The total number of idle connections across all hosts.                          |
| [`MaxIdleConnsPerHost`](https://pkg.go.dev/net/http#Transport.MaxIdleConnsPerHost) | 2              | The number of idle connections for a **single** specific host.                  |
| [`MaxConnsPerHost`](https://pkg.go.dev/net/http#Transport.MaxConnsPerHost)          | 0 (no limit)   | The hard limit on **all** connections (active + idle) to a single host.         |
| [`IdleConnTimeout`](https://pkg.go.dev/net/http#Transport.IdleConnTimeout)   | 90s            | The amount of time an idle connection will be kept open before closing.         |

```go
transport := http.DefaultTransport.(*http.Transport).Clone()

// The default value of 2 is critically low for high-load systems.
// If an application makes 100 requests per second to a single API,
// with a limit of 2, most connections will be closed immediately after use.
transport.MaxIdleConnsPerHost = 100
transport.MaxIdleConns = 100

// Optional: client-side throttling to protect the target server from overload.
// transport.MaxConnsPerHost = 200
```

## When a Connection Isn't Returned to the Pool

The detailed process of closing and reading the response body is covered in the [Reading and Handling Responses](./response-handling) article. In the context of connection pooling, the key takeaway is that a connection cannot be reused if the response wasn't properly completed.

Common reasons include:

1. The response body is not closed (`resp.Body.Close()`).
2. The response body is not fully read (up to [`io.EOF`](https://pkg.go.dev/io#EOF)).
3. The server sent a `Connection: close` header.
4. A network-level error occurred while reading the data.

## Monitoring Connections

Excessive connection creation (indicating the pool isn't working) can be spotted by the following signs:

- An increase in the number of open file descriptors.
- A high number of connections in the `TIME_WAIT` state when running `netstat`.
- Increased latency due to continuous TLS handshakes.

```bash
# Checking the number of connections in TIME_WAIT (Linux)
netstat -ant | grep TIME_WAIT | wc -l
```

## Usage in Microservices

In a microservices architecture, where one service frequently communicates with a few others, it is crucial to increase `MaxIdleConnsPerHost`. This helps maintain "hot" communication channels and minimize latency.

::: warning
If you use a custom `http.Transport`, ensure that you create it **once** and reuse it across your `http.Client` instances. Creating a new transport for every request completely disables connection pooling.
:::
