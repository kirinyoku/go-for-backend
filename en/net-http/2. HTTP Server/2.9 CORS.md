# CORS as Middleware

**CORS (Cross-Origin Resource Sharing)** is a browser mechanism that controls whether JavaScript code can read responses from another site.

A typical setup looks like this: the frontend is served from `https://app.example.com`, while the API runs at `https://api.example.com`. For that request to work in a browser, the server has to explicitly allow access with CORS headers. In `net/http`, this logic fits naturally as [middleware](/en/net-http/server/middleware) wrapped around an [`http.Handler`](https://pkg.go.dev/net/http#Handler).

## How It Works

CORS compares origins. An origin includes the scheme, host and port, so `https://app.example.com`, `https://api.example.com` and `http://app.example.com` are all different origins.

When browser-side code sends a cross-origin request, the browser adds an `Origin` header:

```http
GET /api/profile HTTP/1.1
Origin: https://app.example.com
```

If the server allows that origin, it echoes it back in `Access-Control-Allow-Origin`:

```http
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://app.example.com
```

After that, the browser allows JavaScript code to read the response. If the header is missing or the origin does not match, the request may still have reached the server, but the browser hides the response from frontend code.

::: warning
CORS is not authorization, and it does not protect an endpoint from direct HTTP requests. It is a contract between the server and the browser: the server sends CORS headers, and the browser enforces the restriction. `curl`, server-side clients and tests that use `net/http` do not apply the CORS policy.
:::

For some requests, the browser first sends a **preflight request**: an `OPTIONS` request that checks whether the server will allow the actual request with a specific method and headers.

```http
OPTIONS /api/profile HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization
```

Preflight requests commonly appear for methods such as `PUT`, `PATCH` and `DELETE`, for the `Authorization` header, or for `Content-Type: application/json`. The server responds with CORS headers and does not call the business handler:

```http
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 7200
```

`Access-Control-Max-Age` lets the browser cache a successful preflight response, so it does not have to send `OPTIONS` before every similar request. Very large values can be inconvenient when the CORS policy changes: users' browsers may keep using the old permission for some time.

::: tip
A preflight request usually does not include the user's `Authorization` header and should not mutate application state. This is why CORS middleware is often placed before auth middleware, so `OPTIONS` is not rejected by authorization before the CORS headers are set.
:::

## Implementation

The standard library does not provide a ready-made CORS handler. From the `net/http` side, the server needs to inspect the `Origin` header, return the appropriate `Access-Control-*` headers and handle preflight requests separately. The most convenient shape is middleware around an [`http.Handler`](https://pkg.go.dev/net/http#Handler).

CORS is usually configured with an allowlist: the set of origins that are allowed to read API responses. This list is best supplied through application configuration, so the values can change between development environments without code changes.

```go
type CORSConfig struct {
    AllowedOrigins  map[string]struct{}
    AllowedMethods  string
    AllowedHeaders  string
    MaxAge          string
    AllowCredentials bool
}

func CORS(cfg CORSConfig) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            origin := r.Header.Get("Origin")
            if origin == "" {
                next.ServeHTTP(w, r)
                return
            }

            if _, ok := cfg.AllowedOrigins[origin]; !ok {
                http.Error(w, "CORS origin is not allowed", http.StatusForbidden)
                return
            }

            h := w.Header()
            h.Set("Access-Control-Allow-Origin", origin)
            h.Add("Vary", "Origin")

            if cfg.AllowCredentials {
                h.Set("Access-Control-Allow-Credentials", "true")
            }

            if r.Method == http.MethodOptions &&
                r.Header.Get("Access-Control-Request-Method") != "" {
                h.Set("Access-Control-Allow-Methods", cfg.AllowedMethods)
                h.Set("Access-Control-Allow-Headers", cfg.AllowedHeaders)
                h.Set("Access-Control-Max-Age", cfg.MaxAge)
                h.Add("Vary", "Access-Control-Request-Method")
                h.Add("Vary", "Access-Control-Request-Headers")
                w.WriteHeader(http.StatusNoContent)
                return
            }

            next.ServeHTTP(w, r)
        })
    }
}
```

The example returns a specific allowed origin, not `*`. This matters for APIs that use cookies or `fetch(..., { credentials: "include" })`: when `Access-Control-Allow-Credentials: true` is set, browsers do not accept `Access-Control-Allow-Origin: *`.

`Vary: Origin` is required because the value of `Access-Control-Allow-Origin` is chosen dynamically. Without `Vary`, an intermediate cache could store a response for one origin and serve it to another origin with the wrong CORS headers. Preflight responses add `Vary: Access-Control-Request-Method` and `Vary: Access-Control-Request-Headers` for the same reason.

::: info
If JavaScript code needs to read non-standard response headers, such as `X-Request-ID` or `X-Total-Count`, the server has to expose them explicitly with `Access-Control-Expose-Headers`.
:::
