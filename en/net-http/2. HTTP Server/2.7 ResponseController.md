# Streaming Responses

Most HTTP handlers in Go build a response in one pass: validate input, do the work, write the status and body. For that code, the basic [`http.ResponseWriter`](/en/net-http/intro/responsewriter) methods are enough: `Header`, `WriteHeader` and `Write`.

Things get more complex in other scenarios: a handler writes for a long time, sends data in chunks, works with a slow client or passes through middleware that wraps `ResponseWriter`. In those cases, it matters not only *what* gets written to the response, but also *when* data is sent to the client and which lower-level capabilities remain available after wrappers.

The standard library provides [`http.ResponseController`](https://pkg.go.dev/net/http#ResponseController) for these cases:

```go
rc := http.NewResponseController(w)
```

`ResponseController` does not replace `ResponseWriter`. It is created on top of the current writer and can call advanced operations when the concrete response implementation supports them: `Flush`, `SetReadDeadline`, `SetWriteDeadline`, `EnableFullDuplex` and `Hijack`.

Like `ResponseWriter` itself, the controller is valid only during the current handler call. Do not store it in a struct, pass it to background goroutines or use it after `ServeHTTP` returns.

## Streaming Writes

A normal HTTP response can be built completely and written at the end of the handler. Some responses, however, are useful when sent gradually: Server-Sent Events, progress updates for long operations, tail-like log output or simple streaming endpoints.

In such a handler, calling `Write` is not always enough. Data may remain buffered, and the client may see it later. [`Flush`](https://pkg.go.dev/net/http#ResponseController.Flush) asks the server to send the already written chunk to the client.

```go
func eventsHandler(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")

    rc := http.NewResponseController(w)
    ticker := time.NewTicker(time.Second)
    defer ticker.Stop()

    for i := 0; i < 3; i++ {
        select {
        case <-r.Context().Done():
            return
        case <-ticker.C:
        }

        if _, err := fmt.Fprintf(w, "data: event %d\n\n", i); err != nil {
            return
        }
        if err := rc.Flush(); err != nil {
            return
        }
    }
}
```

::: warning
`Flush` does not guarantee instant delivery all the way to the client application. There may be proxies, load balancers or client-side buffers between the server and the client. But without `Flush`, the server does not even try to send the current chunk before the handler finishes or the buffer fills up.
:::

## Per-handler Deadlines

Global server limits are configured through `http.Server` fields: `ReadTimeout`, `WriteTimeout`, `IdleTimeout` and `MaxHeaderBytes`. They are a good baseline protection layer, but sometimes one handler needs its own deadline.

For example, ordinary JSON endpoints may work with a short `WriteTimeout`, while a streaming endpoint needs to write for longer. In that case, the handler can manage its own write deadline:

```go
func downloadHandler(w http.ResponseWriter, r *http.Request) {
    rc := http.NewResponseController(w)
    if err := rc.SetWriteDeadline(time.Now().Add(30 * time.Second)); err != nil {
        http.Error(w, "write deadline is not supported", http.StatusInternalServerError)
        return
    }

    file, err := os.Open("backup.tar")
    if err != nil {
        http.Error(w, "file not found", http.StatusNotFound)
        return
    }
    defer file.Close()

    w.Header().Set("Content-Type", "application/octet-stream")

    if _, err := io.Copy(w, file); err != nil {
        // The client may have disconnected, or the write deadline may have expired.
        return
    }
}
```

[`SetWriteDeadline`](https://pkg.go.dev/net/http#ResponseController.SetWriteDeadline) sets the deadline for writing the response. After the deadline expires, writes should not block, but a write may still succeed if the data has already reached a buffer. That is why large and streaming responses should check write errors instead of ignoring them.

[`SetReadDeadline`](https://pkg.go.dev/net/http#ResponseController.SetReadDeadline) does the same for reading the request. It sets a deadline for reading the whole request, including the body:

```go
func uploadHandler(w http.ResponseWriter, r *http.Request) {
    rc := http.NewResponseController(w)
    if err := rc.SetReadDeadline(time.Now().Add(10 * time.Second)); err != nil {
        http.Error(w, "read deadline is not supported", http.StatusInternalServerError)
        return
    }

    r.Body = http.MaxBytesReader(w, r.Body, 10<<20)
    defer r.Body.Close()

    if _, err := io.Copy(io.Discard, r.Body); err != nil {
        http.Error(w, "failed to read request body", http.StatusBadRequest)
        return
    }

    w.WriteHeader(http.StatusNoContent)
}
```

A deadline limits time, not size. Request body size still needs to be limited with [`http.MaxBytesReader`](https://pkg.go.dev/net/http#MaxBytesReader).

## Support Depends on the Writer

`ResponseController` works through the capabilities of the concrete `ResponseWriter`. If the required operation is not supported, the method returns an error compatible with [`http.ErrNotSupported`](https://pkg.go.dev/net/http#ErrNotSupported).

```go
if err := rc.Flush(); err != nil {
    if errors.Is(err, http.ErrNotSupported) {
        http.Error(w, "streaming is not supported", http.StatusInternalServerError)
        return
    }

    http.Error(w, "failed to flush response", http.StatusInternalServerError)
    return
}
```

Check this error with [`errors.Is`](https://pkg.go.dev/errors#Is), because it may be wrapped.

## Middleware and Unwrap

Logging middleware often wraps `ResponseWriter` to record the response status and size. That wrapper is useful, but it can accidentally hide the advanced capabilities of the original writer.

To let `ResponseController` pass through the wrapper to the real writer, the wrapper should implement `Unwrap`:

```go
type responseWriter struct {
    http.ResponseWriter
    status int
    bytes  int
}

func (rw *responseWriter) Unwrap() http.ResponseWriter {
    return rw.ResponseWriter
}
```

Without `Unwrap`, code further down the chain may get `http.ErrNotSupported` even though the original writer supported `Flush`, deadline methods or other advanced operations. That is why `ResponseWriter` wrappers should be designed not only around `Write` and `WriteHeader`, but also around compatibility with the rest of the HTTP infrastructure.

## Full Duplex

Normally, a handler reads the request first and writes the response afterwards. For HTTP/1, the Go server may read the remaining request body before it starts writing the response. This is the right behavior for most handlers, but it gets in the way of scenarios where the handler needs to alternate between reading from `r.Body` and writing to `w`.

[`EnableFullDuplex`](https://pkg.go.dev/net/http#ResponseController.EnableFullDuplex) tells the server that the handler intentionally wants to read the request and write the response at the same time.

```go
func duplexHandler(w http.ResponseWriter, r *http.Request) {
    rc := http.NewResponseController(w)
    if err := rc.EnableFullDuplex(); err != nil {
        http.Error(w, "full duplex is not supported", http.StatusInternalServerError)
        return
    }

    w.Header().Set("Content-Type", "text/plain")

    scanner := bufio.NewScanner(r.Body)
    for scanner.Scan() {
        if _, err := fmt.Fprintf(w, "received: %s\n", scanner.Text()); err != nil {
            return
        }
        if err := rc.Flush(); err != nil {
            return
        }
    }
}
```

For HTTP/2, the Go server already allows concurrent request reads and response writes, but the handler should still be ready for an error: a middleware wrapper or another writer may stand between it and the real response implementation.

## Hijack

[`Hijack`](https://pkg.go.dev/net/http#ResponseController.Hijack) takes the network connection away from the HTTP server. After that, `net/http` no longer manages the response: the application writes HTTP lines itself or switches to a non-HTTP protocol over `net.Conn`.

This is a low-level mechanism that ordinary backend APIs rarely need. Common reasons include special protocols over TCP or older WebSocket upgrade code. For most streaming HTTP responses, `Write`, `Flush`, request context handling and correct deadlines are enough.

::: warning
Hijacked connections are not closed by `http.Server.Shutdown` as active HTTP requests. Their lifecycle must be handled separately, for example with `RegisterOnShutdown`. This is covered in [Graceful Shutdown](/en/net-http/server/graceful-shutdown).
:::
