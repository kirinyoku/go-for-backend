# Resource Limits

[`http.Server`](https://pkg.go.dev/net/http#Server) does not know what request sizes and delays are normal for a specific application. That means the server should be configured explicitly: limit request read time, response write time, idle keep-alive connections, header size and request body size.

Without these limits, a slow or oversized request can hold a connection, a goroutine, memory and a file descriptor for too long.

## http.Server Limits

The main limits are configured through fields on `http.Server`:

| Field | What it limits |
| :---- | :------------- |
| [`ReadTimeout`](https://pkg.go.dev/net/http#Server.ReadTimeout) | Maximum time for reading the full request, including headers and body. |
| [`WriteTimeout`](https://pkg.go.dev/net/http#Server.WriteTimeout) | Maximum time for writing the response. |
| [`IdleTimeout`](https://pkg.go.dev/net/http#Server.IdleTimeout) | Maximum time to wait for the next request on a keep-alive connection. |
| [`MaxHeaderBytes`](https://pkg.go.dev/net/http#Server.MaxHeaderBytes) | Maximum request header size. |

```go
server := &http.Server{
    Addr: ":8080",

    // ReadTimeout limits reading the whole request:
    // start-line, headers and body.
    ReadTimeout: 10 * time.Second,

    // WriteTimeout limits writing the response to the client.
    // Choose this value carefully for streaming endpoints.
    WriteTimeout: 15 * time.Second,

    // IdleTimeout limits how long a keep-alive connection can stay idle
    // between a completed response and the next request.
    IdleTimeout: 120 * time.Second,

    // MaxHeaderBytes limits the total header size.
    // If unset, http.DefaultMaxHeaderBytes is used: 1 MiB.
    MaxHeaderBytes: 1 << 20,
}
```

::: tip
Timeouts protect not only against malicious clients, but also against ordinary network failures: a slow client, a stuck connection or a client that starts sending a request and then stops.
:::

## ReadTimeout

`ReadTimeout` sets the overall deadline for reading an incoming request. This includes both headers and body.

This field is useful when a client sends data too slowly: for example, when it uploads a large form body slowly or never finishes sending JSON. When the deadline expires, the server stops reading the request.

```go
server := &http.Server{
    Addr:        ":8080",
    ReadTimeout: 10 * time.Second,
}
```

`ReadTimeout` does not limit body size. A fast client can still send an oversized payload, so request bodies need a separate limit with [`http.MaxBytesReader`](https://pkg.go.dev/net/http#MaxBytesReader).

## WriteTimeout

`WriteTimeout` sets the deadline for writing a response. It protects the server from clients that read the response too slowly and cause the handler to hold the connection for too long.

```go
server := &http.Server{
    Addr:         ":8080",
    WriteTimeout: 15 * time.Second,
}
```

For ordinary JSON APIs, this is usually a safe limit. For streaming responses, such as Server-Sent Events or long file transfers, a small `WriteTimeout` can interrupt a legitimate response. Such handlers need a carefully chosen value, or the global timeout should not be overly strict. More targeted response write control is covered in [Streaming Responses](/en/net-http/server/responsecontroller).

## IdleTimeout

`IdleTimeout` limits how long a keep-alive connection can sit idle after a response has completed. If the client does not send the next request within this time, the server closes the connection.

```go
server := &http.Server{
    Addr:        ":8080",
    IdleTimeout: 120 * time.Second,
}
```

If `IdleTimeout` is not set, Go uses `ReadTimeout`. If both fields are zero, idle connections can remain open without a server-side deadline.

## MaxHeaderBytes

`MaxHeaderBytes` limits the total request header size. If the field is not set, [`http.DefaultMaxHeaderBytes`](https://pkg.go.dev/net/http#DefaultMaxHeaderBytes) is used: 1 MiB.

```go
server := &http.Server{
    Addr:           ":8080",
    MaxHeaderBytes: 1 << 20, // 1 MiB
}
```

The default is enough for most APIs. Decrease it when you need tighter per-connection memory limits, or increase it if the application legitimately accepts large headers, such as large cookies.

## Limiting Request Body Size

`http.Server` has no field that limits body size for all requests. Request bodies are limited at the handler level with `http.MaxBytesReader`.

`MaxBytesReader` wraps `r.Body`. All later readers, such as `json.Decoder`, `io.ReadAll` and `r.ParseMultipartForm`, read from the limited stream. If the limit is exceeded, reading returns an error of type [`*http.MaxBytesError`](https://pkg.go.dev/net/http#MaxBytesError).

```go
func createPostHandler(w http.ResponseWriter, r *http.Request) {
    const maxBodySize = 1 << 20 // 1 MiB

    r.Body = http.MaxBytesReader(w, r.Body, maxBodySize)
    defer r.Body.Close()

    body, err := io.ReadAll(r.Body)
    if err != nil {
        var maxBytesErr *http.MaxBytesError
        if errors.As(err, &maxBytesErr) {
            http.Error(w, "request body too large", http.StatusRequestEntityTooLarge)
            return
        }

        http.Error(w, "failed to read request body", http.StatusBadRequest)
        return
    }

    // ... process body
    _ = body
}
```

For JSON handlers, set `MaxBytesReader` before creating `json.Decoder`. For multipart uploads, set it before calling `r.ParseMultipartForm` or `r.FormFile`.
