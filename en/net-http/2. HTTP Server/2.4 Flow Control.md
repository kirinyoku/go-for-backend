# Handler Flow Control

An HTTP handler in Go has a strict lifecycle: receive a request, run the application logic, write at most one response and return. Flow-control mistakes often lead to double response writes, leaked goroutines or expensive work continuing after the client has closed the connection or cancelled the request.

## One Request, One Response

For each `ServeHTTP` call, a handler should produce one logical response. After writing an error, redirect or successful result, return from the handler.

A common mistake is to call [`http.Error`](https://pkg.go.dev/net/http#Error) and then keep executing the function:

```go
func badHandler(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    if id == "" {
        http.Error(w, "missing id", http.StatusBadRequest)
    }

    // This code still runs after http.Error.
    w.Header().Set("Content-Type", "application/json")
    fmt.Fprintln(w, `{"status":"ok"}`)
}
```

`http.Error` has already written the status and response body. The next write tries to add another response on top of the first one: headers have already been committed, the status can no longer be changed and the response body may become invalid.

The correct version returns early:

```go
func goodHandler(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    if id == "" {
        http.Error(w, "missing id", http.StatusBadRequest)
        return
    }

    w.Header().Set("Content-Type", "application/json")
    fmt.Fprintln(w, `{"status":"ok"}`)
}
```

## Early Returns and Guard Clauses

HTTP handlers work well with guard clauses: validate conditions first, send an error response and return immediately when something is wrong, and keep the successful path at the end of the function.

```go
func updateUserHandler(w http.ResponseWriter, r *http.Request) {
    if r.Method != http.MethodPut {
        http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
        return
    }

    id := r.PathValue("id")
    if id == "" {
        http.Error(w, "missing id", http.StatusBadRequest)
        return
    }

    var req UpdateUserRequest
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
        http.Error(w, "invalid JSON", http.StatusBadRequest)
        return
    }

    w.Header().Set("Content-Type", "application/json")
    fmt.Fprintf(w, `{"id":%q,"status":"updated"}`+"\n", id)
}
```

This style makes an important property visible: each branch either finishes the handler or leads to one successful response.

## Request Context

Every [`http.Request`](https://pkg.go.dev/net/http#Request) carries a [`context.Context`](https://pkg.go.dev/context#Context), available through [`r.Context()`](https://pkg.go.dev/net/http#Request.Context). This context tells the handler and its dependencies that work for the request is no longer needed.

The request context is cancelled when:

- the client closes the connection or cancels the request;
- for HTTP/2, the specific stream is cancelled;
- the server closes active connections through `Server.Close`;
- a child context created by the handler reaches its own deadline.

Context errors are usually checked with [`errors.Is`](https://pkg.go.dev/errors#Is):

| Error                                                                     | Meaning |
| :------------------------------------------------------------------------ | :------ |
| [`context.Canceled`](https://pkg.go.dev/context#Canceled)                 | The operation was cancelled, for example after the client closed the connection. |
| [`context.DeadlineExceeded`](https://pkg.go.dev/context#DeadlineExceeded) | A context deadline or timeout expired. |

```go
func handler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()

    if err := ctx.Err(); err != nil {
        log.Printf("request already cancelled: %v", err)
        return
    }

    fmt.Fprintln(w, "OK")
}
```

::: info
`context.Context` does not stop code by itself. It only carries a cancellation signal through `Done()` and `Err()`. The operation must either check the context itself or pass it to a library that knows how to observe it.
:::

## Passing Context to External Operations

The main rule is simple: if work is being done on behalf of an HTTP request, it should receive that request's context.

### Database Query

The [`database/sql`](https://pkg.go.dev/database/sql) package supports context through methods with the `Context` suffix: [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext), [`QueryRowContext`](https://pkg.go.dev/database/sql#DB.QueryRowContext), [`ExecContext`](https://pkg.go.dev/database/sql#DB.ExecContext), [`BeginTx`](https://pkg.go.dev/database/sql#Conn.BeginTx).

```go
func getUserHandler(db *sql.DB) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        ctx := r.Context()
        id := r.PathValue("id")

        var user User
        err := db.QueryRowContext(ctx, "SELECT id, name FROM users WHERE id = $1", id).
            Scan(&user.ID, &user.Name)
        if err != nil {
            if errors.Is(err, context.Canceled) {
                return
            }
            if errors.Is(err, sql.ErrNoRows) {
                http.Error(w, "not found", http.StatusNotFound)
                return
            }

            http.Error(w, "database error", http.StatusInternalServerError)
            return
        }

        writeJSON(w, http.StatusOK, user)
    }
}
```

If the client disconnects while the database query is running, the driver receives the cancellation signal through the context. The exact behavior depends on the driver and database system, but without passing the context, the cancellation definitely cannot reach that operation.

### Outgoing HTTP Request

Use [`http.NewRequestWithContext`](https://pkg.go.dev/net/http#NewRequestWithContext) when calling an external HTTP service. That way, cancellation of the incoming request also cancels the outgoing request.

```go
func proxyHandler(client *http.Client) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        req, err := http.NewRequestWithContext(
            r.Context(),
            http.MethodGet,
            "https://api.example.com/profile",
            nil,
        )
        if err != nil {
            http.Error(w, "internal error", http.StatusInternalServerError)
            return
        }

        resp, err := client.Do(req)
        if err != nil {
            if errors.Is(err, context.Canceled) {
                return
            }

            http.Error(w, "upstream error", http.StatusBadGateway)
            return
        }
        defer resp.Body.Close()

        w.Header().Set("Content-Type", resp.Header.Get("Content-Type"))
        w.WriteHeader(resp.StatusCode)
        if _, err := io.Copy(w, resp.Body); err != nil {
            log.Printf("failed to copy upstream response: %v", err)
            return
        }
    }
}
```

::: info
HTTP client timeouts are covered in detail in [Timeouts and Context](/en/net-http/client/timeouts-and-context). In a server handler, the important part is to tie outgoing work to the lifecycle of the incoming request.
:::

## Timeout for a Long Operation

Sometimes a handler needs to stop an operation before the client does. For example, an external service may respond too slowly, and the server should return a controlled error to the client.

Create a child context with [`context.WithTimeout`](https://pkg.go.dev/context#WithTimeout):

```go
func reportHandler(w http.ResponseWriter, r *http.Request) {
    ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
    defer cancel()

    report, err := buildReport(ctx)
    if err != nil {
        if errors.Is(err, context.DeadlineExceeded) {
            http.Error(w, "report timeout", http.StatusGatewayTimeout)
            return
        }
        if errors.Is(err, context.Canceled) {
            return
        }

        http.Error(w, "failed to build report", http.StatusInternalServerError)
        return
    }

    writeJSON(w, http.StatusOK, report)
}
```

`defer cancel()` is still needed when the operation succeeds: it releases timer resources associated with the child context.

::: warning
A context timeout does not forcibly interrupt arbitrary code. The `buildReport` function must accept `context.Context` and either check it periodically or pass it further into operations that support cancellation.
:::

## Goroutines Inside a Handler

A goroutine started inside a handler does not stop automatically when the handler returns. If it keeps running after the request is cancelled, it can leak resources.

Problematic example:

```go
func leakyHandler(w http.ResponseWriter, r *http.Request) {
    go func() {
        sendAuditEvent()
    }()

    fmt.Fprintln(w, "accepted")
}
```

If `sendAuditEvent` hangs, the goroutine keeps running independently of the request. If the operation really must outlive the request, it is usually better to hand it off to a queue or worker instead of tying it to `ResponseWriter` or local handler state.

For work that is needed only while the request is alive, the goroutine should observe `ctx.Done()`.

In this pattern, the handler does not wait for the goroutine directly. Instead, it waits for the first of three events:

- the request is cancelled through `ctx.Done()`;
- the worker goroutine returns an error through `errCh`;
- the worker goroutine returns a result through `resultCh`.

After the first event, the handler exits. The channels are buffered with capacity `1` for that reason: if the handler has already returned because the context was cancelled, the worker goroutine can still send a result or error and exit instead of blocking on the send.

```go
func safeHandler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()
    // The worker goroutine reports either a result or an error.
    resultCh := make(chan Result, 1)
    errCh := make(chan error, 1)

    go func() {
        // loadResult receives ctx, so it can stop when the request is cancelled.
        result, err := loadResult(ctx)
        if err != nil {
            errCh <- err
            return
        }
        resultCh <- result
    }()

    select {
    case <-ctx.Done():
        // The request is no longer relevant: the client closed the connection
        // or a deadline expired.
        return
    case err := <-errCh:
        if errors.Is(err, context.Canceled) {
            return
        }
        http.Error(w, "failed to load result", http.StatusInternalServerError)
        return
    case result := <-resultCh:
        // The result arrived before the request was cancelled, so the response can be written.
        writeJSON(w, http.StatusOK, result)
    }
}
```

::: warning
Do not use `http.ResponseWriter` from a background goroutine after the handler returns. `ResponseWriter` belongs to a specific request and is valid only during that request's handling.
:::
