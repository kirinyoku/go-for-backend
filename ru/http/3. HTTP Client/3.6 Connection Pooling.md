# Пул соединений

Для обеспечения высокой производительности Go использует механизм **Keep-Alive**, который позволяет переиспользовать одно и то же TCP-соединение для нескольких последовательных HTTP-запросов к одному хосту. Управление этим механизмом осуществляется через пул соединений в [`http.Transport`](https://pkg.go.dev/net/http#Transport).

## Как работает Keep-Alive

Когда запрос завершён и тело ответа полностью вычитано и закрыто, соединение не разрывается, а помещается в **«пул простаивающих соединений»** (Idle Pool). При следующем запросе к тому же хосту клиент сначала проверяет пул и, если есть свободное соединение, использует его. Это избавляет от накладных расходов на [TCP Three-way Handshake](https://en.wikipedia.org/wiki/Handshake_(computing)#TCP_three-way_handshake) и [TLS Handshake](https://en.wikipedia.org/wiki/Handshake_(computing)#TLS_handshake) (самая дорогая часть).

## Параметры пула

Значения по умолчанию в [`http.DefaultTransport`](https://pkg.go.dev/net/http#DefaultTransport) подходят для большинства сценариев, но могут стать узким местом при высокой нагрузке на конкретные хосты.

| Параметр                                                                    | По умолчанию   | Описание                                                                     |
| :-------------------------------------------------------------------------- | :------------- | :--------------------------------------------------------------------------- |
| [`MaxIdleConns`](https://pkg.go.dev/net/http#Transport.MaxIdleConns)        | 100            | Общее количество простаивающих соединений для всех хостов.                   |
| [`MaxIdleConnsPerHost`](https://pkg.go.dev/net/http#Transport.MaxIdleConnsPerHost) | 2              | Количество простаивающих соединений на **один** конкретный хост.             |
| [`MaxConnsPerHost`](https://pkg.go.dev/net/http#Transport.MaxConnsPerHost)         | 0 (без лимита) | Жёсткий лимит **всех** соединений (активных + простаивающих) к одному хосту. |
| [`IdleConnTimeout`](https://pkg.go.dev/net/http#Transport.IdleConnTimeout)  | 90s            | Время, через которое простаивающее соединение будет закрыто.                 |

```go
transport := http.DefaultTransport.(*http.Transport).Clone()

// Значение 2 по умолчанию критично мало для высоконагруженных систем.
// Если приложение делает 100 запросов в секунду к одному API,
// то при лимите 2 большинство соединений будет закрываться сразу после использования.
transport.MaxIdleConnsPerHost = 100
transport.MaxIdleConns = 100

// Опционально: защита от перегрузки целевого сервера (client-side throttling).
// transport.MaxConnsPerHost = 200
```

## Когда соединение не возвращается в пул

Подробный алгоритм закрытия и вычитывания тела ответа описан в статье [Чтение и обработка ответов](./response-handling). В контексте пула важно помнить последствие: соединение не сможет быть переиспользовано, если ответ не был корректно завершён.

Основные причины:

1. Не закрыто тело ответа (`resp.Body.Close()`).
2. Не вычитано тело ответа до конца ([`io.EOF`](https://pkg.go.dev/io#EOF)).
3. Сервер прислал заголовок `Connection: close`.
4. Произошла ошибка сетевого уровня при чтении данных.

## Мониторинг соединений

Избыточное создание соединений (когда пул не работает) можно заметить по следующим признакам:

- Рост числа файловых дескрипторов.
- Состояния `TIME_WAIT` в выводе `netstat`.
- Повышенные задержки из-за постоянных TLS-рукопожатий.

```bash
# Проверка количества соединений в TIME_WAIT (Linux)
netstat -ant | grep TIME_WAIT | wc -l
```

## Использование в микросервисах

В архитектуре микросервисов, где один сервис постоянно общается с 2–3 другими, крайне важно увеличивать `MaxIdleConnsPerHost`. Это позволяет поддерживать «горячие» каналы связи и минимизировать задержки.

::: warning 
Если вы используете кастомный `http.Transport`, убедитесь, что создаёте его **один раз** и переиспользуете в одном `http.Client`. Создание нового транспорта на каждый запрос полностью отключает пул соединений. 
:::
