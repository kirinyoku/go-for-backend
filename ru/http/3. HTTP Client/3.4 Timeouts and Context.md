# Таймауты и контекст

Управление временем выполнения запроса в Go строится из нескольких независимых механизмов. Они похожи по эффекту — запрос завершается ошибкой, — но отвечают за разные уровни:

- [`context.Context`](https://pkg.go.dev/context#Context) задает deadline конкретной операции и связывает HTTP-запрос с жизненным циклом вызывающего кода.
- [`http.Client.Timeout`](https://pkg.go.dev/net/http#Client.Timeout) задает общий лимит на один HTTP-запрос, включая чтение тела ответа.
- Таймауты [`http.Transport`](https://pkg.go.dev/net/http#Transport) ограничивают отдельные сетевые этапы: подключение, TLS handshake, ожидание заголовков ответа.

Эти механизмы можно использовать вместе. Запрос завершится по первому сработавшему ограничению.

## Таймаут запроса через Context

Контекст удобен, когда ограничение времени нужно задать для конкретного вызова, а не для всего HTTP-клиента. Например, один исходящий запрос может быть частью обработки входящего HTTP-запроса, фоновой задачи, RPC-вызова или цепочки обращений к нескольким сервисам.

```go
// fetchStatus выполняет GET-запрос и возвращает статус-код ответа.
// Функция принимает родительский контекст, чтобы вызывающий код мог отменить операцию.
func fetchStatus(ctx context.Context, client *http.Client, url string) (int, error) {
    ctx, cancel := context.WithTimeout(ctx, 2*time.Second)
    defer cancel()

    req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
    if err != nil {
        return 0, err
    }

    resp, err := client.Do(req)
    if err != nil {
        return 0, err
    }
    defer resp.Body.Close()

    _, _ = io.Copy(io.Discard, resp.Body)

    return resp.StatusCode, nil
}
```

Если родительский контекст отменится раньше, чем истечет локальный `2*time.Second`, HTTP-запрос тоже будет отменен. Это особенно важно на сервере: исходящий запрос не должен продолжаться бесконечно после того, как входящий клиент уже отключился.

## Общий таймаут клиента

Поле `http.Client.Timeout` задает общий лимит на один вызов [`client.Do`](https://pkg.go.dev/net/http#Client.Do) и последующее чтение тела ответа. Это время включает:

1. Установку TCP-соединения.
2. TLS-рукопожатие.
3. Отправку запроса.
4. Ожидание и чтение заголовков ответа.
5. Чтение всего тела ответа.

```go
client := &http.Client{
    Timeout: 10 * time.Second,
}
```

Если за 10 секунд операция не завершилась, соединение будет разорвано. Ошибка может появиться не только из `client.Do`, но и позже, при чтении `resp.Body`, потому что `Client.Timeout` покрывает тело ответа.

## Таймауты http.Transport

`http.Transport` управляет низкоуровневым выполнением запроса: созданием соединений, TLS, keep-alive и ожиданием ответа. Его таймауты нужны, когда требуется ограничить конкретный этап, а не весь запрос целиком.

| Настройка                                                                              | Что ограничивает                                                                                                    |
| :------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------ |
| [`DialContext`](https://pkg.go.dev/net/http#Transport.DialContext)                     | Время на установку сетевого соединения. DNS lookup может входить в этот этап в зависимости от resolver и окружения. |
| [`TLSHandshakeTimeout`](https://pkg.go.dev/net/http#Transport.TLSHandshakeTimeout)     | Время на TLS handshake после подключения.                                                                           |
| [`ResponseHeaderTimeout`](https://pkg.go.dev/net/http#Transport.ResponseHeaderTimeout) | Время ожидания заголовков ответа после отправки запроса.                                                            |
| [`ExpectContinueTimeout`](https://pkg.go.dev/net/http#Transport.ExpectContinueTimeout) | Ожидание ответа `100 Continue`, если запрос использует `Expect: 100-continue`.                                      |
| [`IdleConnTimeout`](https://pkg.go.dev/net/http#Transport.IdleConnTimeout)             | Сколько keep-alive соединение может простаивать в пуле. Это не timeout выполнения одного запроса.                   |

```go
transport := &http.Transport{
    DialContext: (&net.Dialer{
        Timeout:   3 * time.Second,
        KeepAlive: 30 * time.Second,
    }).DialContext,
    TLSHandshakeTimeout:   3 * time.Second,
    ResponseHeaderTimeout: 5 * time.Second,
    ExpectContinueTimeout: 1 * time.Second,
    IdleConnTimeout:       90 * time.Second,
}

client := &http.Client{
    Transport: transport,
    Timeout:   15 * time.Second,
}
```

::: info
В примере `DialContext` настраивается через [`net.Dialer`](https://pkg.go.dev/net#Dialer): его `Timeout` ограничивает время подключения, а метод `DialContext` передается в `http.Transport` как функция открытия соединения.
:::

Такой клиент ограничивает отдельные фазы запроса и дополнительно имеет общий предел `15*time.Second`. Если `ResponseHeaderTimeout` истечет раньше общего `Client.Timeout`, ошибка вернется на этапе ожидания заголовков. Если сервер быстро отправит заголовки, но затем будет медленно отдавать тело, ограничение чтения тела останется за `Client.Timeout` или context deadline.

## Отмена запроса

Контекст позволяет отменить запрос до наступления таймаута. Это происходит, например, когда пользователь закрыл соединение с сервером, родительская задача завершилась или вызывающий код явно вызвал `cancel`.

```go
func process(ctx context.Context, client *http.Client, url string) error {
    req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
    if err != nil {
        return err
    }

    resp, err := client.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()

    _, err = io.Copy(io.Discard, resp.Body)
    return err
}
```

Когда `ctx` отменяется, `client.Do` или чтение `resp.Body` возвращают ошибку. Для диагностики причины нужно смотреть как на ошибку, так и на состояние контекста.

## Определение причины сбоя

Ошибки [`net/http`](https://pkg.go.dev/net/http) обычно приходят обернутыми, например в [`*url.Error`](https://pkg.go.dev/net/url#Error). Поэтому проверку нужно делать через [`errors.Is`](https://pkg.go.dev/errors#Is), [`os.IsTimeout`](https://pkg.go.dev/os#IsTimeout) или интерфейс [`net.Error`](https://pkg.go.dev/net#Error), а не через сравнение строк.

```go
resp, err := client.Do(req)
if err != nil {
    switch {
    case errors.Is(err, context.Canceled):
        return fmt.Errorf("request canceled: %w", err)

    case errors.Is(req.Context().Err(), context.DeadlineExceeded):
        return fmt.Errorf("request context deadline exceeded: %w", err)

    case os.IsTimeout(err):
        return fmt.Errorf("client or transport timeout: %w", err)
    }

    var netErr net.Error
    if errors.As(err, &netErr) && netErr.Timeout() {
        return fmt.Errorf("network timeout: %w", err)
    }

    return fmt.Errorf("network error: %w", err)
}
defer resp.Body.Close()
```

Для чтения тела ответа проверка выполняется отдельно: ошибка может возникнуть уже после успешного `client.Do`.

```go
body, err := io.ReadAll(resp.Body)
if err != nil {
    return fmt.Errorf("read response body: %w", err)
}
_ = body
```

::: tip
`http.Client.Timeout` и context deadline могут проявляться как timeout-ошибки на разных этапах. Для логирования полезно фиксировать не только факт timeout, но и настроенные значения: operation deadline, `Client.Timeout`, `Dialer.Timeout`, `TLSHandshakeTimeout`, `ResponseHeaderTimeout`.
:::
