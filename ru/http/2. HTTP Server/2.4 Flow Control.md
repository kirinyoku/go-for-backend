# Управление выполнением обработчика

У HTTP-обработчик в Go есть строгий жизненный цикл: получить запрос, выполнить бизнес-логику, записать не более одного ответа и завершиться. Ошибки управления выполнением часто приводят к двойной записи ответа, утечкам горутин или продолжению дорогой операции после того, как клиент закрыл соединение или отменил запрос.

## Один запрос — один ответ

Для каждого вызова `ServeHTTP` обработчик должен сформировать один логический ответ. После записи ошибки, редиректа или успешного результата управление нужно вернуть через `return`.

Типичная ошибка — вызвать [`http.Error`](https://pkg.go.dev/net/http#Error), но продолжить выполнение функции:

```go
func badHandler(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    if id == "" {
        http.Error(w, "missing id", http.StatusBadRequest)
    }

    // Код продолжит выполняться даже после http.Error.
    w.Header().Set("Content-Type", "application/json")
    fmt.Fprintln(w, `{"status":"ok"}`)
}
```

`http.Error` уже записывает статус и тело ответа. Следующая запись попытается добавить второй ответ поверх первого: заголовки уже зафиксированы, статус изменить нельзя, а тело ответа может стать некорректным.

Правильный вариант использует ранний выход:

```go
func goodHandler(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    if id == "" {
        http.Error(w, "missing id", http.StatusBadRequest)
        return
    }

    w.Header().Set("Content-Type", "application/json")
    fmt.Fprintln(w, `{"status":"ok"}`)
}
```

## Ранний выход (guard clauses)

Для HTTP-обработчиков хорошо подходит стиль раннего выхода через guard clauses: сначала проверяются условия, при ошибке сразу отправляется ответ и выполняется `return`, а основной успешный путь остаётся в конце функции.

```go
func updateUserHandler(w http.ResponseWriter, r *http.Request) {
    if r.Method != http.MethodPut {
        http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
        return
    }

    id := r.PathValue("id")
    if id == "" {
        http.Error(w, "missing id", http.StatusBadRequest)
        return
    }

    var req UpdateUserRequest
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
        http.Error(w, "invalid JSON", http.StatusBadRequest)
        return
    }

    w.Header().Set("Content-Type", "application/json")
    fmt.Fprintf(w, `{"id":%q,"status":"updated"}`+"\n", id)
}
```

Такой стиль делает важное свойство видимым: каждая ветка либо завершает обработчик, либо ведёт к единственному успешному ответу.

## Контекст запроса

Каждый [`http.Request`](https://pkg.go.dev/net/http#Request) содержит [`context.Context`](https://pkg.go.dev/context#Context), доступный через [`r.Context()`](https://pkg.go.dev/net/http#Request.Context). Этот контекст сообщает обработчику и зависимостям, что работа над запросом больше не нужна.

Контекст запроса отменяется когда:

- клиент закрыл соединение или отменил запрос;
- для HTTP/2 был отменён конкретный stream;
- сервер завершает активные соединения через `Server.Close`;
- дочерний контекст обработчика достиг собственного deadline.

Ошибки контекста обычно проверяются через [`errors.Is`](https://pkg.go.dev/errors#Is):

| Ошибка                                                                    | Значение                                                            |
| :------------------------------------------------------------------------ | :------------------------------------------------------------------ |
| [`context.Canceled`](https://pkg.go.dev/context#Canceled)                 | Операция была отменена, например после закрытия соединения клиентом |
| [`context.DeadlineExceeded`](https://pkg.go.dev/context#DeadlineExceeded) | Истёк deadline или timeout контекста                                |

```go
func handler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()

    if err := ctx.Err(); err != nil {
        log.Printf("request already cancelled: %v", err)
        return
    }

    fmt.Fprintln(w, "OK")
}
```

::: info
`context.Context` сам по себе не останавливает код. Он только передаёт сигнал отмены через `Done()` и `Err()`. Операция должна сама проверять контекст или передать его библиотеке, которая умеет это делать.
:::

## Передача контекста во внешние операции

Главное правило: если работа выполняется от имени HTTP-запроса, она должна получить контекст этого запроса.

### Запрос к базе данных

Пакет [`database/sql`](https://pkg.go.dev/database/sql) поддерживает контекст через методы с суффиксом `Context`: [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext), [`QueryRowContext`](https://pkg.go.dev/database/sql#DB.QueryRowContext), [`ExecContext`](https://pkg.go.dev/database/sql#DB.ExecContext), [`BeginTx`](https://pkg.go.dev/database/sql#Conn.BeginTx).

```go
func getUserHandler(db *sql.DB) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        ctx := r.Context()
        id := r.PathValue("id")

        var user User
        err := db.QueryRowContext(ctx, "SELECT id, name FROM users WHERE id = $1", id).
            Scan(&user.ID, &user.Name)
        if err != nil {
            if errors.Is(err, context.Canceled) {
                return
            }
            if errors.Is(err, sql.ErrNoRows) {
                http.Error(w, "not found", http.StatusNotFound)
                return
            }

            http.Error(w, "database error", http.StatusInternalServerError)
            return
        }

        writeJSON(w, http.StatusOK, user)
    }
}
```

Если клиент отключится во время запроса к базе, драйвер получает сигнал отмены через контекст. Фактическое поведение зависит от драйвера и СУБД, но без передачи контекста отмена точно не дойдёт до этой операции.

### Исходящий HTTP-запрос

Для вызова внешнего HTTP-сервиса используется [`http.NewRequestWithContext`](https://pkg.go.dev/net/http#NewRequestWithContext). Тогда отмена входящего запроса прерывает и исходящий запрос.

```go
func proxyHandler(client *http.Client) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        req, err := http.NewRequestWithContext(
            r.Context(),
            http.MethodGet,
            "https://api.example.com/profile",
            nil,
        )
        if err != nil {
            http.Error(w, "internal error", http.StatusInternalServerError)
            return
        }

        resp, err := client.Do(req)
        if err != nil {
            if errors.Is(err, context.Canceled) {
                return
            }

            http.Error(w, "upstream error", http.StatusBadGateway)
            return
        }
        defer resp.Body.Close()

        w.Header().Set("Content-Type", resp.Header.Get("Content-Type"))
        w.WriteHeader(resp.StatusCode)
        if _, err := io.Copy(w, resp.Body); err != nil {
            log.Printf("failed to copy upstream response: %v", err)
            return
        }
    }
}
```

::: info
Таймауты HTTP-клиента подробно рассматриваются в статье  [Таймауты и контекст](../client/timeouts-and-context). На стороне серверного обработчика важно связать исходящую работу с жизненным циклом входящего запроса.
:::

## Таймаут для долгой операции

Иногда обработчик должен завершить операцию раньше, чем это сделает клиент. Например, внешний сервис может отвечать слишком долго, а серверу нужно вернуть клиенту контролируемую ошибку.

Для этого создаётся дочерний контекст через [`context.WithTimeout`](https://pkg.go.dev/context#WithTimeout):

```go
func reportHandler(w http.ResponseWriter, r *http.Request) {
    ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
    defer cancel()

    report, err := buildReport(ctx)
    if err != nil {
        if errors.Is(err, context.DeadlineExceeded) {
            http.Error(w, "report timeout", http.StatusGatewayTimeout)
            return
        }
        if errors.Is(err, context.Canceled) {
            return
        }

        http.Error(w, "failed to build report", http.StatusInternalServerError)
        return
    }

    writeJSON(w, http.StatusOK, report)
}
```

`defer cancel()` нужен даже при успешном завершении операции: он освобождает ресурсы таймера, связанные с дочерним контекстом.

::: warning
Таймаут через контекст не прерывает произвольный код принудительно. Функция `buildReport` должна принимать `context.Context` и периодически проверять его либо передавать дальше в операции, которые поддерживают отмену.
:::

## Горутины внутри обработчика

Горутина, запущенная внутри обработчика, не завершается автоматически при возврате из обработчика. Если она продолжает работать после отмены запроса, это может привести к утечке ресурсов.

Проблемный пример:

```go
func leakyHandler(w http.ResponseWriter, r *http.Request) {
    go func() {
        sendAuditEvent()
    }()

    fmt.Fprintln(w, "accepted")
}
```

Если `sendAuditEvent` зависнет, горутина останется жить независимо от запроса. Если операция действительно должна пережить запрос, её лучше передать в отдельную очередь или worker, а не привязывать к `ResponseWriter` или локальному состоянию обработчика.

Для работы, которая нужна только пока жив запрос, горутина должна отслеживать `ctx.Done()`.

В таком паттерне обработчик не ждёт горутину напрямую. Вместо этого он ждёт первое из трёх событий:

- запрос был отменён через `ctx.Done()`;
- рабочая горутина вернула ошибку через `errCh`;
- рабочая горутина вернула результат через `resultCh`.

После первого события обработчик завершает работу. Поэтому каналы буферизованы размером `1`: если обработчик уже вышел по отмене контекста, рабочая горутина всё ещё сможет отправить результат или ошибку и завершиться, а не зависнуть на отправке.

```go
func safeHandler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()
    // Рабочая горутина сообщит обработчику либо результат, либо ошибку.
    resultCh := make(chan Result, 1)
    errCh := make(chan error, 1)

    go func() {
        // loadResult получает ctx, поэтому может остановиться при отмене запроса.
        result, err := loadResult(ctx)
        if err != nil {
            errCh <- err
            return
        }
        resultCh <- result
    }()

    select {
    case <-ctx.Done():
        // Запрос больше не актуален: клиент закрыл соединение или истёк deadline.
        return
    case err := <-errCh:
        if errors.Is(err, context.Canceled) {
            return
        }
        http.Error(w, "failed to load result", http.StatusInternalServerError)
        return
    case result := <-resultCh:
        // Результат получен до отмены запроса, поэтому можно записать ответ.
        writeJSON(w, http.StatusOK, result)
    }
}
```

::: warning
`http.ResponseWriter` не следует использовать из фоновой горутины после возврата обработчика. `ResponseWriter` принадлежит конкретному запросу и действителен только в рамках его обработки.
:::
