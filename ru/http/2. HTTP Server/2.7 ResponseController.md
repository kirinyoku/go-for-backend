# Потоковые ответы и http.ResponseController

Большинство HTTP-обработчиков в Go формирует ответ за один проход: проверяет входные данные, выполняет работу, записывает статус и тело. Для такого кода достаточно базового [`http.ResponseWriter`](/ru/net-http/intro/responsewriter): `Header`, `WriteHeader` и `Write`.

Сложность появляется в других сценариях: обработчик долго пишет ответ, отдаёт данные частями, работает с медленным клиентом или проходит через middleware, которое оборачивает `ResponseWriter`. В этих случаях уже важно не только *что* записать в ответ, но и *когда* данные уйдут клиенту и какие низкоуровневые возможности остались доступны после обёрток.

Для таких случаев в стандартной библиотеке есть [`http.ResponseController`](https://pkg.go.dev/net/http#ResponseController):

```go
rc := http.NewResponseController(w)
```

`ResponseController` не заменяет `ResponseWriter`. Он создаётся поверх текущего writer и позволяет вызвать расширенные операции, если конкретная реализация ответа их поддерживает: `Flush`, `SetReadDeadline`, `SetWriteDeadline`, `EnableFullDuplex` и `Hijack`.

Как и сам `ResponseWriter`, контроллер живёт только внутри текущего вызова обработчика. Его не нужно сохранять в структуру, передавать в фоновые горутины или использовать после возврата из `ServeHTTP`.

## Потоковая запись

Обычный HTTP-ответ можно собрать целиком и записать в конце обработчика. Но некоторые ответы полезно отправлять постепенно: Server-Sent Events, прогресс долгой операции, tail-like вывод логов, простые streaming-эндпоинты.

В таком обработчике недостаточно просто вызвать `Write`. Данные могут остаться в буфере, и клиент увидит их позже. Метод [`Flush`](https://pkg.go.dev/net/http#ResponseController.Flush) просит сервер отправить уже записанный фрагмент клиенту.

```go
func eventsHandler(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")

    rc := http.NewResponseController(w)
    ticker := time.NewTicker(time.Second)
    defer ticker.Stop()

    for i := 0; i < 3; i++ {
        select {
        case <-r.Context().Done():
            return
        case <-ticker.C:
        }

        if _, err := fmt.Fprintf(w, "data: event %d\n\n", i); err != nil {
            return
        }
        if err := rc.Flush(); err != nil {
            return
        }
    }
}
```

::: warning
`Flush` не превращает любой ответ в гарантированно мгновенную доставку до приложения-клиента. Между сервером и клиентом могут быть прокси, балансировщик или буферы самого клиента. Но без `Flush` сервер даже не пытается отправить текущий фрагмент раньше завершения обработчика или заполнения буфера.
:::

## Deadline конкретного обработчика

Глобальные лимиты сервера настраиваются через поля [`http.Server`](/ru/net-http/server/resource-limits): `ReadTimeout`, `WriteTimeout`, `IdleTimeout`, `MaxHeaderBytes`. Это хороший базовый слой защиты, но иногда одному обработчику нужен отдельный deadline.

Например, обычные JSON-эндпоинты могут жить с коротким `WriteTimeout`, а streaming-эндпоинт должен писать дольше. В таком случае обработчик может управлять deadline записи сам:

```go
func downloadHandler(w http.ResponseWriter, r *http.Request) {
    rc := http.NewResponseController(w)
    if err := rc.SetWriteDeadline(time.Now().Add(30 * time.Second)); err != nil {
        http.Error(w, "write deadline is not supported", http.StatusInternalServerError)
        return
    }

    file, err := os.Open("backup.tar")
    if err != nil {
        http.Error(w, "file not found", http.StatusNotFound)
        return
    }
    defer file.Close()

    w.Header().Set("Content-Type", "application/octet-stream")

    if _, err := io.Copy(w, file); err != nil {
        // Клиент мог отключиться или истёк deadline записи.
        return
    }
}
```

[`SetWriteDeadline`](https://pkg.go.dev/net/http#ResponseController.SetWriteDeadline) задаёт deadline на запись ответа. После истечения deadline запись не должна блокироваться, но может завершиться успешно, если данные уже попали в буфер. Поэтому для больших и потоковых ответов ошибки записи нужно проверять, а не игнорировать.

Для чтения запроса есть [`SetReadDeadline`](https://pkg.go.dev/net/http#ResponseController.SetReadDeadline). Он задаёт deadline на чтение всего запроса, включая body:

```go
func uploadHandler(w http.ResponseWriter, r *http.Request) {
    rc := http.NewResponseController(w)
    if err := rc.SetReadDeadline(time.Now().Add(10 * time.Second)); err != nil {
        http.Error(w, "read deadline is not supported", http.StatusInternalServerError)
        return
    }

    r.Body = http.MaxBytesReader(w, r.Body, 10<<20)
    defer r.Body.Close()

    if _, err := io.Copy(io.Discard, r.Body); err != nil {
        http.Error(w, "failed to read request body", http.StatusBadRequest)
        return
    }

    w.WriteHeader(http.StatusNoContent)
}
```

Deadline ограничивает время, но не размер. Ограничение размера тела всё равно делается через [`http.MaxBytesReader`](https://pkg.go.dev/net/http#MaxBytesReader).

## Поддержка зависит от writer

`ResponseController` работает через возможности конкретного `ResponseWriter`. Если нужная операция не поддерживается, метод вернёт ошибку, совместимую с [`http.ErrNotSupported`](https://pkg.go.dev/net/http#ErrNotSupported).

```go
if err := rc.Flush(); err != nil {
    if errors.Is(err, http.ErrNotSupported) {
        http.Error(w, "streaming is not supported", http.StatusInternalServerError)
        return
    }

    http.Error(w, "failed to flush response", http.StatusInternalServerError)
    return
}
```

Такую ошибку лучше проверять через [`errors.Is`](https://pkg.go.dev/errors#Is), потому что она может быть обёрнута.

## Middleware и Unwrap

Логирующее middleware часто оборачивает `ResponseWriter`, чтобы запомнить статус и размер ответа. Такая обёртка нужна, но она может случайно спрятать расширенные возможности исходного writer.

Чтобы `ResponseController` мог пройти через обёртку к настоящему writer, обёртка должна реализовать метод `Unwrap`:

```go
type responseWriter struct {
    http.ResponseWriter
    status int
    bytes  int
}

func (rw *responseWriter) Unwrap() http.ResponseWriter {
    return rw.ResponseWriter
}
```

Без `Unwrap` код ниже по цепочке может получить `http.ErrNotSupported`, хотя исходный writer поддерживал `Flush`, deadline-методы или другие расширенные операции. Именно поэтому обёртки `ResponseWriter` лучше проектировать не только вокруг `Write` и `WriteHeader`, но и вокруг совместимости с остальной HTTP-инфраструктурой.

## Full-duplex

Обычно обработчик сначала читает запрос, а затем пишет ответ. Для HTTP/1 сервер Go по умолчанию может дочитать оставшуюся часть тела запроса перед началом записи ответа. Это нормальное поведение для большинства обработчиков, но оно мешает сценариям, где нужно чередовать чтение `r.Body` и запись в `w`.

[`EnableFullDuplex`](https://pkg.go.dev/net/http#ResponseController.EnableFullDuplex) сообщает серверу, что обработчик действительно хочет читать запрос и писать ответ одновременно.

```go
func duplexHandler(w http.ResponseWriter, r *http.Request) {
    rc := http.NewResponseController(w)
    if err := rc.EnableFullDuplex(); err != nil {
        http.Error(w, "full duplex is not supported", http.StatusInternalServerError)
        return
    }

    w.Header().Set("Content-Type", "text/plain")

    scanner := bufio.NewScanner(r.Body)
    for scanner.Scan() {
        if _, err := fmt.Fprintf(w, "received: %s\n", scanner.Text()); err != nil {
            return
        }
        if err := rc.Flush(); err != nil {
            return
        }
    }
}
```

Для HTTP/2 сервер Go и так допускает одновременное чтение запроса и запись ответа, но обработчик всё равно должен быть готов к ошибке: между ним и настоящей реализацией ответа может стоять middleware-обёртка или другой writer.

## Hijack

[`Hijack`](https://pkg.go.dev/net/http#ResponseController.Hijack) забирает сетевое соединение у HTTP-сервера. После этого `net/http` больше не управляет ответом: приложение само пишет HTTP-строки или уже не-HTTP протокол в `net.Conn`.

Это низкоуровневый механизм, который редко нужен обычному backend API. Типичные причины — специальные протоколы поверх TCP или старый WebSocket-upgrade код. Для большинства потоковых HTTP-ответов достаточно `Write`, `Flush`, контекста запроса и корректных deadline.

::: warning
Соединения, переведённые в режим `Hijack`, не завершаются обычным `http.Server.Shutdown` как активные HTTP-запросы. Их жизненный цикл нужно учитывать отдельно, например через `RegisterOnShutdown`. Это разбирается в статье [Graceful Shutdown](/ru/net-http/server/graceful-shutdown).
:::
