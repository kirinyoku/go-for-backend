# CORS как middleware

**CORS (Cross-Origin Resource Sharing)** — это механизм браузера, который управляет доступом JavaScript-кода к ответам с другого сайта.

Типичный сценарий: frontend открыт на `https://app.example.com`, а API работает на `https://api.example.com`. Чтобы такой запрос работал из браузера, сервер должен явно разрешить доступ через CORS-заголовки. В `net/http` такую логику удобно оформлять как middleware поверх [`http.Handler`](https://pkg.go.dev/net/http#Handler).

## Принцип работы

В CORS сравниваются origin. Origin включает схему, хост и порт, поэтому `https://app.example.com`, `https://api.example.com` и `http://app.example.com` — разные origin.

Когда браузерный код отправляет cross-origin запрос, браузер добавляет заголовок `Origin`:

```http
GET /api/profile HTTP/1.1
Origin: https://app.example.com
```

Если сервер разрешает этот origin, он возвращает его в `Access-Control-Allow-Origin`:

```http
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://app.example.com
```

После этого браузер разрешает JavaScript-коду прочитать ответ. Если заголовка нет или origin не совпадает, запрос на сервер всё равно мог быть выполнен, но браузер скроет ответ от frontend-кода.

::: warning
CORS — не авторизация и не защита endpoint от прямых HTTP-запросов. Это контракт между сервером и браузером: сервер отдаёт CORS-заголовки, а браузер применяет ограничение. `curl`, серверные клиенты и тесты через `net/http` CORS-политику не применяют.
:::

Для части запросов браузер сначала отправляет **preflight** — предварительный `OPTIONS`-запрос. Так браузер проверяет, разрешит ли сервер будущий запрос с конкретным методом и заголовками.

```http
OPTIONS /api/profile HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization
```

Preflight часто появляется при методах `PUT`, `PATCH`, `DELETE`, при заголовке `Authorization` или при `Content-Type: application/json`. В ответ сервер возвращает CORS-заголовки и не вызывает бизнес-обработчик:

```http
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 7200
```

`Access-Control-Max-Age` позволяет браузеру кэшировать успешный preflight и не отправлять `OPTIONS` перед каждым похожим запросом. Слишком большое значение неудобно при изменении CORS-политики: браузеры пользователей могут ещё некоторое время использовать старое разрешение.

::: tip
Preflight-запрос обычно не содержит пользовательский `Authorization` и не должен менять состояние приложения. Поэтому CORS middleware часто ставят перед auth middleware, чтобы `OPTIONS` не был отклонён авторизацией до установки CORS-заголовков.
:::

## Реализация

В стандартной библиотеке нет готового CORS-обработчика. На стороне `net/http` задача сервера состоит в том, чтобы проверить заголовок `Origin`, вернуть подходящие `Access-Control-*` заголовки и отдельно обработать preflight-запросы. Удобнее всего это делать через middleware поверх [`http.Handler`](https://pkg.go.dev/net/http#Handler).

Обычно CORS настраивают через allowlist: список origin, которым разрешено читать ответы API. Этот список лучше передавать через конфигурацию приложения, чтобы значения можно было менять между окружениями разработки без изменения кода.

```go
type CORSConfig struct {
    AllowedOrigins  map[string]struct{}
    AllowedMethods  string
    AllowedHeaders  string
    MaxAge          string
    AllowCredentials bool
}

func CORS(cfg CORSConfig) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            origin := r.Header.Get("Origin")
            if origin == "" {
                next.ServeHTTP(w, r)
                return
            }

            if _, ok := cfg.AllowedOrigins[origin]; !ok {
                http.Error(w, "CORS origin is not allowed", http.StatusForbidden)
                return
            }

            h := w.Header()
            h.Set("Access-Control-Allow-Origin", origin)
            h.Add("Vary", "Origin")

            if cfg.AllowCredentials {
                h.Set("Access-Control-Allow-Credentials", "true")
            }

            if r.Method == http.MethodOptions &&
                r.Header.Get("Access-Control-Request-Method") != "" {
                h.Set("Access-Control-Allow-Methods", cfg.AllowedMethods)
                h.Set("Access-Control-Allow-Headers", cfg.AllowedHeaders)
                h.Set("Access-Control-Max-Age", cfg.MaxAge)
                h.Add("Vary", "Access-Control-Request-Method")
                h.Add("Vary", "Access-Control-Request-Headers")
                w.WriteHeader(http.StatusNoContent)
                return
            }

            next.ServeHTTP(w, r)
        })
    }
}
```

В примере сервер возвращает не `*`, а конкретный разрешённый origin. Это важно для API, которые работают с cookies или `fetch(..., { credentials: "include" })`: при `Access-Control-Allow-Credentials: true` браузер не принимает `Access-Control-Allow-Origin: *`.

`Vary: Origin` нужен, потому что значение `Access-Control-Allow-Origin` выбирается динамически. Без `Vary` промежуточный кэш может сохранить ответ для одного origin и отдать его другому origin с неправильными CORS-заголовками. Для preflight по той же причине добавляются `Vary: Access-Control-Request-Method` и `Vary: Access-Control-Request-Headers`.

::: info
Если JavaScript-коду нужно читать нестандартные response headers, например `X-Request-ID` или `X-Total-Count`, сервер должен явно открыть их через `Access-Control-Expose-Headers`.
:::
