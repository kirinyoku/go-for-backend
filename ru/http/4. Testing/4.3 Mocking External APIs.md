# Мокирование внешних API

Если код приложения обращается к внешнему HTTP API, такие тесты не должны зависеть от реального внешнего сервера. Вместо этого поднимается локальный тестовый сервер, который ведёт себя как внешний API и возвращает заранее заданные ответы.

::: info
В этой статье под «мокированием внешнего API» понимается не подмена функции-заглушки, а запуск контролируемого HTTP-сервера в тесте. Такой подход проверяет реальное формирование HTTP-запроса: метод, путь, query-параметры, заголовки, тело запроса, обработку статусов и декодирование ответа.
:::

## Почему реальные API не подходят для тестов

Обращение к настоящему внешнему API делает тесты нестабильными и дорогими в сопровождении:

- сеть может быть недоступна или работать медленно;
- внешний сервис может изменить данные, лимиты или формат ответа;
- тесты начинают зависеть от API-ключей и секретов;
- негативные сценарии сложно воспроизвести детерминированно;
- запуск тестов может расходовать квоты или создавать реальные операции.

Тест клиентского кода должен проверять не то, что внешний сервис работает корректно, а то, что приложение правильно взаимодействует с ожидаемым HTTP-контрактом.

::: warning
Реальные API-ключи не должны использоваться в unit-тестах. Если тесту требуется проверить заголовок авторизации, достаточно фиктивного значения вроде `test-api-key` или `secret-key`.
:::

## Точка подмены: базовый URL и HTTP-клиент

Код трудно тестировать, если URL внешнего API зашит прямо внутри метода:

```go
func CurrentWeather(city string) error {
	resp, err := http.Get("https://api.weather.example/v1/weather/current?city=" + city)
	if err != nil {
		return err
	}
	defer resp.Body.Close()

	return nil
}
```

У такого кода сразу несколько проблем:

- базовый URL нельзя заменить на адрес тестового сервера;
- используется [`http.DefaultClient`](https://pkg.go.dev/net/http#DefaultClient), у которого нет таймаута по умолчанию;
- невозможно проверить запрос без обращения к реальной сети;
- сложнее симулировать сетевые ошибки и отмену контекста.

Идиоматичный вариант — хранить базовый URL и HTTP-клиент в структуре зависимости. В production-коде передаётся настоящий URL, а в тестах — [`ts.URL`](https://pkg.go.dev/net/http/httptest#Server.URL) от `httptest.NewServer`.

```go
// weather.go
package weather

import (
	"context"
	"encoding/json"
	"fmt"
	"net/http"
	"net/url"
	"path"
)

type Client struct {
	BaseURL    string
	APIKey     string
	HTTPClient *http.Client
}

type CurrentWeather struct {
	City  string  `json:"city"`
	TempC float64 `json:"temp_c"`
}

func (c *Client) CurrentWeather(ctx context.Context, city string) (CurrentWeather, error) {
	endpoint, err := url.Parse(c.BaseURL)
	if err != nil {
		return CurrentWeather{}, fmt.Errorf("parse base URL: %w", err)
	}

	// path.Join сохраняет возможный префикс базового URL, например /v1.
	// Если API различает /path и /path/, учитывайте, что path.Join чистит путь
	// и убирает завершающий slash.
	endpoint.Path = path.Join(endpoint.Path, "/weather/current")

	query := endpoint.Query()
	query.Set("city", city)
	endpoint.RawQuery = query.Encode()

	req, err := http.NewRequestWithContext(ctx, http.MethodGet, endpoint.String(), nil)
	if err != nil {
		return CurrentWeather{}, fmt.Errorf("create weather request: %w", err)
	}

	req.Header.Set("Accept", "application/json")
	if c.APIKey != "" {
		req.Header.Set("X-API-Key", c.APIKey)
	}

	httpClient := c.HTTPClient
	if httpClient == nil {
		httpClient = http.DefaultClient
	}

	resp, err := httpClient.Do(req)
	if err != nil {
		return CurrentWeather{}, fmt.Errorf("send weather request: %w", err)
	}
	defer resp.Body.Close()

	if resp.StatusCode != http.StatusOK {
		return CurrentWeather{}, fmt.Errorf("weather API status: %s", resp.Status)
	}

	var weather CurrentWeather
	if err := json.NewDecoder(resp.Body).Decode(&weather); err != nil {
		return CurrentWeather{}, fmt.Errorf("decode weather response: %w", err)
	}

	return weather, nil
}
```

В этой реализации есть две важные точки контроля:

- `BaseURL` позволяет перенаправить запросы на тестовый сервер;
- `HTTPClient` позволяет задать таймауты, TLS-настройки или подменить транспорт в тестах.

## Базовый мок через httptest.NewServer

`httptest.NewServer` запускает локальный HTTP-сервер на случайном свободном порту. Его адрес доступен через поле `ts.URL`. Именно этот адрес передаётся в тестируемый клиент вместо настоящего внешнего API.

Helper для запуска сервера использует тот же паттерн `t.Helper` + `t.Cleanup`, который подробно разобран в статье [Тестирование с использованием серверов](./testing-with-servers).

```go
// weather_test.go
package weather

import (
	"context"
	"fmt"
	"net/http"
	"net/http/httptest"
	"testing"
)

func newWeatherAPIServer(t testing.TB, handler http.Handler) *httptest.Server {
	t.Helper()

	ts := httptest.NewServer(handler)
	t.Cleanup(ts.Close)

	return ts
}

func TestClient_CurrentWeather(t *testing.T) {
	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if r.Method != http.MethodGet {
			http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
			return
		}
		if r.URL.Path != "/weather/current" {
			http.NotFound(w, r)
			return
		}
		if got := r.URL.Query().Get("city"); got != "London" {
			http.Error(w, "city is required", http.StatusBadRequest)
			return
		}
		if got := r.Header.Get("X-API-Key"); got != "secret-key" {
			http.Error(w, "invalid API key", http.StatusUnauthorized)
			return
		}

		w.Header().Set("Content-Type", "application/json")
		fmt.Fprintln(w, `{"city":"London","temp_c":15.5}`)
	}))

	client := &Client{
		BaseURL:    ts.URL,
		APIKey:     "secret-key",
		HTTPClient: ts.Client(),
	}

	got, err := client.CurrentWeather(context.Background(), "London")
	if err != nil {
		t.Fatalf("current weather: %v", err)
	}

	want := CurrentWeather{City: "London", TempC: 15.5}
	if got != want {
		t.Errorf("weather: got %+v, want %+v", got, want)
	}
}
```

Мок-сервер здесь выполняет сразу две задачи:

- проверяет, что клиент отправил правильный HTTP-запрос;
- возвращает контролируемый JSON-ответ для проверки результата.

Такой тест ближе к реальному поведению, чем подмена одной функции, потому что запрос проходит через стандартный HTTP-клиент, сериализацию URL, заголовки и декодирование ответа.

::: warning
`httptest.Server` обрабатывает запросы в отдельных горутинах. Внутри обработчика не следует использовать [`t.Fatal`](https://pkg.go.dev/testing#T.Fatal) или [`t.Fatalf`](https://pkg.go.dev/testing#T.Fatalf): `FailNow` должен вызываться из горутины самого теста, а не из горутин, созданных во время теста. Для ошибок контракта лучше вернуть HTTP-ошибку через [`http.Error`](https://pkg.go.dev/net/http#Error), а итоговую ошибку проверить в основном теле теста.
:::

## Проверка исходящего запроса

Мок внешнего API должен проверять не только итоговый результат, но и сам исходящий запрос. Обычно важны следующие элементы:

| Часть запроса       | Что проверяется                                      |
| :------------------ | :--------------------------------------------------- |
| HTTP-метод          | `GET`, `POST`, `PUT`, `DELETE`                       |
| Путь                | `/weather/current`, `/v1/users`, `/bot/sendMessage`  |
| Query-параметры     | `city=London`, `limit=10`, `page=2`                  |
| Заголовки           | `Accept`, `Content-Type`, `Authorization`, API-ключи |
| Тело запроса        | JSON, form-data, payload для внешнего API            |
| Количество запросов | Повторные попытки, обновление токена, пагинация      |

Для `GET`-запросов чаще всего достаточно проверить метод, путь, query-параметры и заголовки. Для `POST`-запросов дополнительно декодируется тело запроса. Значение `Content-Type` лучше разбирать через [`mime.ParseMediaType`](https://pkg.go.dev/mime#ParseMediaType), чтобы корректно принять заголовки с параметрами вроде `application/json; charset=utf-8`.

```go
func TestSendAlert_SendsJSONBody(t *testing.T) {
	type createAlertRequest struct {
		City    string `json:"city"`
		Message string `json:"message"`
	}

	sendAlert := func(ctx context.Context, client *http.Client, endpoint string, alert createAlertRequest) error {
		var body bytes.Buffer
		if err := json.NewEncoder(&body).Encode(alert); err != nil {
			return fmt.Errorf("encode alert request: %w", err)
		}

		req, err := http.NewRequestWithContext(ctx, http.MethodPost, endpoint+"/alerts", &body)
		if err != nil {
			return fmt.Errorf("create alert request: %w", err)
		}
		req.Header.Set("Content-Type", "application/json")

		resp, err := client.Do(req)
		if err != nil {
			return fmt.Errorf("send alert request: %w", err)
		}
		defer resp.Body.Close()

		if resp.StatusCode != http.StatusCreated {
			return fmt.Errorf("alert API status: %s", resp.Status)
		}

		return nil
	}

	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if r.Method != http.MethodPost {
			http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
			return
		}
		if r.URL.Path != "/alerts" {
			http.NotFound(w, r)
			return
		}
		contentType, _, err := mime.ParseMediaType(r.Header.Get("Content-Type"))
		if err != nil {
			http.Error(w, "invalid content type", http.StatusUnsupportedMediaType)
			return
		}
		if contentType != "application/json" {
			http.Error(w, "content type must be application/json", http.StatusUnsupportedMediaType)
			return
		}

		var got createAlertRequest
		if err := json.NewDecoder(r.Body).Decode(&got); err != nil {
			http.Error(w, "invalid JSON body", http.StatusBadRequest)
			return
		}

		want := createAlertRequest{City: "London", Message: "Rain expected"}
		if got != want {
			http.Error(w, "unexpected request body", http.StatusBadRequest)
			return
		}

		w.WriteHeader(http.StatusCreated)
	}))

	alert := createAlertRequest{City: "London", Message: "Rain expected"}
	if err := sendAlert(context.Background(), ts.Client(), ts.URL, alert); err != nil {
		t.Fatalf("send alert: %v", err)
	}
}
```

Пример показывает сам паттерн проверки тела: мок-обработчик декодирует входящий JSON и возвращает ошибочный статус, если запрос не соответствует ожидаемому контракту. После вызова клиентской функции основной тест проверяет, что ошибка отсутствует.

::: info
При сравнении JSON-тел исходящих запросов удобнее декодировать JSON в структуру, а не сравнивать строки. Порядок полей в JSON не должен влиять на результат теста.
:::

## Табличные тесты для ответов внешнего API

Один клиентский метод обычно должен корректно обрабатывать несколько сценариев:

- успешный ответ `200 OK`;
- ошибочный статус `400 Bad Request`, `401 Unauthorized`, `500 Internal Server Error`;
- пустое тело ответа;
- некорректный JSON;
- неожиданный формат данных.

Для таких случаев подходит тот же паттерн табличных тестов, что и при тестировании обработчиков.

```go
func TestClient_CurrentWeather_UpstreamErrors(t *testing.T) {
	tests := []struct {
		name       string
		statusCode int
		body       string
		wantErr    string
	}{
		{
			name:       "server error",
			statusCode: http.StatusInternalServerError,
			body:       `{"error":"temporary failure"}`,
			wantErr:    "weather API status",
		},
		{
			name:       "invalid JSON",
			statusCode: http.StatusOK,
			body:       `{"city":"London","temp_c":`,
			wantErr:    "decode weather response",
		},
		{
			name:       "empty body",
			statusCode: http.StatusOK,
			body:       ``,
			wantErr:    "decode weather response",
		},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
				w.Header().Set("Content-Type", "application/json")
				w.WriteHeader(tc.statusCode)
				fmt.Fprint(w, tc.body)
			}))

			client := &Client{
				BaseURL:    ts.URL,
				HTTPClient: ts.Client(),
			}

			_, err := client.CurrentWeather(context.Background(), "London")
			if err == nil {
				t.Fatal("error: got nil, want non-nil")
			}
			if !strings.Contains(err.Error(), tc.wantErr) {
				t.Errorf("error: got %q, want substring %q", err.Error(), tc.wantErr)
			}
		})
	}
}
```

Здесь каждый подтест поднимает собственный мок-сервер. Это делает сценарии независимыми: изменение ответа в одном случае не влияет на остальные.

## Симуляция таймаутов и отмены контекста

Внешние API могут отвечать медленно или не отвечать вовсе. Проверка таких сценариев особенно важна для кода, который использует `context.Context`.

```go
func TestClient_CurrentWeather_ContextTimeout(t *testing.T) {
	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// Задержка больше таймаута контекста, чтобы запрос был отменён клиентом.
		time.Sleep(100 * time.Millisecond)
		w.WriteHeader(http.StatusOK)
	}))

	client := &Client{
		BaseURL:    ts.URL,
		HTTPClient: ts.Client(),
	}

	ctx, cancel := context.WithTimeout(context.Background(), 10*time.Millisecond)
	defer cancel()

	_, err := client.CurrentWeather(ctx, "London")
	if err == nil {
		t.Fatal("error: got nil, want timeout error")
	}
	if !errors.Is(err, context.DeadlineExceeded) {
		t.Errorf("error: got %v, want context deadline exceeded", err)
	}
}
```

Такой тест проверяет, что метод действительно передаёт контекст в [`http.NewRequestWithContext`](https://pkg.go.dev/net/http#NewRequestWithContext). Если вместо этого используется [`http.Get`](https://pkg.go.dev/net/http#Get), контекст теста не будет связан с запросом, потому что `http.Get` не принимает `ctx`.

::: tip
При тестировании таймаутов задержка сервера должна заметно превышать таймаут контекста. Слишком близкие значения делают тест чувствительным к нагрузке на машину и могут приводить к случайным падениям.
:::

## Stateful mocks для последовательных ответов

Некоторые сценарии зависят от порядка запросов. Например, внешний API сначала возвращает `500 Internal Server Error`, а при повторной попытке — `200 OK`. Для проверки такой логики мок-сервер может хранить состояние между запросами.

```go
func TestFetchWithRetry(t *testing.T) {
	var attempts atomic.Int32

	ts := newWeatherAPIServer(t, http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if attempts.Add(1) == 1 {
			http.Error(w, "temporary failure", http.StatusInternalServerError)
			return
		}

		w.WriteHeader(http.StatusOK)
	}))

	err := fetchWithRetry(context.Background(), ts.Client(), ts.URL, 2)
	if err != nil {
		t.Fatalf("fetch with retry: %v", err)
	}

	if got, want := attempts.Load(), int32(2); got != want {
		t.Errorf("attempts: got %d, want %d", got, want)
	}
}

func fetchWithRetry(ctx context.Context, client *http.Client, endpoint string, maxAttempts int) error {
	if maxAttempts < 1 {
		return errors.New("max attempts must be positive")
	}

	var lastErr error

	for attempt := 0; attempt < maxAttempts; attempt++ {
		req, err := http.NewRequestWithContext(ctx, http.MethodGet, endpoint, nil)
		if err != nil {
			return fmt.Errorf("create request: %w", err)
		}

		resp, err := client.Do(req)
		if err != nil {
			lastErr = err
			continue
		}

		_, copyErr := io.Copy(io.Discard, resp.Body)
		closeErr := resp.Body.Close()
		if copyErr != nil {
			return fmt.Errorf("read response body: %w", copyErr)
		}
		if closeErr != nil {
			return fmt.Errorf("close response body: %w", closeErr)
		}

		if resp.StatusCode == http.StatusOK {
			return nil
		}

		lastErr = fmt.Errorf("status: %s", resp.Status)
	}

	return fmt.Errorf("all attempts failed: %w", lastErr)
}
```

Для счётчика используется [`atomic.Int32`](https://pkg.go.dev/sync/atomic#Int32), потому что HTTP-сервер может обрабатывать запросы в разных горутинах. Если тест строго последовательный и параллелизм невозможен, обычной переменной иногда достаточно, но атомарный счётчик делает пример безопаснее.

## Подмена транспорта без запуска сервера

`httptest.NewServer` — основной инструмент для мокирования внешних HTTP API. Однако иногда требуется проверить ошибку, которая возникает до получения HTTP-ответа: например, сбой соединения, ошибка DNS или ошибка транспорта. Для таких случаев можно подменить [`http.RoundTripper`](https://pkg.go.dev/net/http#RoundTripper), который используется внутри [`http.Client`](https://pkg.go.dev/net/http#Client) для выполнения одного HTTP-запроса.

```go
type roundTripFunc func(req *http.Request) (*http.Response, error)

func (f roundTripFunc) RoundTrip(req *http.Request) (*http.Response, error) {
	return f(req)
}

func TestClient_CurrentWeather_NetworkError(t *testing.T) {
	httpClient := &http.Client{
		Transport: roundTripFunc(func(req *http.Request) (*http.Response, error) {
			return nil, errors.New("connection refused")
		}),
	}

	client := &Client{
		BaseURL:    "https://weather.example.test",
		HTTPClient: httpClient,
	}

	_, err := client.CurrentWeather(context.Background(), "London")
	if err == nil {
		t.Fatal("error: got nil, want non-nil")
	}
	if !strings.Contains(err.Error(), "send weather request") {
		t.Errorf("error: got %q, want transport error", err.Error())
	}
}
```

Такой тест вообще не открывает сетевое соединение. Клиент формирует запрос как обычно, но вместо настоящего транспорта вызывается функция `roundTripFunc`.

Если fake-транспорт возвращает успешный [`*http.Response`](https://pkg.go.dev/net/http#Response), поле [`Body`](https://pkg.go.dev/net/http#Response.Body) лучше задавать явно. `http.Client` может подставить пустое тело при `nil` и нулевой длине ответа, но явный `io.NopCloser` точнее имитирует реальный ответ и обязателен, если тело непустое.

```go
httpClient := &http.Client{
	Transport: roundTripFunc(func(req *http.Request) (*http.Response, error) {
		return &http.Response{
			StatusCode: http.StatusOK,
			Status:     "200 OK",
			Header:     make(http.Header),
			Body:       io.NopCloser(strings.NewReader(`{"city":"London","temp_c":15.5}`)),
			Request:    req,
		}, nil
	}),
}
```

::: info
Подмена `RoundTripper` полезна для низкоуровневых ошибок клиента. Для проверки HTTP-контракта внешнего API — путей, заголовков, query-параметров и тела запроса лучше использовать `httptest.NewServer`.
:::

## Интерфейс Doer

Ещё один вариант — принимать не `*http.Client`, а небольшой интерфейс с методом `Do`. Такой подход удобен, когда требуется полностью отделить код от конкретной реализации HTTP-клиента.

```go
type Doer interface {
	Do(req *http.Request) (*http.Response, error)
}

type Client struct {
	BaseURL    string
	APIKey     string
	HTTPClient Doer
}
```

Интерфейс должен объявляться на стороне потребителя, а не на стороне внешней зависимости. Это сохраняет минимальный контракт: клиентскому коду нужен только метод `Do`, поэтому интерфейс не должен включать ничего лишнего.

В тесте можно передать простую структуру-заглушку:

```go
type mockDoer struct {
	doFunc func(req *http.Request) (*http.Response, error)
}

func (m mockDoer) Do(req *http.Request) (*http.Response, error) {
	return m.doFunc(req)
}
```

Однако для HTTP-клиентов этот подход не всегда лучше `httptest.NewServer`. Интерфейс `Doer` проверяет взаимодействие на уровне вызова метода `Do`, но не проверяет реальный сетевой путь запроса через HTTP-сервер. Поэтому для большинства тестов внешних API предпочтительнее начинать с `httptest.NewServer`, а интерфейс или fake-транспорт использовать точечно.

## Как выбрать подход

| Подход                  | Когда использовать                                                              |
| :---------------------- | :------------------------------------------------------------------------------ |
| `httptest.NewServer`    | Основной вариант для проверки клиента внешнего HTTP API                         |
| `httptest.NewTLSServer` | Проверка HTTPS, TLS-соединения и клиента, доверяющего тестовому сертификату     |
| Подмена `RoundTripper`  | Симуляция сетевых ошибок без запуска сервера                                    |
| Интерфейс `Doer`        | Полная подмена механизма выполнения запроса через небольшой потребительский API |
| Реальный внешний API    | Только smoke-тесты или отдельные интеграционные тесты, не обычный unit-набор    |

**Главное правило:** тест должен контролировать все входные данные и ответы внешней зависимости. Если результат зависит от состояния реального сервера, сети или API-ключа, такой тест не является детерминированным.
