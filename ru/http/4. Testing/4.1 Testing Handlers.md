# Тестирование обработчиков

Тестирование HTTP-обработчиков относится к категории unit-тестов: проверяется логика конкретной функции в изоляции, без реального сетевого соединения и запуска сервера. Основным инструментом для этого в стандартной библиотеке Go служит пакет [`net/http/httptest`](https://pkg.go.dev/net/http/httptest).

::: info 
Когда требуется проверить поведение всего HTTP-стека — работу клиента, TLS-соединение или обработку редиректов — используется [`httptest.NewServer`](https://pkg.go.dev/net/http/httptest#NewServer). Этот подход рассматривается в [Тестирование с использованием серверов](./testing-with-servers). 
:::

## Запись ответа через ResponseRecorder

Структура [`httptest.ResponseRecorder`](https://pkg.go.dev/net/http/httptest#ResponseRecorder) реализует интерфейс [`http.ResponseWriter`](https://pkg.go.dev/net/http#ResponseWriter). Вместо отправки данных по сети она записывает ответ во внутренний буфер, доступный для последующей проверки.

| Метод / Поле                                                               | Назначение                                                                                          |
| :------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------- |
| [`Code`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Code)       | Записанный HTTP статус-код. Если обработчик ничего не записал, поле остается `0`.                   |
| [`Header()`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Header) | Карта заголовков ответа.                                                                            |
| [`Body`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Body)       | Буфер ([`*bytes.Buffer`](https://pkg.go.dev/bytes#Buffer)) с телом ответа.                          |
| [`Result()`](https://pkg.go.dev/net/http/httptest#ResponseRecorder.Result) | Возвращает [`*http.Response`](https://pkg.go.dev/net/http#Response) для проверки  cookie и trailer. |

::: info
Метод `Result()` возвращает ответ с неявным статусом `200 OK`, если обработчик не вызвал `WriteHeader` и не записал тело. Поле `rr.Code` при этом остается равным `0`, поэтому для проверки «реального» результата HTTP-ответа используйте `rr.Result().StatusCode`.
:::

## Структура теста: Arrange, Act, Assert

Стандартная структура теста обработчика состоит из трёх шагов:

- **Arrange** — подготовка: создание запроса и `ResponseRecorder`.
- **Act** — действие: прямой вызов обработчика.
- **Assert** — проверка: сравнение фактического ответа с ожидаемым.

```go
// handlers_test.go
package main

import (
	"net/http"
	"net/http/httptest"
	"testing"
)

func TestHealthCheckHandler(t *testing.T) {
	// Arrange: создание фиктивного запроса в памяти и инструмента записи ответа.
	req := httptest.NewRequest(http.MethodGet, "/health", nil)
	rr := httptest.NewRecorder()

	// Act: прямой вызов обработчика — сетевое соединение не нужно.
	HealthCheckHandler(rr, req)

	// Assert: проверка статус-кода и тела ответа.
	if rr.Code != http.StatusOK {
		t.Errorf("status: got %d, want %d", rr.Code, http.StatusOK)
	}
	if rr.Body.String() != "OK" {
		t.Errorf("body: got %q, want %q", rr.Body.String(), "OK")
	}
}
```

### t.Errorf и t.Fatalf

В тестах обработчиков применяются две функции фиксации ошибок:

- [`t.Errorf`](https://pkg.go.dev/testing#T.Errorf) — отмечает тест как неуспешный и **продолжает** выполнение. Используется, когда проверки независимы друг от друга.
- [`t.Fatalf`](https://pkg.go.dev/testing#T.Fatalf) — отмечает тест как неуспешный и **немедленно останавливает** его. Используется, когда при неуспешной проверке дальнейшие шаги потеряют смысл или вызовут панику.

**Типичный пример:** если [`Decode`](https://pkg.go.dev/encoding/json#Decoder.Decode) вернул ошибку, обращаться к полям декодированной структуры бессмысленно — здесь необходим `t.Fatalf`.

## Тестирование JSON-ответов

Сравнивать JSON-ответы как строки не рекомендуется: порядок полей в JSON не гарантирован. Идиоматичный подход — десериализовать ответ в структуру и сравнить значения полей. Для проверки `Content-Type` используется [`mime.ParseMediaType`](https://pkg.go.dev/mime#ParseMediaType), чтобы корректно обработать значения вроде `application/json; charset=utf-8`.

```go
func TestUserHandler(t *testing.T) {
	// Arrange
	req := httptest.NewRequest(http.MethodGet, "/user", nil)
	rr := httptest.NewRecorder()

	// Act
	UserHandler(rr, req)

	// Assert: статус проверяется первым через Fatalf — при ошибке декодирование
	// дальнейших данных потеряет смысл.
	if rr.Code != http.StatusOK {
		t.Fatalf("status: got %d, want %d", rr.Code, http.StatusOK)
	}

	gotContentType, _, err := mime.ParseMediaType(rr.Header().Get("Content-Type"))
	if err != nil {
		t.Fatalf("content-type: parse: %v", err)
	}
	if gotContentType != "application/json" {
		t.Errorf("content-type: got %q, want %q", gotContentType, "application/json")
	}

	var got User
	if err := json.NewDecoder(rr.Body).Decode(&got); err != nil {
		t.Fatalf("failed to decode response: %v", err)
	}

	want := User{ID: 1, Username: "gopher"}
	if got != want {
		t.Errorf("response: got %+v, want %+v", got, want)
	}
}
```

::: info 
Для сравнения сложных структур (со срезами, вложенными полями или значением `nil`) удобен пакет [`github.com/google/go-cmp/cmp`](https://pkg.go.dev/github.com/google/go-cmp/cmp). Вместо `t.Errorf("got %+v, want %+v", got, want)` можно написать `t.Errorf("mismatch (-want +got):\n%s", cmp.Diff(want, got))` — вывод покажет только различия, а не два значения целиком. 
:::

## Вспомогательные функции

При написании многих тестов повторяющиеся проверки принято выносить в отдельные функции. Для их корректной работы используются два инструмента.

[`t.Helper()`](https://pkg.go.dev/testing#T.Helper) сообщает тестовому фреймворку, что данная функция является вспомогательной. При неуспехе трассировка будет указывать на вызывающий тест, а не на строку внутри хелпера.

[`testing.TB`](https://pkg.go.dev/testing#TB) — интерфейс, объединяющий [`*testing.T`](https://pkg.go.dev/testing#T) и [`*testing.B`](https://pkg.go.dev/testing#B). Хелперы, принимающие `testing.TB`, без изменений работают как в тестах, так и в бенчмарках.

```go
// assertStatus проверяет HTTP статус-код ответа.
// Принимает testing.TB вместо *testing.T, чтобы хелпер был пригоден и для бенчмарков.
func assertStatus(tb testing.TB, rr *httptest.ResponseRecorder, want int) {
	tb.Helper() // При ошибке трассировка укажет на вызывающий тест, а не на эту строку.
	if rr.Code != want {
		tb.Errorf("status: got %d, want %d", rr.Code, want)
	}
}

// assertBody проверяет тело ответа.
func assertBody(tb testing.TB, rr *httptest.ResponseRecorder, want string) {
	tb.Helper()
	if got := rr.Body.String(); got != want {
		tb.Errorf("body: got %q, want %q", got, want)
	}
}
```

Использование хелперов сокращает тело теста до минимума:

```go
func TestHealthCheckHandler(t *testing.T) {
	req := httptest.NewRequest(http.MethodGet, "/health", nil)
	rr := httptest.NewRecorder()

	HealthCheckHandler(rr, req)

	assertStatus(t, rr, http.StatusOK)
	assertBody(t, rr, "OK")
}
```

## Табличные тесты

Когда один обработчик требует проверки нескольких сценариев, дублировать тест-функции не нужно. Идиоматичный паттерн в Go — табличные тесты (table-driven tests): все случаи описываются в одном срезе структур, а логика выполняется в цикле через [`t.Run`](https://pkg.go.dev/testing#T.Run).

```go
func TestGreetHandler(t *testing.T) {
	tests := []struct {
		name       string
		url        string
		wantStatus int
		wantBody   string
	}{
		{
			name:       "known user",
			url:        "/greet?name=gopher",
			wantStatus: http.StatusOK,
			wantBody:   "Hello, gopher!",
		},
		{
			name:       "missing name",
			url:        "/greet",
			wantStatus: http.StatusBadRequest,
			wantBody:   "name is required\n",
		},
	}

	for _, tc := range tests {
		// t.Run создаёт изолированный подтест. Конкретный случай запускается командой:
		// go test -run TestGreetHandler/known_user
		t.Run(tc.name, func(t *testing.T) {
			req := httptest.NewRequest(http.MethodGet, tc.url, nil)
			rr := httptest.NewRecorder()

			GreetHandler(rr, req)

			assertStatus(t, rr, tc.wantStatus)
			assertBody(t, rr, tc.wantBody)
		})
	}
}
```

::: tip 
Подтесты можно запускать параллельно, добавив [`t.Parallel()`](https://pkg.go.dev/testing#T.Parallel) в начало тела `t.Run`. Это безопасно, если каждый подтест использует собственные `ResponseRecorder` и запрос, а сам обработчик и его зависимости не изменяют общее состояние без синхронизации. 
:::

## Тестирование middleware

Middleware тестируется изолированно: в качестве следующего обработчика передаётся минимальная функция-заглушка, которая лишь фиксирует факт передачи управления. Табличные тесты здесь особенно уместны — позволяют компактно покрыть все ветви логики.

```go
func TestAuthMiddleware(t *testing.T) {
	// nextHandler — заглушка, подтверждающая, что middleware передало управление дальше.
	// Определяется один раз и используется во всех подтестах — она не хранит состояния.
	nextHandler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
	})

	tests := []struct {
		name       string
		authHeader string
		wantStatus int
	}{
		{name: "missing token", authHeader: "", wantStatus: http.StatusUnauthorized},
		{name: "present token", authHeader: "Bearer secret", wantStatus: http.StatusOK},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			t.Parallel()

			req := httptest.NewRequest(http.MethodGet, "/", nil)
			if tc.authHeader != "" {
				req.Header.Set("Authorization", tc.authHeader)
			}
			rr := httptest.NewRecorder()

			AuthMiddleware(nextHandler).ServeHTTP(rr, req)

			assertStatus(t, rr, tc.wantStatus)
		})
	}
}
```

## Тестирование обработчиков с контекстом

Использование строковых литералов в качестве ключей контекста приводит к коллизиям: разные пакеты могут случайно использовать одинаковые строки. Стандартная идиома Go — объявлять приватный тип-ключ на уровне пакета.

```go
// contextKey — приватный тип для ключей контекста данного пакета.
// Значения типа contextKey не пересекаются со значениями типа string,
// даже если строковое содержимое совпадает.
type contextKey string

const userIDKey contextKey = "user_id"
```

Тест для обработчика, работающего с контекстом:

```go
func TestContextHandler(t *testing.T) {
	handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// Конкретный тип утверждается через type assertion — отсутствие ключа
		// или неверный тип возвращают ok == false, а не панику.
		id, ok := r.Context().Value(userIDKey).(int)
		if !ok {
			http.Error(w, "user_id not found in context", http.StatusInternalServerError)
			return
		}
		fmt.Fprintf(w, "%d", id)
	})

	req := httptest.NewRequest(http.MethodGet, "/", nil)
	// Внедрение значения в контекст запроса через типизированный ключ.
	ctx := context.WithValue(req.Context(), userIDKey, 42)
	req = req.WithContext(ctx)

	rr := httptest.NewRecorder()
	handler.ServeHTTP(rr, req)

	assertStatus(t, rr, http.StatusOK)
	assertBody(t, rr, "42")
}
```

## Проверка cookie

Метод `Header()` у `ResponseRecorder` возвращает карту заголовков `http.Header`, где `Set-Cookie` доступен как обычное значение заголовка. Для получения распарсенных cookie удобнее использовать метод `Result()`, возвращающий полноценный `*http.Response`.

```go
func TestCookieHandler(t *testing.T) {
	handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		http.SetCookie(w, &http.Cookie{Name: "session", Value: "abc123"})
		w.WriteHeader(http.StatusOK)
	})

	req := httptest.NewRequest(http.MethodGet, "/", nil)
	rr := httptest.NewRecorder()
	handler.ServeHTTP(rr, req)

	assertStatus(t, rr, http.StatusOK)

	// Result() возвращает *http.Response с распарсенными cookie.
	// Body оборачивается в io.NopCloser — Close является no-op,
	// но закрывать его всё равно идиоматично.
	resp := rr.Result()
	defer resp.Body.Close()

	cookies := resp.Cookies()
	if len(cookies) == 0 {
		t.Fatal("cookies: got none, want at least one")
	}
	if got, want := cookies[0].Name, "session"; got != want {
		t.Errorf("cookie name: got %q, want %q", got, want)
	}
}
```
