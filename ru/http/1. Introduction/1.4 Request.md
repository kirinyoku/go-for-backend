# Структура http.Request

Структура [`http.Request`](https://pkg.go.dev/net/http#Request) содержит данные HTTP-запроса, с которыми работает обработчик. Эта же структура используется и на стороне клиента для исходящих запросов, но в этой статье речь идет прежде всего о входящем запросе на сервере. Создание и отправка клиентских запросов разбираются отдельно в статье [Формирование и отправка запросов](/ru/net-http/client/making-requests).

## Ключевые поля

Поля `http.Request` предоставляют доступ к метаданным, заголовкам и содержимому запроса.

| Поле            | Тип                                                                     | Назначение                                                    |
| :-------------- | :---------------------------------------------------------------------- | :------------------------------------------------------------ |
| `Method`        | `string`                                                                | HTTP-метод (GET, POST, PUT и т.д.)                            |
| `URL`           | [`*url.URL`](https://pkg.go.dev/net/url#URL)                            | Разобранный URL со всеми компонентами (путь, query-параметры) |
| `Proto`         | `string`                                                                | Версия протокола ("HTTP/1.1", "HTTP/2")                       |
| `Header`        | [`http.Header`](https://pkg.go.dev/net/http#Header)                     | Карта заголовков запроса                                      |
| `Body`          | [`io.ReadCloser`](https://pkg.go.dev/io#ReadCloser)                     | Тело запроса, представленное как поток данных                 |
| `Host`          | `string`                                                                | Значение заголовка Host или имя хоста из URL                  |
| `ContentLength` | `int64`                                                                 | Длина тела в байтах (-1, если длина неизвестна)               |
| `RemoteAddr`    | `string`                                                                | Сетевой адрес отправителя (IP:port), заполняется сервером     |
| `TLS`           | [`*tls.ConnectionState`](https://pkg.go.dev/crypto/tls#ConnectionState) | Параметры TLS-соединения для HTTPS-запросов                   |

## Работа с телом запроса

Поле `Body` имеет тип `io.ReadCloser` и представляет тело запроса как поток данных. Потоковая модель определяет две ключевые особенности:

1. **Однократное чтение:** После полного вычитывания данных поток считается исчерпанным. Повторные операции чтения возвращают [`io.EOF`](https://pkg.go.dev/io#EOF).
2. **Эффективность памяти:** Потоковая модель позволяет обрабатывать данные любого объема без их полной загрузки в оперативную память.

::: info
На стороне сервера `net/http` предоставляет ненулевой `Body`, который для запросов без тела сразу возвращает `io.EOF`. На стороне клиента при создании запроса `Body` может быть `nil`, если тело не передавалось.
:::

### Повторное чтение тела

Для многократного использования тела запроса (например, при логировании в middleware) данные необходимо сохранить и заменить `Body` новым потоком. Это технический прием, а не обычный путь обработки данных: JSON, формы, multipart-загрузки и лимиты размера подробно рассматриваются в статье [Данные запроса и ответа](/ru/net-http/server/working-with-data).

```go
func loggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // Сохраняем указатель на оригинальный сетевой поток.
        originalBody := r.Body
        
        // Регистрируем отложенное закрытие оригинального потока.
        // Вызов сработает именно для оригинального потока, несмотря на последующую замену r.Body.
        defer originalBody.Close()

        // Чтение тела в память.
        bodyBytes, err := io.ReadAll(originalBody)
        if err != nil {
            http.Error(w, "Internal Server Error", http.StatusInternalServerError)
            return
        }

        log.Printf("request body: %s", bodyBytes)

        // Создание нового потока из сохраненных байт.
        // io.NopCloser добавляет метод Close(), который не выполняет действий,
        // обеспечивая совместимость с интерфейсом io.ReadCloser.
        r.Body = io.NopCloser(bytes.NewReader(bodyBytes))

        next.ServeHTTP(w, r)
    })
}
```

::: warning
В реальном middleware такой подход требует лимита на размер читаемых данных. Иначе логирование большого тела запроса может само стать источником чрезмерного потребления памяти.
:::

::: info
На стороне сервера `r.Body` закрывается автоматически после завершения работы обработчика.
:::

## Заголовки и cookie

Тип `http.Header` представляет собой `map[string][]string`. Методы этого типа выполняют канонизацию ключей (приведение к виду `Canonical-Key-Name`) при чтении и записи заголовков.

### Управление заголовками

Методы [`Get()`](https://pkg.go.dev/net/http#Header.Get), [`Set()`](https://pkg.go.dev/net/http#Header.Set) и [`Add()`](https://pkg.go.dev/net/http#Header.Add) прозрачно обрабатывают регистр символов и приводят ключи к канонической форме:

```go
r.Header.Set("auth-token", "secret")
token := r.Header.Get("Auth-Token") // Вернет "secret"
```

### Работа с cookie

Для работы с Cookie предусмотрены специализированные методы [`r.Cookie`](https://pkg.go.dev/net/http#Request.Cookie) и [`r.Cookies`](https://pkg.go.dev/net/http#Request.Cookies), которые избавляют от ручного парсинга заголовка `Cookie`:

```go
// Получение cookie по имени.
cookie, err := r.Cookie("session_id")
if err != nil {
    // Обработка отсутствия cookie.
}

// Получение всех cookies запроса в виде среза.
cookies := r.Cookies()
```

## Контекст и копирование

Каждый экземпляр `http.Request` содержит [`context.Context`](https://pkg.go.dev/context#Context), доступный через [`r.Context()`](https://pkg.go.dev/net/http#Request.Context). Это позволяет управлять жизненным циклом запроса и передавать данные между слоями приложения.

### Копирование запроса

Поскольку `http.Request` является структурой, а не интерфейсом, для его изменения рекомендуется создавать копии:

- **[`WithContext(ctx)`](https://pkg.go.dev/net/http#Request.WithContext)**: Создает поверхностную копию структуры с новым контекстом. Ссылочные поля (`Header`, `Body`) остаются общими.
- **[`Clone(ctx)`](https://pkg.go.dev/net/http#Request.Clone)**: Выполняет глубокое копирование большинства ссылочных полей, включая создание копий заголовков. Это безопасный способ модификации метаданных запроса перед его передачей далее по цепочке.

::: warning
`Clone` не создает независимую копию потока `Body`. Оригинальный запрос и его клон ссылаются на одно и то же тело запроса, поэтому чтение из одного экземпляра исчерпывает поток и для другого. Если тело нужно читать повторно, его необходимо явно буферизовать и заменить новым `io.ReadCloser`.
:::

```go
type contextKey string
const requestIDKey contextKey = "request_id"

// Модификация запроса через Clone перед передачей в следующий обработчик.
ctx := context.WithValue(r.Context(), requestIDKey, "uuid-123")
newReq := r.Clone(ctx)
newReq.Header.Set("X-Request-ID", "uuid-123")

next.ServeHTTP(w, newReq)
```

::: tip
В реальных приложениях не рекомендуется использовать базовые типы (такие как `string`) в качестве ключей контекста во избежание конфликтов имен (коллизий) между различными пакетами и частями приложения. Вместо этого следует определять собственные уникальные типы.
:::
