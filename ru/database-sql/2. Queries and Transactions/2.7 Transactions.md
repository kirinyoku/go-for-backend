# Управление транзакциями

[`*sql.Tx`](https://pkg.go.dev/database/sql#Tx) представляет незавершённую транзакцию. Через него несколько SQL-операций выполняются на одном соединении и завершаются вместе: успешные изменения подтверждаются через [`tx.Commit`](https://pkg.go.dev/database/sql#Tx.Commit), а при ошибке отменяются через [`tx.Rollback`](https://pkg.go.dev/database/sql#Tx.Rollback).

Транзакция нужна, когда частичное выполнение операции недопустимо. Например, при переводе средств нельзя списать сумму с одного счёта и не зачислить её на другой. Обе команды должны либо завершиться успешно, либо не изменить данные вовсе.

## Жизненный цикл транзакции

Пакет [`database/sql`](https://pkg.go.dev/database/sql) управляет жизненным циклом транзакции, но саму транзакцию выполняют драйвер и СУБД. Приложение начинает её через [`db.BeginTx`](https://pkg.go.dev/database/sql#DB.BeginTx), получает `*sql.Tx` и использует этот объект до завершения операции.

| Этап | Действие |
| :--- | :--- |
| Начало | `db.BeginTx` получает соединение из пула и создаёт `*sql.Tx`. |
| Выполнение | `tx.ExecContext`, `tx.QueryContext` и другие методы работают внутри транзакции. |
| Успех | `tx.Commit` подтверждает изменения. |
| Ошибка или отказ | `tx.Rollback` отменяет изменения. |

Пока транзакция открыта, она удерживает одно соединение из пула. После `Commit` или `Rollback` соединение возвращается пулу, а дальнейшие операции через `tx` завершаются ошибкой [`sql.ErrTxDone`](https://pkg.go.dev/database/sql#ErrTxDone).

::: warning
Каждая успешно открытая транзакция должна закончиться вызовом `Commit` или `Rollback`. Иначе она продолжит удерживать соединение и может мешать другим запросам получать доступ к пулу.
:::

## Начало транзакции (`BeginTx`)

Метод [`db.BeginTx`](https://pkg.go.dev/database/sql#DB.BeginTx) принимает контекст и необязательные параметры транзакции:

```go
tx, err := db.BeginTx(ctx, nil)
if err != nil {
    return fmt.Errorf("begin transaction: %w", err)
}
defer tx.Rollback()
```

Значение `nil` означает, что будут использованы настройки транзакции по умолчанию драйвера или СУБД. Ошибка может возникнуть ещё на этом этапе: например, если не удалось получить соединение, контекст уже отменён или драйвер не поддерживает запрошенные параметры.

## Параметры транзакции (`sql.TxOptions`)

Структура [`sql.TxOptions`](https://pkg.go.dev/database/sql#TxOptions) позволяет запросить уровень изоляции и режим только для чтения:

| Поле | Назначение |
| :--- | :--- |
| `Isolation` | Уровень изоляции типа [`sql.IsolationLevel`](https://pkg.go.dev/database/sql#IsolationLevel). Нулевое значение использует уровень по умолчанию драйвера или СУБД. |
| `ReadOnly` | Просит открыть транзакцию только для чтения. |

Например, транзакцию для согласованного чтения можно открыть так:

```go
opts := &sql.TxOptions{
    Isolation: sql.LevelSerializable,
    ReadOnly:  true,
}

tx, err := db.BeginTx(ctx, opts)
if err != nil {
    return fmt.Errorf("begin read transaction: %w", err)
}
defer tx.Rollback()
```

`database/sql` определяет общие константы уровней изоляции, но их поддержка и точное поведение зависят от драйвера и СУБД. Если запрошенный уровень не поддерживается, `BeginTx` вернёт ошибку. Режим `ReadOnly` также зависит от поддержки драйвера и не заменяет права доступа пользователя базы данных.

::: tip
Если операция не требует конкретного уровня изоляции или режима только для чтения, передавайте `nil`. Настройки по умолчанию обычно понятнее, чем параметр, выбранный без учёта поведения конкретной СУБД.
:::

## Выполнение запросов

У `*sql.Tx` есть знакомые методы выполнения запросов:

| Операция | Метод транзакции |
| :--- | :--- |
| Команда без набора строк | [`tx.ExecContext`](https://pkg.go.dev/database/sql#Tx.ExecContext) |
| Несколько строк | [`tx.QueryContext`](https://pkg.go.dev/database/sql#Tx.QueryContext) |
| Одна строка | [`tx.QueryRowContext`](https://pkg.go.dev/database/sql#Tx.QueryRowContext) |
| Подготовленный запрос | [`tx.PrepareContext`](https://pkg.go.dev/database/sql#Tx.PrepareContext) или [`tx.StmtContext`](https://pkg.go.dev/database/sql#Tx.StmtContext) |

Все SQL-операции, которые относятся к транзакции, нужно выполнять через `tx`. Вызов метода исходного `db` не становится частью транзакции автоматически и может использовать другое соединение.

## Контекст и время жизни транзакции

Контекст в `BeginTx` и контексты последующих запросов выполняют разные задачи. `database/sql` не подставляет контекст из `BeginTx` в `tx.ExecContext`, `tx.QueryContext` и другие методы автоматически. В качестве контекста конкретного запроса каждый такой метод использует именно тот контекст, который передан ему явно.

Один и тот же контекст передавать не обязательно. На практике обычно используют один из двух вариантов:

- передают один `ctx` в `BeginTx` и во все последующие запросы, если у всей операции единый крайний срок (deadline);
- передают в запрос дочерний контекст с более коротким тайм-аутом, если конкретный запрос нужно ограничить сильнее, чем транзакцию целиком.

Контекст, переданный в `BeginTx`, при этом отдельно действует до `Commit` или `Rollback` и задаёт верхнюю границу жизни всей транзакции. Если он отменяется раньше, `database/sql` откатывает транзакцию, а последующий `Commit` возвращает ошибку.

## Отложенный `Rollback`

Сразу после успешного `BeginTx` обычно регистрируют отложенный откат `defer tx.Rollback()`:

```go
tx, err := db.BeginTx(ctx, nil)
if err != nil {
    return fmt.Errorf("begin transfer: %w", err)
}
defer tx.Rollback()
```

Отложенный вызов сработает при любом последующем выходе из функции, поэтому после ошибки запроса не нужно вручную повторять `Rollback` в каждой ветке.

Если `Commit` прошёл успешно, транзакция уже завершена, последующий отложенный `Rollback` ничего не изменит.

## Подтверждение транзакции

Вызов `tx.Commit` должен быть последней операцией успешного пути. Транзакцию можно считать завершённой успешно только тогда, когда `Commit` вернул `nil`.

Ошибка `Commit` должна быть возвращена вызывающему коду. Она может появиться из-за отмены контекста, потери соединения или отказа базы подтвердить изменения. Результаты запросов внутри такой транзакции нельзя считать подтверждёнными.

Ограничение может проверяться сразу и вернуть ошибку из `ExecContext` или `Scan`, а может быть отложенным (*deferred*) до `Commit`. Поэтому преобразование ошибок для конкретной операции должно применяться и на границе подтверждения. В выбранном стеке PostgreSQL + pgx это можно сделать по структурированному SQLSTATE и имени ограничения:

```go
var ErrEmailTaken = errors.New("email is already used")

func commitUserRegistration(tx *sql.Tx) error {
    err := tx.Commit()
    if err == nil {
        return nil
    }

    commitErr := fmt.Errorf("commit user registration: %w", err)

    var pgErr *pgconn.PgError
    if errors.As(err, &pgErr) &&
        pgErr.Code == "23505" &&
        pgErr.ConstraintName == "users_email_key" {
        return errors.Join(ErrEmailTaken, commitErr)
    }

    return commitErr
}
```

Здесь [`pgconn`](https://pkg.go.dev/github.com/jackc/pgx/v5@v5.10.0/pgconn) — пакет выбранного драйвера, а `23505` означает нарушение уникальности (`unique_violation`). Универсальная вспомогательная функция для транзакций не может знать, какое из ограничений соответствует `ErrEmailTaken`: такое преобразование принадлежит конкретной операции.

::: warning
Не повторяйте автоматически всю операцию после неизвестной ошибки `Commit`. При сетевом сбое приложение может не знать, успела ли база данных подтвердить изменения. Для безопасного повтора операция не должна создавать повторный эффект при повторном вызове — это свойство называют идемпотентностью. Другой вариант — сначала проверить текущее состояние данных.
:::

После вызова `Commit`, независимо от его результата, не пытайтесь продолжать работу через тот же `tx`. Для новой операции нужно открыть новую транзакцию.

## Полный успешный путь

В одной функции должны быть видны владелец транзакции, обе ветки завершения и проверка `Commit`. Например, изменение пользователя и запись аудита выполняются через один `tx`:

```go
func updateUserAndAudit(
    ctx context.Context,
    db *sql.DB,
    userID int64,
    displayName string,
) error {
    tx, err := db.BeginTx(ctx, nil)
    if err != nil {
        return fmt.Errorf("begin user update: %w", err)
    }
    defer tx.Rollback()

    result, err := tx.ExecContext(ctx, `
        UPDATE users
        SET display_name = $1
        WHERE id = $2
    `, displayName, userID)
    if err != nil {
        return fmt.Errorf("update user: %w", err)
    }

    affected, err := result.RowsAffected()
    if err != nil {
        return fmt.Errorf("get updated rows: %w", err)
    }
    if affected != 1 {
        return fmt.Errorf("update user: expected one row, got %d", affected)
    }

    if _, err := tx.ExecContext(ctx, `
        INSERT INTO audit_log (user_id, event)
        VALUES ($1, 'user_updated')
    `, userID); err != nil {
        return fmt.Errorf("write audit event: %w", err)
    }

    if err := tx.Commit(); err != nil {
        return fmt.Errorf("commit user update: %w", err)
    }

    return nil
}
```

Любой ранний `return` приводит к отложенному `Rollback`; успешный путь заканчивается проверенным `Commit`. Вызов `db.ExecContext` вместо `tx.ExecContext` нарушил бы эту атомарность.

## Повтор при конфликте сериализации

PostgreSQL возвращает SQLSTATE `40001` при ошибке сериализации (`serialization_failure`). Если приложение решает повторить такую операцию, [документация PostgreSQL](https://www.postgresql.org/docs/current/mvcc-serialization-failure-handling.html) требует повторять **всю транзакцию с новым `BeginTx`**, включая чтения и логику выбора последующих SQL-команд. Ошибка взаимной блокировки (`deadlock_detected`, код `40P01`) также может быть кандидатом на такой повтор.

Один из вариантов координатора с ограниченным числом повторов выглядит так:

```go
func isRetryablePostgresTxError(err error) bool {
    var pgErr *pgconn.PgError
    if !errors.As(err, &pgErr) {
        return false
    }

    return pgErr.Code == "40001" || pgErr.Code == "40P01"
}

func runSerializableAttempt(
    ctx context.Context,
    db *sql.DB,
    fn func(context.Context, *sql.Tx) error,
) error {
    tx, err := db.BeginTx(ctx, &sql.TxOptions{
        Isolation: sql.LevelSerializable,
    })
    if err != nil {
        return fmt.Errorf("begin serializable transaction: %w", err)
    }
    defer tx.Rollback()

    if err := fn(ctx, tx); err != nil {
        return fmt.Errorf("run serializable transaction: %w", err)
    }
    if err := tx.Commit(); err != nil {
        return fmt.Errorf("commit serializable transaction: %w", err)
    }

    return nil
}

func runSerializable(
    ctx context.Context,
    db *sql.DB,
    fn func(context.Context, *sql.Tx) error,
) error {
    const maxAttempts = 3

    var lastErr error
    for attempt := 1; attempt <= maxAttempts; attempt++ {
        lastErr = runSerializableAttempt(ctx, db, fn)
        if lastErr == nil {
            return nil
        }
        if !isRetryablePostgresTxError(lastErr) ||
            attempt == maxAttempts {
            return lastErr
        }

        backoff := 20*time.Millisecond << (attempt - 1)
        jitter := time.Duration(rand.IntN(20)) * time.Millisecond

        select {
        case <-ctx.Done():
            return fmt.Errorf("wait before transaction retry: %w", ctx.Err())
        case <-time.After(backoff + jitter):
        }
    }

    return lastErr
}
```

В примере `rand` — это `math/rand/v2`, доступный начиная с Go 1.22. Лимит попыток защищает базу от бесконечного цикла повторов, экспоненциальная задержка (*exponential backoff*) увеличивает паузу между попытками, а случайный разброс (*jitter*) снижает вероятность нового одновременного конфликта. Ожидание прекращается вместе с контекстом.

::: warning
Callback-функция `fn` может выполниться несколько раз. Внутри неё допустимы операции, чьи эффекты входят в ту же транзакцию; не отправляйте там HTTP-запросы, электронные письма или сообщения во внешний брокер. Координатор не повторяет неизвестные сетевые ошибки, тайм-ауты и ошибки `Commit` без структурированного `40001`/`40P01`.
:::

## Обработка ошибок `Rollback`

В простом шаблоне `defer tx.Rollback()` возвращаемая ошибка намеренно игнорируется. После успешного `Commit` отложенный вызов обычно получает `sql.ErrTxDone`, потому что транзакция уже завершена. Это ожидаемое поведение, а не ошибка выполненной операции.

Если ошибка отката важна для диагностики, её можно отфильтровать и записать в лог:

```go
defer func() {
    err := tx.Rollback()
    if err != nil && !errors.Is(err, sql.ErrTxDone) {
        slog.Error("rollback transaction", "error", err)
    }
}()
```

Ошибка отката незавершённой транзакции может указывать на проблему с соединением или драйвером. При этом `*sql.Tx` всё равно больше нельзя использовать. Если до отката уже произошла основная ошибка запроса, сохраняйте её как причину неуспеха операции, а ошибку rollback добавляйте в лог или объединяйте с основной ошибкой там, где это действительно нужно.
