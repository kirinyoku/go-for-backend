# Параметризация SQL-запросов

Методы [`ExecContext`](https://pkg.go.dev/database/sql#DB.ExecContext), [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext) и [`QueryRowContext`](https://pkg.go.dev/database/sql#DB.QueryRowContext) принимают SQL-строку и отдельный список аргументов. Такая форма позволяет не подставлять пользовательские значения непосредственно в текст запроса.

Разделение SQL и данных не позволяет передаваемым значениям становиться частью SQL-синтаксиса и избавляет приложение от ручного экранирования строк. При этом placeholder-синтаксис не задаётся пакетом `database/sql`: форму `?`, `$1` или `@p1` определяет выбранный драйвер.

## Опасность подстановки значений в SQL

Опасный способ построить запрос выглядит так:

```go
query := fmt.Sprintf(`
    SELECT id, email
    FROM users
    WHERE email = '%s'
`, email)

rows, err := db.QueryContext(ctx, query)
if err != nil {
    return nil, err
}
defer rows.Close()
```

Если `email` пришёл от пользователя, он может содержать не только адрес, но и часть SQL-синтаксиса. Например, значение:

```text
x' OR 1=1 -- comment
```

превратит условие в запрос, похожий на этот:

```sql
SELECT id, email
FROM users
WHERE email = 'x' OR 1=1 -- comment'
```

Условие `OR 1=1` истинно для всех строк, а `-- ` с последующим пробелом начинает однострочный комментарий в том числе в MySQL. В результате запрос может вернуть данные, к которым пользователь не должен иметь доступ. Точный синтаксис комментариев и другие детали строки атаки всё равно зависят от SQL-диалекта; защита должна строиться на параметрах, а не на попытке распознавать отдельные строки.

Та же проблема возникает при конкатенации:

```go
query := "SELECT id, email FROM users WHERE email = '" + email + "'"
```

Ручное добавление кавычек или замена отдельных символов не является надёжной защитой. Правила экранирования различаются между базами, типами данных и режимами работы драйвера.

::: info
Проблема заключается не в самом использовании `fmt.Sprintf`, а в подстановке недоверенных данных в SQL. Ниже он будет безопасно использоваться для генерации порядковых номеров placeholder-параметров из индекса цикла, который контролирует приложение.
:::

## Placeholder-параметры и аргументы

Placeholder обозначает место, куда драйвер должен передать отдельное значение. Само значение указывается после SQL-строки:

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE email = $1",
    email,
)
if err != nil {
    return nil, fmt.Errorf("query user by email: %w", err)
}
defer rows.Close()
```

В этом вызове есть две независимые части:

- `"SELECT ... WHERE email = $1"` задаёт структуру SQL-запроса;
- `email` передаётся как значение для placeholder-параметра `$1`.

`database/sql` передаёт запрос и аргументы драйверу. Драйвер сам решает, как связать значения с placeholder-параметрами: через протокол базы, подготовленный запрос или другой поддерживаемый механизм. Приложению не нужно самостоятельно заключать строку в кавычки или экранировать её.

Placeholder не нужно помещать в SQL-кавычки:

```sql
-- Правильно: placeholder-параметр находится на месте значения.
WHERE email = $1

-- Неправильно: placeholder $1 становится частью строкового литерала.
WHERE email = '$1'
```

## Синтаксис placeholder-параметров в разных драйверах

Пакет `database/sql` не переводит один placeholder-синтаксис в другой. Запрос должен использовать форму, которую ожидает конкретный драйвер.

Часто встречаются такие варианты:

| Форма | Где часто встречается |
| :--- | :--- |
| `?` | Драйверы MySQL и SQLite |
| `$1` | Драйверы PostgreSQL |
| `@p1` | Драйверы SQL Server |
| `@name`, `:name` | Драйверы с именованными параметрами |

Эта таблица показывает распространённые соглашения, а не универсальное правило для каждой СУБД. Точный синтаксис нужно проверять в документации выбранного драйвера.

Смена драйвера может потребовать изменения SQL-запросов, даже если Go-код продолжает использовать те же методы `database/sql`. Общий API не делает SQL-диалекты и placeholder-синтаксис переносимыми автоматически.

## Порядок и типы аргументов

Методы выполнения запросов принимают аргументы после SQL-строки в виде `args ...any`. Правило их сопоставления зависит от формы placeholder-параметров:

- каждый анонимный `?` обычно получает следующий аргумент по порядку;
- `$n` ссылается на аргумент с номером `n`, начиная с единицы, поэтому `$1` можно использовать несколько раз, а номера могут встречаться не по порядку;
- именованный placeholder-параметр сопоставляется с именем из `sql.Named`, если драйвер поддерживает именованные параметры.

В PostgreSQL-примере ниже `$1` получает `email`, `$2` — `active`, а `$3` — `userID`:

```go
_, err := db.ExecContext(ctx, `
    UPDATE users
    SET email = $1, active = $2
    WHERE id = $3
`, email, active, userID)
if err != nil {
    return fmt.Errorf("update user: %w", err)
}
```

Запрос должен получать аргументы, соответствующие использованным placeholder-параметрам. Точные правила повторного использования, пропущенных номеров и место проверки определяет выбранный драйвер.

Аргументами могут быть строки, числа, булевы значения, `time.Time`, `nil` и другие значения, поддерживаемые `database/sql` или конкретным драйвером. Преобразование собственных типов через `driver.Valuer` разобрано в статье [Собственные типы данных](/ru/database-sql/queries/custom-data-types).

::: info
Передача `nil` создаёт SQL-значение `NULL`, но обычное условие `column = NULL` не работает как сравнение с отсутствующим значением. Для проверки `NULL` в SQL используется `IS NULL`; структуру такого условия нужно добавить в текст SQL-запроса.
:::

## Именованные аргументы

Пакет предоставляет функцию [`sql.Named`](https://pkg.go.dev/database/sql#Named) для создания именованных аргументов:

```go
_, err := db.ExecContext(ctx, `
    UPDATE users
    SET email = @email
    WHERE id = @id
`,
    sql.Named("email", email),
    sql.Named("id", userID),
)
if err != nil {
    return fmt.Errorf("update user: %w", err)
}
```

Имя в `sql.Named` указывается без префикса: `"email"`, а не `"@email"`. Символ в SQL-запросе зависит от синтаксиса драйвера.

Именованные аргументы могут сделать длинный запрос понятнее и снизить риск перепутать значения одинакового типа. Однако драйвер должен поддерживать соответствующую форму именованных параметров.

## Безопасное построение динамических фильтров

Иногда набор условий зависит от входных данных: например, запрос может фильтровать пользователей по email, активности или обоим полям сразу. В таком случае динамически собирается только структура SQL, а значения продолжают передаваться отдельными аргументами.

```go
type UserFilter struct {
    Email  *string
    Active *bool
}

func buildUserQuery(filter UserFilter) (string, []any) {
    query := "SELECT id, email, active FROM users"
    conditions := make([]string, 0, 2)
    args := make([]any, 0, 2)

    if filter.Email != nil {
        args = append(args, *filter.Email)
        placeholder := fmt.Sprintf("$%d", len(args))
        conditions = append(conditions, "email = "+placeholder)
    }

    if filter.Active != nil {
        args = append(args, *filter.Active)
        placeholder := fmt.Sprintf("$%d", len(args))
        conditions = append(conditions, "active = "+placeholder)
    }

    if len(conditions) > 0 {
        query += " WHERE " + strings.Join(conditions, " AND ")
    }

    query += " ORDER BY id"

    return query, args
}
```

Полученные аргументы передаются с `...`:

```go
query, args := buildUserQuery(filter)

rows, err := db.QueryContext(ctx, query, args...)
if err != nil {
    return nil, fmt.Errorf("query users: %w", err)
}
defer rows.Close()
```

Фрагменты `"email = $n"` и `"active = $n"` формирует приложение из позиции аргумента. Пользовательские значения не становятся частью SQL, а попадают только в `args`.

Указатели в `UserFilter` помогают различить два состояния: фильтр отсутствует или передано конкретное значение. Для `Active` это особенно важно, потому что `false` является полноценным значением фильтра.

Номер placeholder-параметра безопасно строить из длины `args`, потому что он определяется кодом, а не пользовательским вводом.

## Переменное число значений в `IN`

Один placeholder-параметр обычно соответствует одному значению. Поэтому такой код чаще всего не раскрывает срез в несколько SQL-аргументов:

```go
ids := []int64{10, 20, 30}

rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE id IN ($1)",
    ids,
)
```

Драйвер получает один аргумент типа `[]int64`, а не три отдельных значения. Он может вернуть ошибку о неподдерживаемом типе или обработать значение не так, как ожидает код.

Для PostgreSQL нужное количество нумерованных placeholder-параметров можно сгенерировать из длины среза:

```go
if len(ids) == 0 {
    return []User{}, nil
}

placeholders := make([]string, len(ids))
args := make([]any, len(ids))

for i, id := range ids {
    placeholders[i] = fmt.Sprintf("$%d", i+1)
    args[i] = id
}

query := `
    SELECT id, email
    FROM users
    WHERE id IN (` + strings.Join(placeholders, ", ") + `)
`

rows, err := db.QueryContext(ctx, query, args...)
if err != nil {
    return nil, fmt.Errorf("query users by ids: %w", err)
}
defer rows.Close()
```

Отдельная обработка пустого среза нужна потому, что `IN ()` является ошибкой синтаксиса во многих SQL-диалектах. Если по смыслу пустой список ID означает пустой результат, функция может вернуть его без обращения к базе.

Некоторые базы и драйверы поддерживают массивы или специальные конструкции для передачи списка одним аргументом. Это возможность конкретной СУБД, которую нужно проверять отдельно.

## Динамические идентификаторы: только разрешённые значения

Placeholder-параметр представляет значение, но не SQL-синтаксис. Через него нельзя подставить:

- имя таблицы;
- имя колонки;
- `ASC` или `DESC`;
- оператор сравнения;
- произвольный фрагмент `WHERE`.

Например, этот запрос не сортирует строки по колонке, имя которой находится в `sortColumn`:

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users ORDER BY $1",
    sortColumn,
)
```

Драйвер воспринимает `sortColumn` как обычное значение, а не как часть SQL. Запрос может завершиться ошибкой или фактически сортировать по одной константе.

Для динамических идентификаторов используется список разрешённых значений: внешнее значение сопоставляется с заранее известным безопасным SQL-фрагментом.

```go
allowedSortColumns := map[string]string{
    "email":      "email",
    "created_at": "created_at",
}

sortColumn, ok := allowedSortColumns[sortKey]
if !ok {
    return nil, ErrInvalidSort
}

direction := "ASC"
if descending {
    direction = "DESC"
}

query := `
    SELECT id, email
    FROM users
    ORDER BY ` + sortColumn + " " + direction

rows, err := db.QueryContext(ctx, query)
if err != nil {
    return nil, fmt.Errorf("query users: %w", err)
}
defer rows.Close()
```

В SQL попадают только строки из `allowedSortColumns` и два значения `direction`, заданные приложением. `sortKey` не добавляется в запрос напрямую.
