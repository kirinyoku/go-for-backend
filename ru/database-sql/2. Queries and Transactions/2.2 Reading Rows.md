# Чтение результатов запроса

Метод [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext) возвращает [`*sql.Rows`](https://pkg.go.dev/database/sql#Rows) — объект для последовательного чтения строк результата. Он не превращает результат запроса в готовый срез Go-структур: приложение само переходит между строками, сканирует колонки и собирает нужные значения.

Работа с `*sql.Rows` состоит из нескольких обязательных шагов. Сначала нужно проверить ошибку `QueryContext`, затем запланировать закрытие `Rows`, прочитать строки в цикле и после цикла проверить ошибку итерации. Пропуск одного из этих шагов может привести к потере ошибки или удержанию соединения из пула.

## Полный цикл чтения

Типичный метод, возвращающий список пользователей, выглядит так:

```go
func listActiveUsers(ctx context.Context, db *sql.DB) ([]User, error) {
    rows, err := db.QueryContext(ctx, `
        SELECT id, email, active
        FROM users
        WHERE active = $1
        ORDER BY id
    `, true)
    if err != nil {
        return nil, fmt.Errorf("query active users: %w", err)
    }
    defer rows.Close()

    var users []User

    for rows.Next() {
        var user User

        if err := rows.Scan(&user.ID, &user.Email, &user.Active); err != nil {
            return nil, fmt.Errorf("scan user: %w", err)
        }

        users = append(users, user)
    }

    if err := rows.Err(); err != nil {
        return nil, fmt.Errorf("iterate users: %w", err)
    }

    return users, nil
}
```

Порядок действий здесь важен:

1. `QueryContext` выполняет запрос и возвращает `*sql.Rows`.
2. `defer rows.Close()` гарантирует закрытие результата при любом выходе из функции.
3. `rows.Next()` подготавливает очередную строку.
4. `rows.Scan(...)` копирует колонки текущей строки в поля Go-структуры.
5. `rows.Err()` проверяет, не завершилась ли итерация из-за ошибки.

## Получение набора строк (`*sql.Rows`)

Перед чтением строк нужно проверить ошибку, возвращённую `QueryContext`:

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE active = $1",
    true,
)
if err != nil {
    return nil, fmt.Errorf("query users: %w", err)
}
defer rows.Close()
```

Ошибка на этом этапе может означать, что пакет не смог получить соединение, контекст был отменён, драйвер отклонил запрос или база сообщила о проблеме при его выполнении.

Успешный `QueryContext` означает, что приложение получило объект для чтения результата. Это ещё не гарантирует, что все строки будут прочитаны без ошибок: проблема может возникнуть позднее, во время итерации.

## Закрытие результата (`rows.Close`)

Метод [`rows.Close`](https://pkg.go.dev/database/sql#Rows.Close) завершает чтение результата и освобождает связанные с ним ресурсы. Обычно его вызывают через `defer` сразу после успешного `QueryContext`:

```go
rows, err := db.QueryContext(ctx, query, args...)
if err != nil {
    return err
}
defer rows.Close()
```

Такой порядок гарантирует закрытие `Rows`, если функция завершится раньше обычного: например, из-за ошибки `Scan`, ошибки бизнес-логики или явного `return` внутри цикла.

::: warning
Закрытие нужно планировать только после проверки `err`. Если `QueryContext` завершился с ошибкой, `rows` равен `nil`, и попытка вызвать у него `Close` приведёт к `panic`.
:::

Когда `rows.Next()` доходит до конца последнего набора строк, `database/sql` закрывает `Rows` автоматически. Тем не менее `defer rows.Close()` остаётся полезной страховкой для случаев, когда код прочитал только часть результата или завершился раньше.

Повторный вызов `Close` безопасен: метод идемпотентен. Поэтому автоматическое закрытие после последнего `Next` не конфликтует с отложенным `defer rows.Close()`.

## Переход между строками (`rows.Next`)

Сразу после `QueryContext` курсор находится перед первой строкой результата. Метод [`rows.Next`](https://pkg.go.dev/database/sql#Rows.Next) подготавливает следующую строку и возвращает `true`, если её можно прочитать через `Scan`.

```go
for rows.Next() {
    // Здесь доступна текущая строка.
}
```

Каждый вызов `Scan`, включая первый, должен выполняться только после того, как `Next` вернул `true`. В следующей итерации `Next` переводит курсор на следующую строку.

С точки зрения приложения результат обрабатывается по одной текущей строке, а не возвращается сразу как `[]User`. Драйвер при этом может получать данные от базы порциями; точная стратегия буферизации зависит от его реализации.

`Next` возвращает `false` в двух случаях:

- строки закончились;
- во время подготовки следующей строки произошла ошибка.

По одному значению `false` эти случаи не различить. Поэтому после цикла обязательно вызывается [`rows.Err()`](https://pkg.go.dev/database/sql#Rows.Err).

Если запрос не вернул строк, первый `Next` вернёт `false`, а `rows.Err()` — `nil`. Это нормальный успешный результат `QueryContext`, а не `sql.ErrNoRows`. Если прикладной контракт должен отличать пустой набор от непустого, проверяйте длину итогового среза или устанавливайте флаг внутри цикла.

## Чтение значений (`rows.Scan`)

Метод [`rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan) копирует колонки текущей строки в переданные Go-переменные:

```go
for rows.Next() {
    var user User

    if err := rows.Scan(&user.ID, &user.Email, &user.Active); err != nil {
        return nil, fmt.Errorf("scan user: %w", err)
    }

    users = append(users, user)
}
```

Аргументы `Scan` должны соответствовать колонкам из `SELECT` по количеству и порядку:

```sql
SELECT id, email, active FROM users
```

```go
rows.Scan(&user.ID, &user.Email, &user.Active)
```

Названия полей структуры не сопоставляются с названиями колонок автоматически; значение имеет именно позиция.

В `Scan` передаются указатели, потому что метод должен изменить значения переменных. Такой вызов неверен:

```go
// Переданы значения, а не адреса переменных.
err := rows.Scan(user.ID, user.Email, user.Active)
```

Если количество аргументов не совпадает с количеством колонок или значение нельзя преобразовать в выбранный Go-тип, `Scan` вернёт ошибку. Эту ошибку нужно обработать непосредственно внутри цикла: `rows.Err()` её не заменяет.

::: tip
Перечисляйте нужные колонки явно вместо `SELECT *`. Тогда изменение порядка или добавление колонок в таблицу не нарушит соответствие между результатом запроса и аргументами `Scan`.
:::

Работа с `NULL`, `time.Time`, числовыми типами, `[]byte` и типами `sql.Null*` разобрана в статье [Преобразование SQL-значений в Go-типы](/ru/database-sql/queries/scanning-types). Здесь достаточно помнить, что Go-тип должен быть совместим со значением выбранной колонки.

## Проверка ошибок итерации (`rows.Err`)

Когда `rows.Next()` возвращает `false`, цикл заканчивается. Чтобы понять, завершилось чтение нормально или из-за ошибки, нужно вызвать [`rows.Err`](https://pkg.go.dev/database/sql#Rows.Err):

```go
for rows.Next() {
    var user User

    if err := rows.Scan(&user.ID, &user.Email); err != nil {
        return nil, fmt.Errorf("scan user: %w", err)
    }

    users = append(users, user)
}

if err := rows.Err(); err != nil {
    return nil, fmt.Errorf("iterate users: %w", err)
}
```

Ошибка итерации может появиться уже после нескольких успешно прочитанных строк: например, если контекст отменился, соединение оборвалось или драйвер не смог получить следующую часть результата.

Без проверки `rows.Err()` функция вернула бы неполный список как будто запрос завершился успешно. Это особенно опасно, потому что часть данных выглядит корректно и ошибка остаётся незаметной.

При нормальном завершении или пустом результате `rows.Err()` возвращает `nil`.

## Почему результат удерживает соединение

Для выполнения `QueryContext` пакет получает соединение из пула. Пока `*sql.Rows` остаётся открытым, это соединение занято чтением результата и обычно не может обслуживать другой запрос через `*sql.DB`.

Если приложение забывает закрывать `Rows`, количество свободных соединений уменьшается. Когда пул исчерпан, новые запросы ждут освобождения соединения и могут завершиться по тайм-ауту.

::: warning
Не выполняйте без необходимости дополнительные запросы через `db` внутри цикла по открытым `Rows`. При маленьком пуле новый запрос может ждать соединение, которое удерживается текущим результатом. Часто данные лучше получить одним SQL-запросом или сначала завершить чтение и закрыть `Rows`.
:::

По той же причине не стоит накапливать `defer rows.Close()` внутри длинного цикла, который выполняет много отдельных запросов: все отложенные вызовы сработают только при выходе из окружающей функции. Вынесите одну итерацию в отдельную функцию с собственным `defer` или закрывайте каждый `Rows` до перехода к следующему запросу.
