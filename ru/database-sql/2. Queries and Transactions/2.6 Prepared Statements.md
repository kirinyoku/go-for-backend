# Подготовленные запросы

Подготовленный запрос (*prepared statement*) позволяет создать объект для фиксированного SQL-шаблона, а затем выполнить его один или несколько раз с разными аргументами.

В [`database/sql`](https://pkg.go.dev/database/sql) подготовленный запрос представлен типом [`*sql.Stmt`](https://pkg.go.dev/database/sql#Stmt). Он создаётся через [`PrepareContext`](https://pkg.go.dev/database/sql#DB.PrepareContext), выполняется методами [`ExecContext`](https://pkg.go.dev/database/sql#Stmt.ExecContext), [`QueryContext`](https://pkg.go.dev/database/sql#Stmt.QueryContext) или [`QueryRowContext`](https://pkg.go.dev/database/sql#Stmt.QueryRowContext) и закрывается через [`Close`](https://pkg.go.dev/database/sql#Stmt.Close).

## Как работает подготовленный запрос

Обычный параметризованный запрос передаёт SQL и аргументы за один вызов:

```go
err := db.QueryRowContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
    userID,
).Scan(&user.ID, &user.Email)
if err != nil {
    return fmt.Errorf("find user: %w", err)
}
```

При явной подготовке SQL передаётся заранее, а выполнение получает только аргументы:

```go
stmt, err := db.PrepareContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
)
if err != nil {
    return fmt.Errorf("prepare user query: %w", err)
}
defer stmt.Close()

err = stmt.QueryRowContext(ctx, userID).Scan(&user.ID, &user.Email)
if err != nil {
    return fmt.Errorf("find user: %w", err)
}
```

На уровне `database/sql` вызов `PrepareContext` гарантирует создание `*sql.Stmt`, связанного с конкретным SQL-текстом. Этот объект можно выполнять многократно, передавая только аргументы, пока он не закрыт и продолжает жить его владелец.

При этом `database/sql` не гарантирует, что SQL будет отправлен серверу ровно один раз, что СУБД сохранит один план или что прямой `db.QueryContext` каждый раз проходит полную подготовку заново. Реализация зависит от драйвера, протокола и СУБД:

- драйвер может автоматически подготавливать и кэшировать обычные запросы без явного `PrepareContext`;
- подготовленный запрос, созданный через `*sql.DB`, может подготавливаться отдельно на нескольких физических соединениях пула;
- СУБД может выбирать между общим планом и планом, построенным с учётом конкретных аргументов.

В выбранном для примеров [pgx](https://pkg.go.dev/github.com/jackc/pgx/v5) обычные вызовы по умолчанию уже используют автоматический кеш подготовленных запросов (statement cache). Поэтому явный `PrepareContext` нужно рассматривать как API для осознанного владения и повторного использования `*sql.Stmt`, а не как гарантированную оптимизацию.

## Подготовка через `PrepareContext`

Метод [`db.PrepareContext`](https://pkg.go.dev/database/sql#DB.PrepareContext) создаёт `*sql.Stmt` для последующих запросов или команд:

```go
stmt, err := db.PrepareContext(ctx, `
    SELECT id, email, active
    FROM users
    WHERE id = $1
`)
if err != nil {
    return nil, fmt.Errorf("prepare find user: %w", err)
}
defer stmt.Close()
```

В SQL остаются placeholder-параметры, но значения пока не передаются. Они появятся при выполнении `stmt`.

Контекст в `PrepareContext` ограничивает только этап подготовки. Он не сохраняется внутри `*sql.Stmt` и не управляет будущими выполнениями. Поэтому каждый вызов подготовленного запроса должен получать контекст текущей операции: например, [`r.Context()`](https://pkg.go.dev/net/http#Request.Context) HTTP-запроса или отдельный контекст с тайм-аутом.

## Выполнение через `*sql.Stmt`

Методы `*sql.Stmt` соответствуют основным методам `*sql.DB`, но не принимают SQL-строку повторно.

### Одна строка: `stmt.QueryRowContext`

Для запроса одной строки аргументы передаются сразу в `QueryRowContext`, а ошибка проверяется при `Scan`:

```go
findUserStmt, err := db.PrepareContext(ctx, `
    SELECT id, email
    FROM users
    WHERE id = $1
`)
if err != nil {
    return User{}, fmt.Errorf("prepare find user: %w", err)
}
defer findUserStmt.Close()

var user User

err = findUserStmt.QueryRowContext(ctx, userID).Scan(
    &user.ID,
    &user.Email,
)
if err != nil {
    if errors.Is(err, sql.ErrNoRows) {
        return User{}, ErrUserNotFound
    }

    return User{}, fmt.Errorf("find user: %w", err)
}

return user, nil
```

Как и у `db.QueryRowContext`, ошибка выполнения откладывается до `Scan`.

### `stmt.ExecContext`

Для команды без возвращаемых строк используется `ExecContext`:

```go
updateUserStmt, err := db.PrepareContext(ctx, `
    UPDATE users
    SET active = $1
    WHERE id = $2
`)
if err != nil {
    return fmt.Errorf("prepare update user: %w", err)
}
defer updateUserStmt.Close()

_, err = updateUserStmt.ExecContext(ctx, active, userID)
if err != nil {
    return fmt.Errorf("update user: %w", err)
}
```

Метод также возвращает [`sql.Result`](https://pkg.go.dev/database/sql#Result), который работает так же, как результат `db.ExecContext`. Методы [`RowsAffected`](https://pkg.go.dev/database/sql#Result.RowsAffected) и [`LastInsertId`](https://pkg.go.dev/database/sql#Result.LastInsertId) разобраны в статье [Выбор метода выполнения запроса](/ru/database-sql/queries/exec-query-queryrow).

### `stmt.QueryContext`

Если запрос возвращает набор строк, `*sql.Stmt` создаёт обычный `*sql.Rows`:

```go
listActiveStmt, err := db.PrepareContext(ctx, `
    SELECT id, email
    FROM users
    WHERE active = $1
    ORDER BY id
`)
if err != nil {
    return nil, fmt.Errorf("prepare active users: %w", err)
}
defer listActiveStmt.Close()

rows, err := listActiveStmt.QueryContext(ctx, true)
if err != nil {
    return nil, fmt.Errorf("query active users: %w", err)
}
defer rows.Close()
```

Дальше действуют обычные правила [`rows.Next`](https://pkg.go.dev/database/sql#Rows.Next), [`rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan) и [`rows.Err`](https://pkg.go.dev/database/sql#Rows.Err). Полный шаблон находится в статье [Чтение результатов запроса](/ru/database-sql/queries/reading-rows).

## Повторное использование и конкурентность

`*sql.Stmt` безопасен для конкуретного использования несколькими горутинами. Один подготовленный запрос можно передать нескольким запросам приложения, если его время жизни соответствует владельцу и он не привязан к уже завершённой транзакции или закрытому соединению.

Подготовку стоит вынести за пределы цикла:

```go
activateUserStmt, err := db.PrepareContext(ctx,
    "UPDATE users SET active = $1 WHERE id = $2",
)
if err != nil {
    return fmt.Errorf("prepare update user: %w", err)
}
defer activateUserStmt.Close()

for _, userID := range userIDs {
    if _, err := activateUserStmt.ExecContext(ctx, true, userID); err != nil {
        return fmt.Errorf("activate user %d: %w", userID, err)
    }
}
```

Если вызывать `PrepareContext` внутри каждой итерации, приложение повторяет подготовку и создаёт дополнительные ресурсы без переиспользования подготовленного запроса.

::: info
Повторные отдельные `UPDATE` не становятся одной атомарной операцией только из-за общего подготовленного запроса. Если все изменения должны либо примениться вместе, либо вместе откатиться, нужна транзакция. Подробнее в следующей статье [Управление транзакциями](/ru/database-sql/queries/transactions).
:::

## `*sql.Stmt`

Время жизни и привязка `*sql.Stmt` зависят от объекта, через который он подготовлен:

| Где создан | Привязка | Когда становится непригоден |
| :--- | :--- | :--- |
| `db.PrepareContext` | Управляется пулом `*sql.DB` | После `stmt.Close()` или `db.Close()` |
| `conn.PrepareContext` | Одно выделенное соединение | После `stmt.Close()` или `conn.Close()` |
| `tx.PrepareContext` | Соединение и границы одной транзакции | После `stmt.Close()`, `tx.Commit()` или `tx.Rollback()` |

Подготовленный запрос через `*sql.DB` не закрепляет за собой одно соединение навсегда. Он остаётся доступен на протяжении жизни `DB`, а когда выполнение попадает на новое соединение из пула, `database/sql` автоматически подготавливает SQL на этом соединении.

Из этого следует важная деталь: один `*sql.Stmt` в Go может соответствовать нескольким подготовленным запросам на разных соединениях. Чем больше пул и чем больше долгоживущих `*sql.Stmt`, тем больше ресурсов может потребоваться драйверу и базе.

Подготовленный запрос через `*sql.Conn` или `*sql.Tx`, наоборот, навсегда связан с одним физическим соединением. После закрытия `Conn` или завершения транзакции использовать его нельзя.

## Закрытие `*sql.Stmt`

`*sql.Stmt` нужно закрывать, когда он больше не используется:

```go
stmt, err := db.PrepareContext(ctx, query)
if err != nil {
    return err
}
defer stmt.Close()
```

`Close` освобождает ресурсы, связанные с этим `*sql.Stmt`, и запрещает новые операции через него.

Для локального подготовленного запроса удобно вызывать `defer stmt.Close()` сразу после успешной подготовки. Долгоживущий `*sql.Stmt`, сохранённый в структуре приложения, должен закрываться владельцем этой структуры при завершении работы, обычно рядом с закрытием `*sql.DB`.

## Когда нужна явная подготовка

Явная подготовка уместна, когда один SQL-шаблон выполняется многократно в пределах понятного времени жизни: например, в коротком цикле, во многих запросах приложения через долгоживущий объект или несколько раз внутри одной транзакции.

Для короткого цикла запрос подготавливают перед циклом и закрывают после него. Долгоживущий `*sql.Stmt` на `*sql.DB` закрывает владелец компонента при завершении работы, а подготовленный запрос внутри транзакции создают через `tx.PrepareContext` и не используют после её завершения.

Явная подготовка не гарантирует ускорения. Драйвер уже может автоматически кэшировать запросы, а один `*sql.Stmt` на `*sql.DB` способен занимать ресурсы на нескольких соединениях пула. Если осознанного повторного использования нет, прямой вызов `db.ExecContext`, `db.QueryContext` или `db.QueryRowContext` обычно проще. Пользу подготовки для производительности стоит подтверждать измерениями на реальной нагрузке.
