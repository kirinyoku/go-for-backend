# Собственные типы данных

Пакет [`database/sql`](https://pkg.go.dev/database/sql) умеет сканировать стандартные значения в `string`, `int64`, `time.Time`, `[]byte` и другие базовые Go-типы. Но приложению часто нужны более выразительные типы: статус заказа с ограниченным набором значений, JSON-метаданные, UUID или другой тип, специфичный для приложения.

Чтобы такой тип можно было напрямую передавать в SQL-запрос и получать через `Scan`, он должен реализовать два стандартных интерфейса: [`sql.Scanner`](https://pkg.go.dev/database/sql#Scanner) для чтения из базы и [`driver.Valuer`](https://pkg.go.dev/database/sql/driver#Valuer) для записи в базу.

## Зачем нужны собственные типы

Без собственного типа преобразование часто повторяется в каждом методе доступа к данным. Например, строковый статус приходится прочитать, проверить и преобразовать вручную:

```go
var rawStatus string

if err := row.Scan(&order.ID, &rawStatus); err != nil {
    return OrderWithStatus{}, fmt.Errorf("scan order: %w", err)
}

status := OrderStatus(rawStatus)
if !status.Valid() {
    return OrderWithStatus{}, fmt.Errorf("invalid order status %q", rawStatus)
}

order.Status = status
```

Если такой код встречается во многих запросах, легко забыть проверку или реализовать её по-разному. Собственный тип переносит преобразование и валидацию в одно место:

```go
if err := row.Scan(&order.ID, &order.Status); err != nil {
    return OrderWithStatus{}, fmt.Errorf("scan order: %w", err)
}
```

Тот же тип можно передать как аргумент запроса:

```go
_, err := db.ExecContext(ctx,
    "UPDATE orders SET status = $1 WHERE id = $2",
    order.Status,
    order.ID,
)
if err != nil {
    return fmt.Errorf("update order status: %w", err)
}
```

`database/sql` автоматически вызовет нужный интерфейс в каждом направлении.

## Чтение из базы: `sql.Scanner`

Интерфейс [`sql.Scanner`](https://pkg.go.dev/database/sql#Scanner) содержит один метод:

```go
type Scanner interface {
    Scan(src any) error
}
```

Когда адрес типа передан в [`Row.Scan`](https://pkg.go.dev/database/sql#Row.Scan) или [`Rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan), пакет вызывает этот метод со значением, полученным от драйвера.

Аргумент `src` имеет один из поддерживаемых типов:

| Тип `src` | Обычное назначение |
| :--- | :--- |
| `int64` | Целые числа |
| `float64` | Числа с плавающей точкой |
| `bool` | Логические значения |
| `string` | Текст |
| `[]byte` | Текстовое или бинарное представление |
| `time.Time` | Дата и время, если драйвер поддерживает преобразование |
| `nil` | SQL `NULL` |

Реализация `Scan` должна:

- учитывать, что драйвер может передать данные в разных базовых типах (например, текст может прийти и как `string`, и как `[]byte`), и уметь их обрабатывать;
- явно решить, допустим ли `NULL`;
- полностью проверить и преобразовать данные, и только в случае успеха записывать их в переменную;
- вернуть ошибку, если преобразование невозможно или приводит к потере данных.

Метод `Scan` обычно реализуют для указателя, поскольку его задача — изменить состояние самого объекта и записать в него прочитанные данные.

```go
func (s *OrderStatus) Scan(src any) error {
    var raw string

    switch value := src.(type) {
    case nil:
        return fmt.Errorf("scan order status: NULL is not allowed")
    case string:
        raw = value
    case []byte:
        raw = string(value)
    default:
        return fmt.Errorf(
            "scan order status: unsupported source type %T",
            src,
        )
    }

    status := OrderStatus(raw)
    if !status.Valid() {
        return fmt.Errorf("scan order status: invalid value %q", raw)
    }

    *s = status
    return nil
}
```

::: warning
Срез `[]byte`, переданный драйвером в `Scan`, может ссылаться на память, действительную только до следующего вызова `Scan`. Если собственный тип должен сохранить исходные байты, сначала скопируйте их через `bytes.Clone` или `append([]byte(nil), src...)`.
:::

## Запись в базу: `driver.Valuer`

Интерфейс [`driver.Valuer`](https://pkg.go.dev/database/sql/driver#Valuer) отвечает за обратное преобразование:

```go
type Valuer interface {
    Value() (driver.Value, error)
}
```

Когда собственный тип передаётся в `ExecContext`, `QueryContext` или `QueryRowContext` как аргумент, `database/sql` вызывает его метод `Value` и передаёт результат драйверу. Метод должен вернуть значение типа [`driver.Value`](https://pkg.go.dev/database/sql/driver#Value) и ошибку `error`.

`Value` не должен паниковать. Если текущее значение типа нельзя записать в базу данных, метод возвращает ошибку:

```go
func (s OrderStatus) Value() (driver.Value, error) {
    if !s.Valid() {
        return nil, fmt.Errorf("invalid order status %q", s)
    }

    return string(s), nil
}
```

В этом примере `Value` объявлен для значения, поскольку метод не изменяет объект. Это удобный выбор, но не требование интерфейса: метод допустимо объявить и для указателя, если в SQL-вызов передаётся указатель.

Ошибки, возвращённые `Value`, оборачиваются пакетом `database/sql`. Если собственная ошибка сохранена через `%w`, вызывающий код может распознать её через `errors.Is` после `ExecContext` или другого SQL-вызова.

## Когда интерфейсы не нужны

Собственный `Scanner` и `Valuer` полезны, когда преобразование является устойчивым свойством самого типа и повторяется в разных запросах. Но не каждый тип с новым именем нуждается в этих интерфейсах.

Например, простой ID часто сканируется без дополнительного кода:

```go
type UserID int64

var id UserID

if err := row.Scan(&id); err != nil {
    return 0, fmt.Errorf("scan user id: %w", err)
}
```

`database/sql` умеет выполнять безопасные преобразования в пользовательские типы с подходящим базовым типом. Реализация `Scanner` для `UserID` нужна только при дополнительной проверке или нестандартном представлении.

Хорошие кандидаты для собственных интерфейсов:

- значения с повторяющейся валидацией;
- JSON и другие сериализованные структуры;
- типы, которые драйвер возвращает не в форме доменной модели.

::: tip
Методы `Scan` и `Value` должны оставаться небольшими, предсказуемыми и независимыми от внешнего состояния. Их задача — преобразовать одно значение, а не скрыть внутри типа запросы к базе или бизнес-процесс.
:::
