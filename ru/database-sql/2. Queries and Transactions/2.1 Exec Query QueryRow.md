# Выбор метода выполнения запроса

Пакет [`database/sql`](https://pkg.go.dev/database/sql) предлагает три основных способа выполнить SQL-запрос: [`ExecContext`](https://pkg.go.dev/database/sql#DB.ExecContext), [`QueryContext`](https://pkg.go.dev/database/sql#DB.QueryContext) и [`QueryRowContext`](https://pkg.go.dev/database/sql#DB.QueryRowContext). Они различаются не типом SQL-команды, а формой ожидаемого результата: без строк, несколько строк или одна строка.

Правильный выбор метода определяет, какой объект вернётся из вызова, где нужно проверить ошибку и какие ресурсы потребуется освободить. Например, обычный `INSERT` выполняют через `ExecContext`, а `INSERT ... RETURNING id` уже возвращает строку и обычно требует `QueryRowContext`.

::: tip
Хорошей практикой считается по умолчанию использовать методы с суффиксом `Context`. Они позволяют ограничить время выполнения запроса и отменить его вместе с вызывающей операцией.
:::

## Как выбрать метод

Выбирать метод нужно по тому, какой результат должен получить Go-код:

| Ожидаемый результат | Метод | Возвращаемое значение |
| :--- | :--- | :--- |
| Строки результата не нужны | `ExecContext` | [`sql.Result`](https://pkg.go.dev/database/sql#Result) и `error` |
| Может быть несколько строк | `QueryContext` | [`*sql.Rows`](https://pkg.go.dev/database/sql#Rows) и `error` |
| Ожидается не более одной строки | `QueryRowContext` | [`*sql.Row`](https://pkg.go.dev/database/sql#Row) |

Главный ориентир — не первое слово SQL-запроса, а наличие и ожидаемое количество строк в его результате.

::: info
Связанные примеры руководства используют PostgreSQL и pgx, поэтому в SQL ниже применяются placeholder-параметры `$1`, `$2` и так далее. Правила сопоставления аргументов разобраны в статье [Параметризация SQL-запросов](/ru/database-sql/queries/query-parameters).
:::

## `ExecContext`

`ExecContext` выполняет команду, от которой не нужно получать строки.

```go
_, err := db.ExecContext(ctx,
    "UPDATE users SET active = $1 WHERE id = $2",
    false,
    userID,
)
if err != nil {
    return fmt.Errorf("deactivate user: %w", err)
}
```

Если база данных не смогла выполнить команду, `ExecContext` вернёт ошибку: например, из-за неверного SQL, нарушения ограничения, отмены контекста или проблемы с соединением.

::: warning
Если во время `ExecContext` произошёл тайм-аут или был отменён контекст, по одной ошибке нельзя всегда определить, успела ли база применить изменение. Команда могла завершиться на сервере до того, как приложение получило результат. Поэтому не повторяйте `INSERT`, `UPDATE` или `DELETE` автоматически, если операция не спроектирована как идемпотентная, то есть её повтор не создаёт дополнительного эффекта.
:::

При успешном выполнении метод возвращает `sql.Result`. Через него можно узнать количество затронутых строк или получить созданный идентификатор, если драйвер поддерживает такую возможность.

### `sql.Result`

| Метод | Назначение |
| :--- | :--- |
| [`RowsAffected`](https://pkg.go.dev/database/sql#Result.RowsAffected) | Возвращает количество строк, затронутых командой |
| [`LastInsertId`](https://pkg.go.dev/database/sql#Result.LastInsertId) | Возвращает целочисленный идентификатор, созданный базой |

Оба метода сами возвращают `error`, потому что их поддержка зависит от СУБД и драйвера.

#### `RowsAffected`

Количество затронутых строк позволяет проверить, нашла ли команда нужную запись:

```go
result, err := db.ExecContext(ctx,
    "DELETE FROM users WHERE id = $1",
    userID,
)
if err != nil {
    return fmt.Errorf("delete user: %w", err)
}

affected, err := result.RowsAffected()
if err != nil {
    return fmt.Errorf("get affected rows: %w", err)
}
if affected == 0 {
    return ErrUserNotFound
}

return nil
```

Точное значение `RowsAffected` определяется СУБД и драйвером. Особенно это важно для `UPDATE`: разные системы могут по-разному считать строки, которые подошли под условие, но фактически не изменились.

#### `LastInsertId` и `RETURNING`

Поддержка `LastInsertId` так же зависит от драйвера. В руководстве PostgreSQL и pgx — этот метод не возвращает созданный идентификатор. Для PostgreSQL используется `RETURNING` вместе с `QueryRowContext`:

```go
var id int64

err := db.QueryRowContext(ctx,
    "INSERT INTO users (email) VALUES ($1) RETURNING id",
    email,
).Scan(&id)
if err != nil {
    return 0, fmt.Errorf("insert user: %w", err)
}

return id, nil
```

Если другой драйвер документирует поддержку `LastInsertId`, всё равно нужно проверять ошибку самого метода.

## `QueryRowContext`

`QueryRowContext` используют, когда запрос должен вернуть не более одной строки: например, пользователя по уникальному ID, количество записей через `COUNT(*)` или идентификатор после `INSERT ... RETURNING`.

```go
row := db.QueryRowContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
    userID,
)

var user User
if err := row.Scan(&user.ID, &user.Email); err != nil {
    return User{}, fmt.Errorf("scan user: %w", err)
}

return user, nil
```

У этого метода нет отдельного возвращаемого значения `error`. Он всегда возвращает ненулевой `*sql.Row`, а ошибка выполнения запроса откладывается до вызова [`row.Scan`](https://pkg.go.dev/database/sql#Row.Scan).

Поэтому вызов обычно записывают короче:

```go
var count int

err := db.QueryRowContext(ctx,
    "SELECT COUNT(*) FROM users WHERE active = $1",
    true,
).Scan(&count)
if err != nil {
    return 0, fmt.Errorf("count active users: %w", err)
}
```

`QueryRowContext` не проверяет, что запрос действительно вернул только одну строку. Если строк несколько, `Scan` прочитает первую, а остальные будут отброшены. Запрос одной записи лучше строить по уникальному полю или другому условию, которое гарантирует ожидаемый результат.

### `sql.ErrNoRows`

Если `QueryRowContext` не нашёл ни одной строки, `Scan` возвращает [`sql.ErrNoRows`](https://pkg.go.dev/database/sql#ErrNoRows). Поскольку отсутствие записи — это часто нормальный сценарий, важно отличать такую ситуацию от системных ошибок при подключении, выполнении запроса или чтении данных.

```go
var user User

err := db.QueryRowContext(ctx,
    "SELECT id, email FROM users WHERE id = $1",
    userID,
).Scan(&user.ID, &user.Email)
if err != nil {
    if errors.Is(err, sql.ErrNoRows) {
        return User{}, ErrUserNotFound
    }

    return User{}, fmt.Errorf("find user: %w", err)
}

return user, nil
```

В этом примере код доступа к данным преобразует техническую ошибку `sql.ErrNoRows` в ошибку приложения `ErrUserNotFound`. HTTP-обработчик выше по стеку может сопоставить её со статусом `404 Not Found`, а другой вызывающий код — обработать по своим правилам.

Не превращайте любую ошибку `Scan` в «запись не найдена». Ошибка может означать отмену контекста, потерю соединения, несовместимость типов или проблему в SQL-запросе. Для сценария отсутствия строки проверяйте именно `errors.Is(err, sql.ErrNoRows)`.

## `QueryContext`

`QueryContext` используют, когда запрос может вернуть несколько строк. Метод возвращает `*sql.Rows`, который нужно последовательно прочитать и закрыть после использования.

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, email FROM users WHERE active = $1 ORDER BY id",
    true,
)
if err != nil {
    return nil, fmt.Errorf("query active users: %w", err)
}
defer rows.Close()
```

После успешного вызова результат читается через [`rows.Next`](https://pkg.go.dev/database/sql#Rows.Next), [`rows.Scan`](https://pkg.go.dev/database/sql#Rows.Scan) и финальную проверку [`rows.Err()`](https://pkg.go.dev/database/sql#Rows.Err). Полный безопасный шаблон разобран в следующей статье [Чтение результатов запроса](/ru/database-sql/queries/reading-rows).

В отличие от `QueryRowContext`, пустой результат `QueryContext` не приводит к [`sql.ErrNoRows`](https://pkg.go.dev/database/sql#ErrNoRows): он выражается отсутствием строк во время итерации.
