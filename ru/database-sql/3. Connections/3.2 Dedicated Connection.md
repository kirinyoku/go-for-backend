# Выделенное соединение

[`*sql.Conn`](https://pkg.go.dev/database/sql#Conn) представляет одно соединение драйвера, временно выделенное из пула `*sql.DB`. При прямом подключении к базе или использовании прокси в режиме пула сессий (*session pooling*) все запросы через этот объект попадают в одну серверную сессию, пока приложение не вызовет `Conn.Close`.

Большинству методов доступа к данным `*sql.Conn` не нужен: `*sql.DB` сам выбирает свободное соединение для каждого запроса. Выделенное соединение используют только тогда, когда несколько операций должны выполняться через одно соединение драйвера.

Сессия базы данных хранит состояние, связанное с конкретным соединением: настройки, временные объекты и некоторые виды блокировок. Обычные запросы через пул не обязаны попадать в одну и ту же сессию.

## Получение выделенного соединения (`db.Conn`)

Метод [`db.Conn`](https://pkg.go.dev/database/sql#DB.Conn) возвращает свободное соединение из пула или открывает новое:

```go
conn, err := db.Conn(ctx)
if err != nil {
    return fmt.Errorf("get dedicated connection: %w", err)
}
defer conn.Close()
```

Если пул достиг `MaxOpenConns` и все соединения заняты, `db.Conn(ctx)` будет ждать. Ожидание закончится, когда освободится соединение или отменится переданный контекст.

Контекст в `db.Conn(ctx)` относится к получению соединения. После успешного вызова он не закрывает `Conn` автоматически. Каждая следующая операция получает собственный контекст:

```go
if err := conn.PingContext(queryCtx); err != nil {
    return fmt.Errorf("ping dedicated connection: %w", err)
}
```

::: warning
После успешного `db.Conn` приложение обязано вызвать `Conn.Close`, даже если следующая операция завершилась ошибкой. Регистрируйте `defer conn.Close()` сразу после проверки ошибки.
:::

## Привязка к одной сессии

Состояние сессии состоит из настроек и объектов, которые база связывает с текущим соединением. Например, это может быть выбранная временная схема, часовой пояс, роль пользователя или служебное имя клиента.

Последовательные запросы через `*sql.DB` не обязаны попасть в одну сессию. Поэтому нельзя выполнить настройку на уровне соединения через `db.ExecContext`, а затем рассчитывать, что следующий вызов через `db` увидит это состояние.

Саму привязку можно безопасно увидеть на примере прямого подключения к PostgreSQL, который только читает данные:

```go
var firstBackendPID int
if err := conn.QueryRowContext(ctx,
    "SELECT pg_backend_pid()",
).Scan(&firstBackendPID); err != nil {
    return fmt.Errorf("read first backend pid: %w", err)
}

var secondBackendPID int
if err := conn.QueryRowContext(ctx,
    "SELECT pg_backend_pid()",
).Scan(&secondBackendPID); err != nil {
    return fmt.Errorf("read second backend pid: %w", err)
}
```

При прямом подключении оба значения обозначают один серверный процесс PostgreSQL, потому что вызовы выполнены через один `*sql.Conn`. При тех же вызовах через `*sql.DB` такое равенство не гарантируется. Прокси в режиме пула транзакций (*transaction pooling*) может назначать разные серверные процессы даже одному соединению клиента.

Привязка к сессии требуется для временных объектов, настроек сессии и некоторых видов блокировок. Но изменение такого состояния требует отдельной стратегии очистки, зависящей от драйвера и СУБД.

Если операция, привязанная к сессии, должна быть атомарной, транзакцию можно открыть через [`conn.BeginTx`](https://pkg.go.dev/database/sql#Conn.BeginTx):

```go
tx, err := conn.BeginTx(ctx, nil)
if err != nil {
    return fmt.Errorf("begin transaction on connection: %w", err)
}
defer tx.Rollback()

// Все операции tx выполняются на соединении, выделенном через conn.

if err := tx.Commit(); err != nil {
    return fmt.Errorf("commit transaction: %w", err)
}
```

Сначала нужно завершить `tx`, а затем закрыть `conn`. Но если состояние сессии не требуется, обычного `db.BeginTx` достаточно: транзакция и так привязана к одному соединению. Полный шаблон завершения разобран в статье [Управление транзакциями](/ru/database-sql/queries/transactions).

## Возврат соединения в пул (`Conn.Close`)

Метод [`conn.Close`](https://pkg.go.dev/database/sql#Conn.Close) возвращает соединение в пул. Обычно это не означает немедленное закрытие сетевого подключения.

`Close` ждёт завершения уже запущенных операций. Если запрос должен быть остановлен, сначала отмените его контекст, а затем закрывайте `Conn`.

После `Close` любые вызовы через этот объект завершаются ошибкой [`sql.ErrConnDone`](https://pkg.go.dev/database/sql#ErrConnDone). Сам объект `*sql.Conn` использовать повторно нельзя. При необходимости снова вызовите `db.Conn`: пул при этом может выдать то же физическое соединение через новый объект.
