# Подключение к базе данных

После выбора драйвера приложение может создать общий пул соединений [`*sql.DB`](https://pkg.go.dev/database/sql#DB) и проверить доступность базы. Сам пакет [`database/sql`](https://pkg.go.dev/database/sql) управляет стандартными объектами и пулом, а формат подключения и установку физического соединения делегирует выбранному драйверу.

Базовый код инициализации состоит из трёх шагов: подключить драйвер, создать [`*sql.DB`](https://pkg.go.dev/database/sql#DB), а затем проверить доступность базы через [`db.PingContext`](https://pkg.go.dev/database/sql#DB.PingContext). Ниже сначала показан основной путь через [`sql.Open`](https://pkg.go.dev/database/sql#Open), а затем — необязательная альтернатива через [`sql.OpenDB`](https://pkg.go.dev/database/sql#OpenDB) и типизированный [`driver.Connector`](https://pkg.go.dev/database/sql/driver#Connector). Важно не путать создание объекта `*sql.DB` с установленным сетевым соединением: это разные этапы.

## Регистрация выбранного драйвера

Драйвер подключается к программе как обычный Go-пакет, но часто используется пустой импорт (*blank import*):

```go
import (
    "database/sql"

    _ "path/to/driver" // замените на пакет выбранного драйвера
)
```

Символ `_` означает, что пакет импортируется ради побочного эффекта. Код приложения не обращается к экспортируемым идентификаторам драйвера напрямую, но при инициализации пакет регистрирует себя внутри `database/sql`.

После регистрации `database/sql` знает, какой драйвер использовать, когда приложение вызовет `sql.Open("driver-name", dsn)`.

Конкретный путь импорта и имя для `sql.Open` указаны в документации выбранного драйвера. Они не стандартизируются пакетом `database/sql`.

## Строка подключения (DSN)

Второй важный элемент подключения — DSN (*data source name*). Это строка с параметрами доступа к базе: адресом, именем базы, пользователем, паролем, режимом TLS, тайм-аутом и другими настройками.

Формат DSN зависит от драйвера. `database/sql` передаёт эту строку ниже, но не разбирает её как универсальный стандарт.

Примеры форматов, которые могут встречаться у разных драйверов:

```text
postgres://user:password@localhost:5432/app?sslmode=disable

postgres://user:password@db.example.com:5432/app?sslmode=verify-full&sslrootcert=/etc/app/ca.crt

user:password@tcp(localhost:3306)/app?parseTime=true

file:app.db?cache=shared
```

::: warning
`sslmode=disable` не шифрует соединение и не проверяет сервер. Используйте его только в контролируемом локальном окружении. Для удалённого PostgreSQL настройте проверку сертификата и имени сервера по документации драйвера; в показанном формате DSN для этого используется `verify-full`.
:::

В реальном приложении DSN обычно приходит из конфигурации: переменных окружения, файла настроек или хранилища секретов. Не стоит зашивать пароль и адрес базы прямо в код.

## Создание пула через `sql.Open`

Функция [`sql.Open`](https://pkg.go.dev/database/sql#Open) создаёт новый экземпляр [`*sql.DB`](https://pkg.go.dev/database/sql#DB):

```go
db, err := sql.Open("driver-name", dsn)
if err != nil {
    return nil, fmt.Errorf("open db: %w", err)
}
```

Первый аргумент — имя зарегистрированного драйвера. Второй аргумент — DSN в формате этого драйвера.

Несмотря на название, `sql.Open` не обязательно открывает сетевое соединение сразу. Он может только проверить имя драйвера и базовую корректность аргументов. Поэтому успешный вызов `sql.Open` ещё не означает, что база доступна, пароль правильный, сеть работает, а нужная база существует.

Не используйте успешный `sql.Open` как проверку подключения. Если база недоступна, ошибка часто появится только при первом запросе или при явном вызове `PingContext`.

## Альтернатива: `sql.OpenDB`

Для базового подключения обычно достаточно `sql.Open`. `sql.OpenDB` используют, когда приложение получает [`driver.Connector`](https://pkg.go.dev/database/sql/driver#Connector) для типизированной конфигурации или дополнительных настроек, недоступных через DSN.

Например, pgx позволяет разобрать конфигурацию, получить `driver.Connector` и передать его стандартной библиотеке:

```go
config, err := pgx.ParseConfig(os.Getenv("DATABASE_URL"))
if err != nil {
    return nil, fmt.Errorf("parse database config: %w", err)
}

connector := stdlib.GetConnector(*config)
db := sql.OpenDB(connector)
```

Здесь используются сторонние пакеты `github.com/jackc/pgx/v5` и `github.com/jackc/pgx/v5/stdlib`. `sql.OpenDB`, как и `sql.Open`, не подтверждает доступность сервера: после настройки пула всё равно нужен `PingContext`. Конкретный драйвер может также предоставить вспомогательную функцию, сразу возвращающую `*sql.DB`; выбирайте документированный им путь и сохраняйте явное владение пулом.

## Когда открываются соединения

`*sql.DB` — это не одно соединение, а управляющий объект с пулом соединений. Он открывает реальные соединения по мере необходимости: при `PingContext`, `ExecContext`, `QueryContext`, `QueryRowContext`, начале транзакции или получении выделенного соединения.

Такая модель позволяет создать `*sql.DB` один раз при старте приложения и передавать его дальше как зависимость. Сам объект безопасен для конкурентного использования несколькими горутинами и сам управляет открытыми и свободными соединениями.

::: tip
Создавайте один `*sql.DB` на одну конфигурацию подключения и переиспользуйте его. Отдельный пул оправдан, когда отличаются DSN, учётные данные или требования к изоляции нагрузки; не создавайте пул для каждого обработчика или метода доступа к данным.
:::

## Проверка соединения

[`db.PingContext`](https://pkg.go.dev/database/sql#DB.PingContext) проверяет, что подключение к базе живо, и при необходимости устанавливает соединение.

Обычно его вызывают при старте приложения, чтобы обнаружить проблему сразу, а не во время первого пользовательского запроса.

Проверку удобно включить в функцию, которая создаёт пул. Если `PingContext` завершился ошибкой, функция закрывает ещё не переданный вызывающему коду `*sql.DB`:

```go
func openDB(
    ctx context.Context,
    driverName string,
    dsn string,
) (*sql.DB, error) {
    db, err := sql.Open(driverName, dsn)
    if err != nil {
        return nil, fmt.Errorf("open db: %w", err)
    }

    pingCtx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()

    if err := db.PingContext(pingCtx); err != nil {
        if closeErr := db.Close(); closeErr != nil {
            return nil, errors.Join(
                fmt.Errorf("ping db: %w", err),
                fmt.Errorf("close db after failed ping: %w", closeErr),
            )
        }

        return nil, fmt.Errorf("ping db: %w", err)
    }

    return db, nil
}
```

::: info
Успешный `PingContext` подтверждает доступность только в момент проверки, поэтому ошибки всё равно нужно обрабатывать при каждой операции.
:::

## Время жизни и закрытие пула

`*sql.DB` рассчитан на долгий жизненный цикл. Обычно его закрывают один раз: при завершении приложения, остановке теста или освобождении временного окружения.

Метод [`db.Close`](https://pkg.go.dev/database/sql#DB.Close) запрещает запуск новых запросов и ждёт завершения уже начатых операций. Поэтому закрывать `*sql.DB` внутри обработчика или после каждого метода доступа к данным нельзя.

Если закрыть `*sql.DB` слишком рано, последующие запросы к базе начнут завершаться ошибками. Если не закрыть его при завершении короткоживущей программы или теста, ресурсы могут освобождаться позже, чем ожидается.
