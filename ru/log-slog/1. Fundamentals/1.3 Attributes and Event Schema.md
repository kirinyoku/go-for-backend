# Атрибуты и схема событий

Атрибут связывает имя поля с его значением. В событии о завершении HTTP-запроса атрибутами могут быть метод, маршрут, статус и длительность:

```go
logger.Info("request completed",
    "method", "GET",
    "route", "/users/{id}",
    "status", 200,
    "duration", 37*time.Millisecond,
)
```

Каждая отдельная запись содержит данные одного запроса, но набор полей остаётся одинаковым. Благодаря этому события можно искать и сравнивать по известным ключам. По сути, атрибуты образуют небольшую схему, за устойчивость которой отвечает приложение.

## Пары «ключ — значение»

Метод [`Logger.Info`](https://pkg.go.dev/log/slog#Logger.Info) принимает после сообщения аргументы типа `...any`. В простейшем случае они читаются попарно: строковый ключ называет следующее за ним значение.

```go
logger.Info("user loaded",
    "user_id", userID,
    "from_cache", fromCache,
)
```

Ключ описывает смысл поля, поэтому обычно известен ещё при написании кода. Конкретный идентификатор пользователя передаётся как значение постоянного ключа `user_id`, а не становится ключом сам:

```go
// Неправильно: имя поля меняется вместе с данными.
logger.Info("user loaded", strconv.FormatInt(userID, 10), true)

// Правильно: имя поля стабильно, меняется только значение.
logger.Info("user loaded", "user_id", userID)
```

Динамические ключи затрудняют поиск и могут создавать неограниченное количество разных полей в системе обработки логов. Для изменяемых данных следует заранее выбрать постоянное имя.

## Явные атрибуты

Вместо чередующихся аргументов можно передавать готовые значения типа [`slog.Attr`](https://pkg.go.dev/log/slog#Attr). Каждый такой атрибут хранит строковый ключ и [`slog.Value`](https://pkg.go.dev/log/slog#Value):

```go
logger.Info("request completed",
    slog.String("method", "GET"),
    slog.String("route", "/users/{id}"),
    slog.Int("status", 200),
    slog.Duration("duration", 37*time.Millisecond),
)
```

Явная форма длиннее, зато ключ и значение находятся в одном выражении, а ожидаемый тип поля виден сразу. Пары `"status", 200` и `slog.Int("status", 200)` описывают одинаковый атрибут, поэтому обе формы допустимы. Внутри одного события лучше выбирать ту форму, при которой порядок аргументов проще проверить.

Для распространённых типов пакет предоставляет отдельные функции:

| Функция | Тип значения | Пример поля |
| :--- | :--- | :--- |
| [`slog.String`](https://pkg.go.dev/log/slog#String) | `string` | метод или маршрут |
| [`slog.Int`](https://pkg.go.dev/log/slog#Int) | `int` | HTTP-статус |
| [`slog.Duration`](https://pkg.go.dev/log/slog#Duration) | `time.Duration` | длительность операции |
| [`slog.Any`](https://pkg.go.dev/log/slog#Any) | произвольный Go-тип | ошибка или собственный тип |

Функцию выбирают по типу значения, а не по тому, как поле должно выглядеть в готовой строке. Например, статус полезнее передать через `slog.Int`, чем заранее преобразовать в `"200"`. Так обработчик получает число и может сохранить его числом в формате, который это поддерживает.

`slog.Any` нужен, когда для значения нет отдельной функции:

```go
logger.Error("database query failed",
    slog.String("operation", "find_user"),
    slog.Any("err", err),
)
```

Передавать через `slog.Any` крупную структуру целиком обычно не стоит: вместе с полезными данными в запись могут попасть лишние или чувствительные поля. Для события лучше выбирать только те значения, которые действительно нужны для диагностики.

## Стабильная схема события

Для событий одного вида следует заранее определить имена и типы полей. Например, завершённый HTTP-запрос может иметь такую схему:

| Ключ | Тип | Пример |
| :--- | :--- | :--- |
| `method` | `string` | `GET` |
| `route` | `string` | `/users/{id}` |
| `status` | `int` | `200` |
| `duration` | `time.Duration` | `37ms` |

Из этой схемы следуют несколько практических правил:

- используйте одно имя поля: не чередуйте `status`, `status_code` и `code`;
- сохраняйте тип: `status` не должен в одних событиях быть числом, а в других строкой;
- записывайте шаблон маршрута `/users/{id}`, а не фактический путь `/users/42`;
- не используйте пользовательские данные как имена ключей.

Пакет `log/slog` не проверяет такую схему и не требует заранее объявлять её. Дисциплина нужна на уровне приложения: если одинаковое событие постоянно меняет ключи или их типы, обработка структурированных записей снова превращается в разбор множества частных случаев.

## Повторяющиеся ключи

`log/slog` позволяет добавить один ключ несколько раз:

```go
logger.Info("request completed",
    "status", 200,
    "status", 500,
)
```

`TextHandler` выведет оба значения:

```text
level=INFO msg="request completed" status=200 status=500
```

Для приложения смысл такой записи неоднозначен: неизвестно, какой статус считать итоговым. Обработчики и внешние системы могут по-разному работать с повторяющимися ключами, поэтому в одной области события лучше использовать уникальные имена. Если значения описывают разные стороны операции, различие нужно выразить в ключах, например `status` и `upstream_status`.

## Ошибочные аргументы и `!BADKEY`

Короткая форма требует, чтобы у каждого ключа было значение. Если последний аргумент остался без пары, `slog` не завершает программу, а добавляет его под служебным ключом `!BADKEY`:

```go
logger.Info("request completed",
    "status", 200,
    "duration",
)
```

Примерный текстовый вывод без времени:

```text
level=INFO msg="request completed" status=200 !BADKEY=duration
```

Такой же ключ появится, если передать значение там, где `slog` ожидает строковый ключ или готовый `slog.Attr`:

```go
userID := 42
logger.Info("user %d", userID)
```

`slog.Info` не обрабатывает `%d` по правилам `fmt.Printf`. Сообщение останется равным `user %d`, а `userID` попадёт в атрибут `!BADKEY`. Правильный вызов отделяет постоянное сообщение от данных:

```go
logger.Info("user loaded", "user_id", userID)
```

Начиная с Go 1.22 команда [`go vet`](https://pkg.go.dev/cmd/vet) проверяет прямые вызовы `log/slog` с чередующимися ключами и значениями. Она сообщает о ключе без значения и об аргументе неподходящего типа в позиции ключа:

```sh
go vet ./...
```
