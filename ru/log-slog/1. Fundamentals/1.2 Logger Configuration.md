# Создание и настройка логгера

Функции уровня пакета, такие как [`slog.Info`](https://pkg.go.dev/log/slog#Info), позволяют начать логирование без предварительной настройки. В приложении логгер обычно создают явно, чтобы в одном месте выбрать представление записей и поток, в который они попадут.

Настройка выполняется при запуске приложения. Остальной код получает готовый `*slog.Logger` и описывает события, не решая при каждом вызове, куда и в каком виде их записывать.

## Создание собственного логгера

Для создания логгера сначала нужен обработчик. Например, [`slog.NewJSONHandler`](https://pkg.go.dev/log/slog#NewJSONHandler) создаёт обработчик, который записывает события в формате JSON:

```go
handler := slog.NewJSONHandler(os.Stdout, nil)
logger := slog.New(handler)

logger.Info("service started", "address", ":8080")
```

Результат будет выглядеть примерно так:

```json
{"time":"2026-07-22T10:15:42.123+02:00","level":"INFO","msg":"service started","address":":8080"}
```

В этой связке [`slog.Logger`](https://pkg.go.dev/log/slog#Logger) принимает события от приложения. Обработчик, представленный интерфейсом [`slog.Handler`](https://pkg.go.dev/log/slog#Handler), решает, какие записи обрабатывать и как представить их в выбранном потоке вывода.

Функция [`slog.New`](https://pkg.go.dev/log/slog#New) связывает эти части и возвращает новый `*slog.Logger`. Обработчик не может быть равен `nil`: `slog.New(nil)` приводит к панике. Значение `nil` во втором аргументе `NewJSONHandler` имеет другой смысл — обработчик использует стандартные настройки.

## Текстовое и JSON-представление

Стандартная библиотека предоставляет два готовых обработчика. Сравним [`slog.NewTextHandler`](https://pkg.go.dev/log/slog#NewTextHandler) и уже знакомый `slog.NewJSONHandler` на одном событии:

```go
textLogger := slog.New(slog.NewTextHandler(os.Stdout, nil))
jsonLogger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

textLogger.Info("request completed", "method", "GET", "status", 200)
jsonLogger.Info("request completed", "method", "GET", "status", 200)
```

[`slog.TextHandler`](https://pkg.go.dev/log/slog#TextHandler) выводит поля в форме `key=value`:

```text
time=2026-07-22T10:15:42.123+02:00 level=INFO msg="request completed" method=GET status=200
```

`slog.JSONHandler` выводит каждое событие как отдельный JSON-объект:

```json
{"time":"2026-07-22T10:15:42.123+02:00","level":"INFO","msg":"request completed","method":"GET","status":200}
```

Оба обработчика получили одно и то же структурированное событие. Различается только его представление: `TextHandler` удобен для чтения в терминале, а JSON обычно проще передавать системам, которые разбирают и индексируют поля. Вывод `JSONHandler` представляет собой последовательность объектов, разделённых переводами строк, а не один JSON-массив.

Выбор обработчика не меняет прикладные вызовы. Компонент продолжает передавать сообщение и атрибуты через методы логгера независимо от того, как событие будет выглядеть после обработки.

## Поток вывода

Конструкторы `TextHandler` и `JSONHandler` принимают [`io.Writer`](https://pkg.go.dev/io#Writer). В обычном приложении это часто [`os.Stdout`](https://pkg.go.dev/os#Stdout) или [`os.Stderr`](https://pkg.go.dev/os#Stderr), но `slog` не привязывает обработчик к конкретному потоку:

```go
func newLogger(out io.Writer) *slog.Logger {
    handler := slog.NewJSONHandler(out, nil)
    return slog.New(handler)
}
```

Такую функцию можно вызвать с `os.Stdout` при запуске приложения, а в тесте передать ей `bytes.Buffer`. Сам обработчик не становится владельцем потока. У `slog.Logger` нет методов `Close` или `Flush`, поэтому открытый файл или другой ресурс закрывает тот код, который его создал.

::: tip
Если среда запуска уже собирает стандартный вывод процесса, обычно достаточно писать туда, а хранение и ротацию поручить внешней системе. Выбор между `os.Stdout` и `os.Stderr` определяется правилами этой среды.
:::

## Передача логгера компонентам

Базовый логгер удобно создать один раз в `main`, а затем передать компонентам как обычную зависимость:

```go
type UserService struct {
    logger *slog.Logger
}

func NewUserService(logger *slog.Logger) *UserService {
    return &UserService{logger: logger}
}

func (s *UserService) CreateUser(userID int64) {
    // Основная логика создания пользователя опущена.
    s.logger.Info("user created", "user_id", userID)
}

func main() {
    handler := slog.NewJSONHandler(os.Stdout, nil)
    logger := slog.New(handler)

    users := NewUserService(logger)
    users.CreateUser(42)
}
```

Только `main` знает, что приложение пишет JSON в `os.Stdout`. `UserService` зависит от `*slog.Logger`, но не выбирает формат и поток вывода. Благодаря этому настройка остаётся общей для приложения и не дублируется в каждом компоненте.

Один `*slog.Logger` можно безопасно использовать одновременно из нескольких горутин. Создавать отдельный базовый логгер для каждого HTTP-запроса не нужно. Позже от него можно получить производные логгеры с общими атрибутами, не меняя исходную настройку.

## Логгер по умолчанию

Функция [`slog.Default`](https://pkg.go.dev/log/slog#Default) возвращает логгер по умолчанию. Функции уровня пакета используют именно его, поэтому следующие вызовы записывают события через один логгер:

```go
slog.Info("service started")

logger := slog.Default()
logger.Info("service ready")
```

Такой вариант удобен для небольших программ и первых примеров. В компонентах приложения явно переданный `*slog.Logger` лучше показывает зависимость и не скрывает место, где выбраны обработчик, представление и поток вывода.

## Совместимость с пакетом `log`

Часть существующего кода и некоторые библиотеки по-прежнему используют пакет [`log`](https://pkg.go.dev/log). `log/slog` позволяет направить их сообщения тому же обработчику, но способ настройки зависит от того, принимает код отдельный `*log.Logger` или вызывает функции уровня пакета.

### Отдельный `*log.Logger`

Если библиотека принимает [`*log.Logger`](https://pkg.go.dev/log#Logger), для неё можно создать совместимый экземпляр через [`slog.NewLogLogger`](https://pkg.go.dev/log/slog#NewLogLogger):

```go
legacyLogger := slog.NewLogLogger(
    logger.Handler(),
    slog.LevelWarn,
)

legacyLogger.Printf("connection attempt failed: %v", err)
```

Каждый вызов этого логгера создаёт событие уровня `Warn` и передаёт его обработчику `logger`. Пакет `log` формирует только текстовое сообщение, поэтому аргументы `Printf` не становятся отдельными атрибутами `slog`.

Созданный экземпляр не добавляет собственные дату, время и префикс: служебные поля формирует обработчик `slog`. Такой логгер передают только тому компоненту, которому требуется старый API.

### Функции `log.Print` и `log.Printf`

Если код вызывает функции уровня пакета, [`slog.SetDefault`](https://pkg.go.dev/log/slog#SetDefault) направит сообщения стандартного логгера `log` обработчику настроенного `slog.Logger`:

```go
slog.SetDefault(logger)
slog.SetLogLoggerLevel(slog.LevelWarn)

log.Printf("legacy subsystem is unavailable: %v", err)
```

Функция [`slog.SetLogLoggerLevel`](https://pkg.go.dev/log/slog#SetLogLoggerLevel), доступная начиная с Go 1.22, задаёт уровень, с которым такие сообщения попадут в `slog`. Без этого вызова используется уровень `Info`.

Обе настройки относятся ко всему процессу, поэтому их выполняют один раз при запуске приложения. Они не изменяют экземпляры, созданные через [`log.New`](https://pkg.go.dev/log#New): если библиотека принимает отдельный `*log.Logger`, ей нужно явно передать результат `slog.NewLogLogger`.
