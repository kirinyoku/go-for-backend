# От строки к структурированному событию

Пакет [`log/slog`](https://pkg.go.dev/log/slog), доступный с Go 1.21, предназначен для структурированного логирования. Вместо готовой строки приложение передаёт ему событие: сообщение, уровень и именованные атрибуты. Такие записи можно отправлять в централизованную систему сбора логов и искать, например, по идентификатору запроса, маршруту или коду ответа. Если формат вывода и система сбора сохраняют атрибуты, для такого поиска не приходится разбирать текст сообщения.

## Текстовая запись через `log.Printf`

Для простой текстовой записи подходит [`log.Printf`](https://pkg.go.dev/log#Printf):

```go
method := http.MethodGet
route := "/users/{id}"
status := http.StatusOK
duration := 37 * time.Millisecond

log.Printf(
    "request completed: method=%s route=%s status=%d duration=%s",
    method,
    route,
    status,
    duration,
)
```

При стандартных настройках результат будет выглядеть примерно так:

```text
2026/07/22 10:15:42 request completed: method=GET route=/users/{id} status=200 duration=37ms
```

Строка понятна человеку: по ней легко увидеть, какой запрос завершился и сколько времени он занял. Для небольшого приложения или редких диагностических сообщений этого может быть достаточно.

Ограничение проявляется, когда с записью нужно работать как с данными. `log.Printf` форматирует аргументы и записывает результат как одно сообщение. В готовой записи уже не сохранены границы полей и их исходные типы: `status` представлен фрагментом текста, а `duration` — результатом форматирования `time.Duration`.

## Структурированная запись через `slog.Info`

Функция [`slog.Info`](https://pkg.go.dev/log/slog#Info) принимает сообщение, а затем отдельные пары «ключ — значение»:

```go
slog.Info("request completed",
    "method", method,
    "route", route,
    "status", status,
    "duration", duration,
)
```

Аргументы после сообщения читаются последовательно: строковый ключ называет следующее за ним значение. Так к одному событию можно добавить несколько независимых атрибутов.

При стандартной настройке вывод изменится незначительно:

```text
2026/07/22 10:15:42 INFO request completed method=GET route=/users/{id} status=200 duration=37ms
```

После метки времени в записи можно выделить три основные части:

| Часть | Пример | Назначение |
| :--- | :--- | :--- |
| Уровень | `INFO` | Показывает значимость события. |
| Сообщение | `request completed` | Кратко называет, что произошло. |
| Атрибуты | `method`, `route`, `status`, `duration` | Описывают контекст и значимые детали конкретного события. |

Главное отличие возникло ещё до появления строки. Приложение передало статус как число, длительность как `time.Duration`, а метод и маршрут как отдельные строки. `slog` получает эти значения раздельно и в таком виде передаёт их выбранному способу обработки записи.

## Разделение сообщения и атрибутов

Сообщение отвечает на вопрос «что произошло», а атрибуты описывают конкретный случай. Если заранее собрать всю запись через [`fmt.Sprintf`](https://pkg.go.dev/fmt#Sprintf), данные снова окажутся внутри сообщения:

```go
message := fmt.Sprintf(
    "%s %s completed with status %d in %s",
    method,
    route,
    status,
    duration,
)
slog.Info(message)
```

В таком вызове время и уровень по-прежнему остаются служебными частями события, но данные HTTP-запроса уже скрыты внутри сообщения. Структурированный вариант сохраняет их отдельно:

```go
slog.Info("request completed",
    "method", http.MethodGet,
    "route", "/users/{id}",
    "status", http.StatusOK,
    "duration", 37*time.Millisecond,
)
```

Сообщение `request completed` остаётся одинаковым для всех завершённых запросов. Метод, маршрут, статус и длительность меняются от случая к случаю, поэтому становятся атрибутами.

::: tip
Если значение часто меняется, но не меняет смысл самого события, обычно ему место в атрибуте. При этом превращать в поля каждое слово не нужно. Сообщение по-прежнему должно оставаться коротким и понятным, а отдельно передаются данные, по которым запись потребуется искать, фильтровать или связывать с другими записями.
:::

::: info
`slog.Info` не форматирует сообщение по правилам `fmt`. Оставляйте в нём постоянный текст, а изменяемые значения передавайте атрибутами.
:::

## Структура события и формат вывода

Пары `key=value` можно собрать и через обычный текстовый логгер:

```go
log.Printf(
    "msg=request_completed method=%s status=%d",
    method,
    status,
)
```

Такая строка выглядит структурированной, но для пакета `log` она остаётся одним текстовым значением. Разделение на `msg`, `method` и `status` существует только как соглашение, которое должен знать внешний разборщик.

В случае `slog.Info("request completed", "status", status)` сообщение, ключ и числовое значение существуют отдельно до формирования вывода. Поэтому структурированное логирование определяется не наличием фигурных скобок, JSON или пар `key=value`, а способом, которым приложение передало событие логгеру.

Дальше событие получает обработчик ([`slog.Handler`](https://pkg.go.dev/log/slog#Handler)) — компонент `slog`, который решает, как представить запись или передать её другой системе. Например, [`slog.JSONHandler`](https://pkg.go.dev/log/slog#JSONHandler) может вывести предыдущий вызов в виде JSON:

```json
{"time":"2026-07-22T10:15:42.123456789+02:00","level":"INFO","msg":"request completed","status":200}
```

Здесь `status` остался числом, а не фрагментом сообщения. JSON — только один из возможных форматов: структура появилась раньше, в момент вызова `slog.Info`.

## Обработка структурированных записей

Если обработчик и система обработки логов сохраняют именованные поля, с ними можно работать без разбора текста:

- отобрать по условию `status >= 500`;
- сгруппировать по значению `route`;
- сравнить длительность однотипных запросов.

Сам `log/slog` не предоставляет хранилище или язык запросов. Его задача заканчивается после передачи сообщения, уровня и атрибутов обработчику. Формат, хранение, поиск и доставка записей зависят от выбранной инфраструктуры.

Структура события возникает уже в месте вызова: сообщение называет, что произошло, а атрибуты сохраняют значения конкретного случая. Поэтому до выбора формата вывода важно определить, какая часть записи постоянна, а какие данные должны остаться отдельными полями.
