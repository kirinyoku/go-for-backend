# Уровни и фильтрация

После создания логгера можно столкнуться с неожиданным результатом:

```go
logger.Debug("configuration loaded")
logger.Info("service started")
```

При стандартных настройках в вывод попадёт только `service started`. Запись `Debug` не потерялась из-за ошибки: обработчик по умолчанию пропускает события начиная с уровня `Info`.

Уровень описывает значимость события, а порог обработчика определяет, следует ли его записывать. Благодаря этому подробную диагностику можно отключить, не удаляя соответствующие вызовы из кода.

## Стандартные уровни

Для обычных событий у [`slog.Logger`](https://pkg.go.dev/log/slog#Logger) есть четыре метода:

```go
logger.Debug("cache lookup started")
logger.Info("request completed")
logger.Warn("cache unavailable")
logger.Error("database query failed")
```

Уровень удобнее выбирать по результату операции, а не только по формулировке сообщения:

| Уровень | Когда подходит |
| :--- | :--- |
| `Debug` | Подробность нужна для диагностики, но обычно не важна при штатной работе. |
| `Info` | Произошло ожидаемое значимое событие приложения. |
| `Warn` | Возникла проблема, но операция или сервис продолжают работу. |
| `Error` | Операция не получила ожидаемый результат из-за ошибки. |

Например, отсутствие значения в кеше может быть событием `Debug`, если приложение штатно обращается к базе данных. Если кеш полностью недоступен, но запасной путь работает, подходит `Warn`. Если запрос к базе тоже завершился ошибкой и операцию нельзя продолжить, событие относится к `Error`.

Уровень отражает результат с точки зрения приложения. Ответ HTTP 4xx не обязан быть событием `Error`: ошибка валидации или отсутствие ресурса могут быть ожидаемым результатом клиентского запроса. Важнее определить понятную политику проекта и применять её одинаково.

## Порог обработчика

Встроенные уровни образуют упорядоченную шкалу:

```text
Debug < Info < Warn < Error
```

Обработчик пропускает событие, если его уровень не ниже настроенного порога. Поэтому чем выше порог, тем меньше записей попадёт в вывод:

| Порог | Записываемые уровни |
| :--- | :--- |
| `Debug` | `Debug`, `Info`, `Warn`, `Error` |
| `Info` | `Info`, `Warn`, `Error` |
| `Warn` | `Warn`, `Error` |
| `Error` | `Error` |

Порог задаётся через поле `Level` структуры [`slog.HandlerOptions`](https://pkg.go.dev/log/slog#HandlerOptions):

```go
handler := slog.NewTextHandler(os.Stdout, &slog.HandlerOptions{
    Level: slog.LevelWarn,
})
logger := slog.New(handler)

logger.Debug("cache lookup started")
logger.Info("request completed")
logger.Warn("cache unavailable")
logger.Error("database query failed")
```

Первые две записи будут отброшены. В выводе останутся только `Warn` и `Error`:

```text
time=... level=WARN msg="cache unavailable"
time=... level=ERROR msg="database query failed"
```

Если `HandlerOptions.Level` равен `nil`, `TextHandler` и `JSONHandler` используют порог `Info`. Именно поэтому переданный в обработчик `nil` из предыдущих примеров скрывал записи уровня `Debug`.

Чтобы включить подробную диагностику, достаточно опустить порог до [`slog.LevelDebug`](https://pkg.go.dev/log/slog#LevelDebug):

```go
handler := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
    Level: slog.LevelDebug,
})
logger := slog.New(handler)
```

Выбирать порог следует при настройке приложения, а не внутри отдельных компонентов. Тогда один и тот же прикладной код может писать больше записей в среде разработки и меньше — при обычной эксплуатации.

## `Error` и управление программой

Метод [`Logger.Error`](https://pkg.go.dev/log/slog#Logger.Error) только создаёт событие уровня `Error`. Он не возвращает значение типа `error`, не меняет уже существующую ошибку и не завершает программу.

Если после ошибки операцию нужно прервать, управление остаётся явным:

```go
if err != nil {
    logger.Error("configuration loading failed", "err", err)
    return err
}
```

Запись события и `return` выполняют разные задачи. Первая сохраняет диагностические данные, второй передаёт ошибку вызывающему коду. Сам вызов `logger.Error` без `return` позволил бы функции продолжить работу.

В `log/slog` нет аналогов `Fatal` и `Panic`. Решение вызвать `return`, `panic` или `os.Exit` принимает код приложения отдельно от логирования. Такое разделение делает завершение программы заметным в потоке управления, а не скрывает его внутри метода логгера.

## Динамический порог через `LevelVar`

Обычное значение `slog.LevelWarn` задаёт постоянный порог. Если уровень требуется менять во время работы приложения, используется [`slog.LevelVar`](https://pkg.go.dev/log/slog#LevelVar):

```go
var minimumLevel slog.LevelVar // Нулевое значение соответствует Info.

handler := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
    Level: &minimumLevel,
})
logger := slog.New(handler)

logger.Debug("configuration loaded") // Запись отбрасывается.

minimumLevel.Set(slog.LevelDebug)
logger.Debug("configuration loaded") // Запись проходит.
```

Обработчик обращается к `LevelVar` при проверке каждого события, поэтому `Set` влияет на уже созданный логгер. `LevelVar` безопасен для конкурентного использования. Откуда поступает команда изменить порог — из сигнала, служебного HTTP-маршрута или другого механизма — решает само приложение.

Для большинства сервисов достаточно выбрать порог при запуске. Динамическое изменение полезно, когда подробную диагностику нужно временно включить без перезапуска процесса.

## Собственные уровни

Тип [`slog.Level`](https://pkg.go.dev/log/slog#Level) является целым числом, а встроенные значения расположены с промежутками: `Debug` равен −4, `Info` — 0, `Warn` — 4, `Error` — 8. При необходимости между ними можно определить дополнительный уровень и записать событие через [`Logger.Log`](https://pkg.go.dev/log/slog#Logger.Log):

```go
const LevelNotice = slog.LevelInfo + 2

logger.Log(ctx, LevelNotice, "configuration reloaded")
```

Стандартный обработчик обозначит такой уровень как `INFO+2`. Большинству приложений достаточно четырёх встроенных уровней: собственное значение имеет смысл только вместе с общей политикой и понятным преобразованием для системы обработки логов.

Уровень помогает отделить подробную диагностику от событий, требующих внимания, а порог позволяет управлять объёмом вывода без изменения прикладных вызовов. При этом уровень остаётся свойством записи и не заменяет обычное управление ошибками в коде.
