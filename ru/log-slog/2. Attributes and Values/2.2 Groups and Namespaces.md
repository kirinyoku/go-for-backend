# Группы и пространства имён

В одном событии могут встретиться поля с одинаковым смыслом, но относящиеся к разным объектам. Например, у HTTP-запроса и пользователя есть собственные идентификаторы:

```go
logger.Info("request authorized",
    "id", requestID,
    "id", userID,
)
```

Такая запись содержит два одинаковых ключа, поэтому получателю придётся угадывать, к чему относится каждое значение. Имена `request_id` и `user_id` решают простую коллизию, но когда у каждого объекта несколько полей, удобнее объединить их в отдельные группы.

## Группа внутри события

Функция [`slog.Group`](https://pkg.go.dev/log/slog#Group) собирает связанные атрибуты под одним ключом:

```go
logger.Info("request authorized",
    slog.Group("request",
        slog.String("id", requestID),
        slog.String("method", "GET"),
    ),
    slog.Group("user",
        slog.Int64("id", userID),
        slog.String("role", "admin"),
    ),
)
```

Внутри каждой группы можно использовать короткие ключи `id`, потому что полное имя включает пространство `request` или `user`. Группа действует только в этом событии и сама является атрибутом.

Аргументы `slog.Group` разбираются так же, как аргументы методов логгера: допустимы пары `"key", value` и готовые значения `slog.Attr`. Явные атрибуты помогают визуально отделить содержимое одной группы от другой.

## Текстовое и JSON-представление

Группа описывает логическую структуру события, а её внешний вид зависит от обработчика. Для краткости в следующих примерах опущено время. `TextHandler` соединяет имя группы и вложенный ключ точкой:

```text
level=INFO msg="request authorized" request.id=req-7 request.method=GET user.id=42 user.role=admin
```

`JSONHandler` представляет группы вложенными объектами:

```json
{"level":"INFO","msg":"request authorized","request":{"id":"req-7","method":"GET"},"user":{"id":42,"role":"admin"}}
```

В обоих случаях `request.id` и `user.id` остаются разными полями. Код приложения не выбирает между составным текстовым ключом и вложенным объектом — он задаёт группу, а представление определяет обработчик.

Группировать каждое одиночное поле не требуется. Для привычной плоской схемы `request_id`, `method`, `route` и `status` отдельная группа может только усложнить запросы к логам. Группа полезна, когда она действительно объединяет несколько связанных полей или устраняет коллизию имён.

## Постоянная область через `WithGroup`

Если один компонент регулярно записывает атрибуты в одном пространстве имён, можно создать производный логгер через [`Logger.WithGroup`](https://pkg.go.dev/log/slog#Logger.WithGroup):

```go
upstreamLogger := logger.
    With(slog.String("component", "http-client")).
    WithGroup("http")

upstreamLogger.Info("upstream request completed",
    slog.String("method", "GET"),
    slog.String("route", "/profiles/{id}"),
    slog.Int("status", 200),
)
```

Атрибут `component` был добавлен до `WithGroup`, поэтому остаётся на верхнем уровне. Поля события добавлены после открытия группы и попадают в `http`:

```text
level=INFO msg="upstream request completed" component=http-client http.method=GET http.route=/profiles/{id} http.status=200
```

```json
{"level":"INFO","msg":"upstream request completed","component":"http-client","http":{"method":"GET","route":"/profiles/{id}","status":200}}
```

Как и `With`, метод `WithGroup` возвращает новый `*slog.Logger` и не изменяет исходный. Все атрибуты последующих событий производного логгера будут находиться в открытой группе.

## Порядок `With` и `WithGroup`

Порядок вызовов определяет, где окажется общий атрибут:

```go
logger.
    With(slog.String("service", "users-api")).
    WithGroup("http")
// service находится на верхнем уровне.

logger.
    WithGroup("http").
    With(slog.String("service", "profiles-api"))
// service находится внутри http.
```

`WithGroup` влияет только на атрибуты, добавленные после него. Это относится и к последующим вызовам `With`, и к полям отдельных событий. Если порядок длинной цепочки трудно понять, производные логгеры лучше создавать поэтапно и давать им имена.

Группы можно вкладывать:

```go
databaseLogger := logger.
    WithGroup("database").
    WithGroup("query")

databaseLogger.Info("query completed",
    slog.Duration("duration", duration),
)
```

В текстовом выводе получится ключ `database.query.duration`, а в JSON — объект `database`, внутри которого находится `query`. Большая глубина быстро усложняет схему, поэтому для прикладных событий обычно достаточно одного или двух устойчивых уровней.

## Выбор между `Group` и `WithGroup`

Оба API создают одинаковую логическую структуру, но отличаются областью действия:

| API | Область действия | Подходящий сценарий |
| :--- | :--- | :--- |
| `slog.Group` | одна группа в одном событии | несколько связанных полей конкретной операции |
| `Logger.WithGroup` | последующие атрибуты производного логгера | устойчивое пространство имён компонента |

Если группировка нужна только одному событию, `slog.Group` оставляет её рядом с соответствующими полями. Если все события клиента внешнего API должны содержать `http.method`, `http.route` и `http.status`, производный логгер с `WithGroup("http")` избавляет от повторения.

::: info Go 1.25+
Функция [`slog.GroupAttrs`](https://pkg.go.dev/log/slog#GroupAttrs), добавленная в [Go 1.25](https://go.dev/doc/go1.25#log/slog), создаёт группу только из готовых `slog.Attr`. Она удобна, когда атрибуты уже собраны в срез:

```go
attrs := []slog.Attr{
    slog.String("method", "GET"),
    slog.Int("status", 200),
}

logger.Info("request completed",
    slog.GroupAttrs("http", attrs...),
)
```

Для перечисления пар непосредственно в вызове достаточно `slog.Group`, который доступен начиная с Go 1.21.
:::

## Стабильная схема групп

Имя группы становится частью схемы события. Одно и то же поле не следует попеременно записывать как `method`, `http.method` и `request.method`: для системы обработки логов это три разных пути.

Не стоит также использовать точки внутри обычных ключей:

```go
logger.Info("request completed", "http.method", "GET")
logger.Info("request completed", slog.Group("http", "method", "GET"))
```

В выводе `TextHandler` оба варианта могут выглядеть как `http.method=GET`, но JSON-структура различается:

```json
{"http.method":"GET"}
{"http":{"method":"GET"}}
```

Имена групп и вложенных полей без точек сохраняют однозначное соответствие между текстовым и JSON-представлением. Стабильное расположение атрибута не менее важно, чем его имя и тип.
