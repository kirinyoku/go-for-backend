# Общие поля и производные логгеры

Некоторые атрибуты повторяются во множестве событий. Например, имя сервиса и компонента приходится добавлять и при запуске, и при обработке запросов, и при обращении к базе данных:

```go
logger.Info("service started",
    "service", "users-api",
    "component", "http",
)

logger.Info("request completed",
    "service", "users-api",
    "component", "http",
    "status", 200,
)
```

Такой код легко рассинхронизировать: один вызов забудет поле, другой использует иное имя, а третий передаст неправильное значение. Метод [`Logger.With`](https://pkg.go.dev/log/slog#Logger.With) позволяет один раз связать общие атрибуты с производным логгером.

## Общие атрибуты через `With`

`With` возвращает новый `*slog.Logger`, который добавляет переданные атрибуты к каждому своему событию:

```go
baseLogger := slog.New(slog.NewTextHandler(os.Stdout, nil))

serviceLogger := baseLogger.With(
    slog.String("service", "users-api"),
)

httpLogger := serviceLogger.With(
    slog.String("component", "http"),
)

httpLogger.Info("service started")
httpLogger.Info("request completed", slog.Int("status", 200))
```

В выводе общие поля появятся автоматически. Для краткости время опущено:

```text
level=INFO msg="service started" service=users-api component=http
level=INFO msg="request completed" service=users-api component=http status=200
```

Аргументы `With` разбираются по тем же правилам, что и атрибуты обычного события. Можно использовать пары `"service", "users-api"` или готовые значения `slog.String`.

## Исходный логгер не изменяется

`With` не добавляет поля в существующий логгер. Он создаёт производный экземпляр, поэтому базовый, сервисный и компонентный логгеры можно продолжать использовать независимо:

```go
baseLogger.Info("base logger ready")
serviceLogger.Info("service started")
httpLogger.Info("listener started", "address", ":8080")
```

```text
level=INFO msg="base logger ready"
level=INFO msg="service started" service=users-api
level=INFO msg="listener started" service=users-api component=http address=:8080
```

Атрибут `service` не появился в `baseLogger`, а `component` не изменил `serviceLogger`. В этом примере все три логгера используют один формат и поток вывода, но сохраняют собственный набор общих атрибутов.

Такое поведение позволяет строить логгеры последовательно:

```text
baseLogger
└── serviceLogger          service=users-api
    ├── httpLogger         component=http
    └── databaseLogger     component=database
```

Каждая ветвь получает поля родителя и добавляет свои, не влияя на соседние ветви.

## Область жизни атрибутов

Поле, добавленное через `With`, присутствует во всех последующих событиях производного логгера. Поэтому его область жизни должна совпадать с областью жизни самого логгера.

| Поле | Подходящая область | Способ передачи |
| :--- | :--- | :--- |
| `service` | всё приложение | сервисный логгер через `With` |
| `component` | один долгоживущий компонент | компонентный логгер через `With` |
| `request_id` | один HTTP-запрос | локальный логгер запроса через `With` |
| `status` | одно завершённое действие | атрибут конкретного события |
| `duration` | одно завершённое действие | атрибут конкретного события |

Статус, длительность и ошибка меняются от операции к операции, поэтому обычно не относятся к общим полям. Если добавить `status` в долгоживущий логгер, одно значение будет ошибочно сопровождать все его события.

## Логгеры для компонентов

Базовый сервисный логгер можно разделить на логгеры отдельных компонентов в `main`, а затем передать их как зависимости:

```go
baseLogger := slog.New(slog.NewJSONHandler(os.Stdout, nil))
serviceLogger := baseLogger.With(
    slog.String("service", "users-api"),
)

httpLogger := serviceLogger.With(
    slog.String("component", "http"),
)
databaseLogger := serviceLogger.With(
    slog.String("component", "database"),
)

handler := NewUserHandler(httpLogger)
repository := NewUserRepository(databaseLogger)
```

`UserHandler` и `UserRepository` получают готовый `*slog.Logger`. Им не нужно знать общий формат или повторять имя сервиса. Каждый компонент добавляет к событиям только данные своей операции:

```go
func (r *UserRepository) FindUser(userID int64) {
    // Обращение к базе данных опущено.

    r.logger.Info("user loaded",
        slog.Int64("user_id", userID),
    )
}
```

Событие репозитория уже содержит `service=users-api` и `component=database`, хотя метод `FindUser` не перечисляет их вручную.

## Логгер для запроса

Если в течение одного HTTP-запроса создаётся несколько событий, общий `request_id` можно добавить в локальный производный логгер:

```go
func handleRequest(logger *slog.Logger, requestID string) {
    requestLogger := logger.With(
        slog.String("request_id", requestID),
    )

    requestLogger.Info("request accepted")

    // Обработка запроса опущена.

    requestLogger.Info("request completed",
        slog.Int("status", 200),
    )
}
```

Такой логгер существует только во время обработки конкретного запроса. Его нельзя сохранять в общей структуре приложения: иначе `request_id` одного клиента может попасть в события другого. Для единственного события проще передать `request_id` непосредственно в вызов логгера.

Поскольку `With` не изменяет исходный экземпляр, параллельные запросы могут безопасно создавать собственные производные логгеры от одного `httpLogger`. У каждого будет свой `request_id`, а общий логгер останется без данных конкретного запроса.

## Коллизии общих и локальных полей

Атрибут события с тем же ключом не заменяет поле, добавленное через `With`:

```go
serviceLogger := baseLogger.With("service", "users-api")

serviceLogger.Info("worker started",
    "service", "background-worker",
)
```

`TextHandler` выведет оба значения:

```text
level=INFO msg="worker started" service=users-api service=background-worker
```

Получатель таких данных может выбрать первое значение, последнее или сохранить оба. На переопределение рассчитывать нельзя. Если событие относится к другому сервису или компоненту, для него следует создать логгер от подходящего родителя либо использовать отдельный ключ с другим смыслом.

Производный логгер полезен тогда, когда общий атрибут действительно остаётся неизменным на всей его области жизни. `service` и `component` подходят для долгоживущих экземпляров, `request_id` — для локального логгера запроса, а результат отдельной операции остаётся в самом событии.
